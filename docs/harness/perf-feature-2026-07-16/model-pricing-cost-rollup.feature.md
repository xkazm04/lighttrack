## 1. Unpriced models vanish into $0 — no coverage signal anywhere in the cost rollup

- **Severity**: High
- **Category**: money-truth
- **File**: `crates/render/src/costs.rs:26-52`, `crates/core/src/pricing.rs:156-174`, `crates/api/src/events.rs:33-57`
- **Scenario**: A team adds `openai/gpt-5.2-mini` (or a self-hosted `unknown/llama-3.3`) to their app. It isn't in the price book, so `PriceBook::cost_usd_mode` returns `None`, `ensure_cost` leaves `cost_usd: null`, and `mark_cost_source` deliberately stamps nothing (`events.rs:46-48`). The event is stored and counted. The cost summary renders that model's row with `Cost` = `$0.00` (`costs.rs:29` — `f(r,"cost_usd")` on a null field yields `0.0`) and folds `0.0` into `**Total: …**`. The operator reads a total that is confidently wrong and has no way to know: a $4,000/mo model is visually identical to a genuinely free local model.
- **Root cause**: The `None` → unpriced signal is created correctly in core and then destroyed at every consumer. `cost_source` is stamped only when a cost exists, so "unpriced" has no positive marker to filter on. `render/src/costs.rs` `summary()`/`usecases()` have no unpriced column, no footnote, no divergence between "$0 because free" and "$0 because unknown". `crates/render/src/prices.rs` renders the book but never cross-references it against models actually observed in events. `crates/runner/src/util.rs:18` proves the team already understands this failure mode — it warns on unpriced rows in the *runner* — but the API/rollup path has no equivalent. Nothing in the product ever answers "what fraction of my spend is actually priced?"
- **Impact**: This is the load-bearing number of the whole product. Every downstream figure (budgets, forecast, margin, revenue netting — `revenue.rs:134` sums `llm_cost_usd` straight) silently inherits the understatement. Competitors (Helicone, Langfuse) surface unpriced/unknown-model warnings precisely because new model IDs ship weekly; a price book is *always* behind. Coverage reporting is what converts "our numbers are stale" into "our numbers are stale *and we told you exactly where*", which is the difference between a trusted and an abandoned cost tool.
- **Fix sketch**:
  1. Stamp the negative case: in `mark_cost_source`, when `ev.cost_usd.is_none()`, set `metadata.cost_source = "unpriced"` instead of early-returning. This costs nothing and makes unpriced events queryable across all three backends (metadata already rides everywhere).
  2. Add `GET /v1/prices/coverage` in `crates/api/src/prices.rs`: for a window, group events by `(provider, model)`, left-join against `store.list_prices()`, return `{provider, model, calls, input_tokens, output_tokens, priced: bool}` for each — i.e. the list of models burning tokens with no price row, ranked by volume. This is the operator's to-do list for the price book, and it is derived entirely from data already stored.
  3. Render it: a new `crates/render/src/prices.rs::coverage()` table, plus in `costs.rs::summary()` mark unpriced rows' Cost cell as `—` (not `$0.00`) and append `_N model(s) unpriced — total excludes their spend._` under the total. Distinguishing `—` from `$0.00` is the whole point; do not let an unpriced row render a number.
- **Trade-offs**: Requires an events-by-model aggregation for the coverage endpoint — reuse the existing cost-summary rollup query rather than adding a new scan. Changing `$0.00` → `—` shifts a rendered value; that is the intended correction, but note it for anyone diffing golden output.

## 2. `effective_date` is a structurally dead field — the price book cannot restate history or be audited

- **Severity**: High
- **Category**: half-implemented
- **File**: `crates/core/src/pricing.rs:18-19,99-114`, `crates/api/src/prices.rs:44-55`, `crates/store/src/sqlite/prices.rs:13-33`
- **Scenario**: Anthropic cuts a model's output price on the 10th. An operator PUTs the new price. It applies to everything ingested from that moment on — correct. Then they notice their book had a *typo* since the 1st (`$15` entered as `$1.50`), so ten days of events were stamped with a cost 10× too low. There is no way to fix it: the events carry a baked-in `cost_usd`, and re-pricing them would need to know what the price was *at each event's `ts`* — which the store cannot answer, because the old row no longer exists. Equally, an auditor asking "what price did you charge this customer's March calls at, and where did that number come from?" gets nothing.
- **Root cause**: `ModelPriceRow` carries `effective_date` (`pricing.rs:19`) and every backend faithfully persists it — but **nothing ever reads it for pricing**. `PriceBook::from_rows` (`pricing.rs:99-114`) constructs `ModelPrice` from exactly three fields and drops `effective_date` and `source_url` on the floor; `lookup`/`resolve` have no time parameter at all. The persistence layer cements this: the primary key is `(provider, model)` and the upsert is `ON CONFLICT(provider, model) DO UPDATE SET … effective_date=excluded.effective_date` (`sqlite/prices.rs:18-21`, identically `store-pg/src/prices.rs:18-21`, and Firestore's doc id is `format!("{provider}__{model}")` — `store-firestore/src/prices.rs:12`). So a price change **destructively overwrites** its predecessor in all three backends. `effective_date` is therefore not an effective date; it is a last-modified timestamp that only ever holds `Utc::now()` (`api/src/prices.rs:50`) and is read back solely to be rendered — and `render/src/prices.rs:19-26` doesn't even render it. A field written by three backends and read by zero consumers is the definition of a promised capability that is structurally dead.
- **Impact**: Effective-dated pricing is table stakes for anyone billing customers off these numbers (which `revenue.rs` invites them to do). It unlocks: correct restatement after a price fix, honest historical margin, "price changed on date X" annotations on spend charts, and an auditable answer to how any past cost was derived. It also turns the price book into an asset with history rather than a mutable global variable — the single highest-leverage upgrade available here, since the field, the storage, and the serialization already exist. Only the key and the lookup need to change.
- **Fix sketch**:
  1. Widen the key to `(provider, model, effective_date)` in all three backends — sqlite/pg PK + `ON CONFLICT` target, Firestore doc id `{provider}__{model}__{effective_date}`. Upsert then becomes append-only per date; re-PUTing the same date still corrects in place.
  2. Add `PriceBook::from_rows_asof(rows, at: DateTime<Utc>)` that, per `(provider, model)`, selects the row with the greatest `effective_date <= at`. Keep `from_rows` as `from_rows_asof(rows, Utc::now())` so every existing caller (`api/src/main.rs:182`, `prices.rs:62`) is unchanged and the hot-path ingest book stays a cheap current-prices snapshot.
  3. Accept an optional `effective_date` in `PutPriceReq` (default `Utc::now()`), so an operator can backdate a correction rather than silently rewriting the present.
  4. Add `GET /v1/prices/:provider/:model/history` returning the version list, and render `effective_date` + a `Since` column in `render/src/prices.rs` — the field finally becomes visible.
  5. Follow-up (separate change, gated behind step 1): a re-price/restate admin job that walks events in a window and recomputes `cost_usd` via `from_rows_asof(rows, ev.ts)` for events with `cost_source == "book"` — never touching `"client"` costs, which are provider-reported truth.
- **Trade-offs**: A schema/key migration on three backends; existing rows migrate cleanly since each is its own first version. Steps 1–2 are self-contained and deliver history + audit on their own — step 5 is the payoff but is genuinely separable, so this should not land as one commit.

## 3. Price maintenance is entirely manual — no catalog seed refresh, no staleness signal

- **Severity**: Medium
- **Category**: automation
- **File**: `crates/api/src/main.rs:166-182`, `crates/api/src/prices.rs:37-64`, `crates/render/src/prices.rs:28-32`
- **Scenario**: An operator deploys LightTrack. On first boot the DB is empty, so `config/pricing.json` is seeded once (`main.rs:166` — `if store.list_prices()?.is_empty()`). Six months later, providers have cut prices twice and shipped four new models. The seed file in the repo has been updated by upgrades; the operator's DB has not, because the seed only ever runs against an *empty* book. Every cost figure is quietly running on six-month-old rates and there is no indication anywhere — the price table renders the numbers with total confidence.
- **Root cause**: The only two ways a price enters the book are (a) a one-shot seed that self-disables forever after first boot, and (b) hand-typing `PUT /v1/prices/:provider/:model`. There is no refresh path, no diff against the shipped catalog, and no freshness surface. `source_url` is stored by all three backends and rendered as a bare truncated hostname (`render/src/prices.rs:29-32`) — it records where a price *claims* to come from but is never fetched or checked. `effective_date` (see #2) is the natural staleness signal and isn't rendered. This is exactly the maintenance burden LangSmith/Langfuse/Helicone remove by shipping a maintained catalog — leaving it manual makes the "$/Mtok" column a hand-typed guess.
- **Impact**: Turns the price book from a decaying liability into something that stays true on its own. Directly protects every downstream number, and removes the single most tedious operator chore in the product.
- **Fix sketch**:
  1. Cheapest win first: render `effective_date` as an `Age` column in `render/src/prices.rs` and flag rows older than ~90 days (`⚠ stale`). Pure render change, no schema, immediate honesty.
  2. `POST /v1/prices/seed?mode=missing|diff` (admin-gated via the existing `ensure_can_admin`): `missing` adds only models absent from the DB — the safe idempotent upgrade path that the boot-time seed should have been; `diff` returns the shipped-catalog-vs-DB delta as a **preview** without writing, so the operator sees exactly which rates moved before accepting.
  3. Have the boot path call the `missing` logic instead of the `is_empty()` guard, so a version upgrade that ships new models actually lands them, while never clobbering an operator's hand-tuned rate.
  4. Only then consider fetching remote catalogs — and if so, write them through the same effective-dated append (#2) with `source_url` set, never as a blind overwrite.
- **Trade-offs**: Steps 1–3 use only the repo's own `pricing.json` and add no network dependency; that's deliberate. Auto-fetching provider prices is genuinely hard (no stable machine-readable feeds; scraped rates can be wrong) — a wrong auto-applied price is worse than a stale known one, which is why the design keeps a human accept step and defers fetching to last.
