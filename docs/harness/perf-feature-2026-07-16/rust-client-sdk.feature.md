## 1. Span tree is unreachable: no `span_id` / `parent_span_id` setters

- **Severity**: High
- **Category**: capability-gap
- **File**: `clients/rust/src/lib.rs:179-202` (builder init), `clients/rust/src/lib.rs:260-263` (`trace_id` is the only linkage setter)
- **Scenario**: A Rust agent app makes a planner call, then three tool calls, then a summarizer call per user request. The developer sets `.trace_id("req-123")` on all five and opens `GET /v1/traces/req-123` expecting the waterfall the docs advertise. Instead all five spans render as five sibling roots at `offset_ms` with no nesting — the planner→tool→summarizer structure, the thing they instrumented for, is gone. There is no SDK call that can fix this, so they cannot recover the tree at any price short of hand-building `LlmEvent` and calling `Client::track`.
- **Root cause**: `EventBuilder::new` hardcodes `span_id: None, parent_span_id: None` (lib.rs:183-184) and `impl EventBuilder` exposes a setter for `trace_id` but for neither span field. `grep span_id clients/rust/src/lib.rs` returns exactly those two initializer lines — the fields are written once, to `None`, and never again. Meanwhile `crates/core/src/trace.rs:179-216` (`build_forest`) nests spans strictly by `parent_span_id` → `span_id`; an event with `parent_span_id: None` takes the `_ => roots.push(i)` arm (trace.rs:191-197). Every Rust-emitted event is structurally forced to be a root. The server-side investment behind this is large and all of it is dark for Rust callers: `TraceSpan.children`, `offset_ms`, `Trace::root_event_id`, `GET /v1/traces/:id`, and `POST /v1/traces/:id/score` (main.rs:244-246). Both sibling SDKs reach it — Python `instrument.py:102-112` (`_span_ids`, with a documented span-ownership rule) and TS `instrument.ts:64-81` (`currentSpanId` / `withSpan`) — and `clients/README.md:14-17` sells nested spans as an SDK-suite feature, so this is a parity hole against the project's own marketing.
- **Impact**: Restores the product's flagship differentiator (trace trees / cost-per-request attribution) for Rust users. Rust is the language LightTrack itself is written in, so Rust users are the most likely early adopters and the ones most likely to dogfood — shipping them the one SDK that cannot draw a trace tree is the worst possible allocation of the gap.
- **Fix sketch**:
  1. Add `span_id(impl Into<String>)` and `parent_span_id(impl Into<String>)` to `EventBuilder` (~6 lines, mirrors `trace_id` at lib.rs:260-263). This alone unblocks manual tree construction.
  2. Add ambient context so users don't thread ids by hand. Rust has no `contextvars`, so use `tokio::task_local!` for async callers plus a `thread_local!` fallback, and port Python's span-ownership rule verbatim (`instrument.py:102-112`): first call in a span claims the span id and parents to the enclosing span; later calls get a fresh id parented to the span. This avoids emitting synthetic zero-cost placeholder events.
  3. Expose `Client::trace(...)` / `Client::span(...)` returning RAII guards that set/reset the task-local on drop; have `EventBuilder::new` default `trace_id`/`span_id`/`parent_span_id` from the ambient context when unset.
- **Trade-offs**: Step 1 is additive and zero-risk. Steps 2-3 add a `tokio` dependency to a currently runtime-agnostic client — gate it behind a default-off `context` feature so sync/non-tokio users keep today's dependency footprint. Ordering matters: ship step 1 immediately (it is the mechanical blocker), then 2-3.

## 2. No timing helper and no provider wrapper — every user hand-rolls `Instant::now()`

- **Severity**: High
- **Category**: onboarding
- **File**: `clients/rust/src/lib.rs:126-154` (`track_*_json`), `clients/rust/src/lib.rs:235-238` (`latency_ms`), `clients/rust/examples/quickstart.rs:16-33`
- **Scenario**: A developer follows the quickstart, wraps their OpenAI call, and ships. Latency on the dashboard is empty for every event — they never noticed `latency_ms` was an opt-in field they had to measure themselves. The p95-latency panel, one of the three things an observability tool exists to show, is blank and they conclude the product is broken. The failure is silent: no type error, no warning, just a missing column.
- **Root cause**: `grep -E "Instant|elapsed" clients/rust/src/lib.rs` returns nothing — the SDK has no clock. `latency_ms` (lib.rs:235-238) is a manual setter the caller must feed from their own `Instant`. Worse, the three convenience entry points that look like the easy path actively cannot carry it: `track_openai_json` / `track_anthropic_json` / `track_gemini_json` (lib.rs:126-154) each end in a hardcoded `.usage(input, output, cached).send()` — they take only `(&Value, Option<&str>)`, return `()`, and expose no seam for `latency_ms`, `name`, `trace_id`, or the error path. A user who calls `track_openai_json` is *guaranteed* an event with no latency, no use-case name (the `name` field is the LLM-Overview rollup key per `crates/core/src/event.rs:133-137`), and no trace linkage. The convenience method silently produces the least useful event the schema allows. Both siblings solve this: TS ships a `Span` class that stamps `startedAt` in the constructor and computes `latencyMs: Date.now() - this.startedAt` in `end()` (`typescript/src/index.ts:416-464`, with `setOpenAI`/`setAnthropic`/`setGemini`); Python ships `lt.span(...)` as a context manager plus full monkey-patch auto-instrumentation (`instrument.py:274-312`). `clients/README.md:48-52` lists Rust as a peer of both while the Rust snippet (README lines 97-110) shows only the manual builder.
- **Impact**: This is the adoption funnel. Competitors (Helicone's one-line base-URL swap, Langfuse/LangSmith decorators) set the expectation that instrumentation is a one-line change; LightTrack's Rust story is "hand-construct every event and remember to time it yourself." Closing this makes correct-by-default the easy path rather than the expert path.
- **Fix sketch**:
  1. Add a `Span` RAII guard mirroring the TS shape: `Client::span(provider, model)` captures `Instant::now()`, offers `set_openai(&Value)` / `set_anthropic(&Value)` / `set_gemini(&Value)` / `set_usage(..)`, and emits on `Drop` (or explicit `end()`) with `latency_ms` filled from `started.elapsed().as_millis()`. Reuse the existing extraction logic in lib.rs:127-153 rather than duplicating the JSON paths.
  2. Change `track_*_json` to return `EventBuilder<'_>` instead of `()` so the convenience path composes: `lt.track_openai_json(&resp, None).latency_ms(x).name("summarize").send()`. This is a breaking change to those three signatures — mitigate by keeping the current void methods as `track_*_json_now` shims, or accept the break given the SDK's pre-1.0 status.
  3. Rust cannot monkey-patch, so true auto-instrumentation needs a different seam. The idiomatic option is a `reqwest_middleware::Middleware` that observes provider HTTP traffic and reports it — genuinely one-line for users on `async-openai` and friends, but a larger build. Ship 1-2 first; scope 3 separately.
- **Trade-offs**: (1) is additive and small. (2) is a breaking API change — do it now while adoption is low, not later. (3) is a real project and should not be bundled with the quick wins; note that the middleware approach only sees calls that go through `reqwest`, so it complements rather than replaces the `Span` API.

## 3. Batch ingest endpoint unused — worker POSTs one event per request

- **Severity**: Medium
- **Category**: capability-gap
- **File**: `clients/rust/src/lib.rs:51-69` (worker loop), `clients/rust/src/lib.rs:90-94` (`send_raw`)
- **Scenario**: A batch job scores 10,000 documents in a tight loop. The SDK issues 10,000 sequential HTTP POSTs on one worker thread, each with a 2-second timeout. Because `send_raw` never blocks (lib.rs:91-93 does an unbounded `tx.send`), the producer races ahead and the channel backlog grows without bound; if LightTrack is slow, the queue balloons in RSS. At `flush()` the user waits for the worker to drain that queue one round-trip at a time — and `Client::flush` (lib.rs:157-159) is `drop(self)`, which joins with no timeout, so process exit stalls behind the backlog.
- **Root cause**: The worker's `while let Ok((path, body)) = rx.recv()` loop (lib.rs:61-67) handles exactly one item per iteration and POSTs it immediately. There is no `try_recv` drain, no coalescing, no size or time window. Meanwhile `POST /v1/events/batch` exists and is well-built for this: it takes a JSON array, defaults to 500 items per call (`LIGHTTRACK_MAX_BATCH`, main.rs:55), runs each item through the identical pipeline via shared `prepare_event` / `on_admission`, and returns ordered per-item `accepted | rejected | invalid` results under one HTTP 200 (`events_batch.rs:29-54`). It even keeps admission correct across the batch — `insert_events_checked` is one critical section so a batch cannot bypass a spend cap (`events_batch.rs:101-108`). Verified cross-SDK: `grep -rn "events/batch" clients/` returns **zero hits**, so no client reaches it — the endpoint has no callers at all. That makes this a suite-wide gap that happens to bite Rust hardest, since Rust is the SDK most likely to be used for high-throughput batch work.
- **Impact**: Order-of-magnitude fewer HTTP round-trips on high-volume workloads, bounded flush time at shutdown, and it retires an entire server feature's dead-code status. Also the natural place to add a queue cap — `clients/README.md:127-129` claims "a full queue drops events rather than blocking," which the Rust client's unbounded `mpsc::channel` does not currently honor.
- **Fix sketch**:
  1. In the worker loop, after the blocking `recv()`, drain with `try_recv()` into a buffer until it is empty or the buffer hits N (default 100, well under the server's 500 cap).
  2. Group the buffer by path. Send `/v1/events` groups of 2+ to `/v1/events/batch` as an array; keep singletons and all `/v1/scores` items on the existing per-item path (there is no batch scores endpoint).
  3. Add a small linger (~50ms) before the drain so bursts coalesce instead of trickling; skip the linger when the channel is already closed so `flush()`/`Drop` stays prompt.
  4. Separately, swap `mpsc::channel` for `sync_channel(capacity)` with `try_send`, dropping on full — this makes the README's stated drop-rather-than-block guarantee true.
- **Trade-offs**: Adds bounded latency (the linger) between call and visibility — irrelevant for telemetry, but document it. Batching changes the failure blast radius: today a failed POST loses one event, batched it loses up to N. The current code already discards the response entirely (`let _ = req.send()`, lib.rs:66), so the rich per-item `BatchResponse` would be thrown away — worth pairing with an optional debug callback, but do not let that expand the scope. Step 4 is behavior-visible (events start dropping under sustained overload where they previously queued) and should ship as its own change.
