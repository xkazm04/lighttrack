## 1. No "me vs. the collective" comparison — the pooled data only ever renders a global table

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/core/src/collective/merge.rs:207-252` (and `Acc::contributors`, `crates/core/src/collective/merge.rs:62`)
- **Scenario**: An operator opts in, contributes their digest, and opens the leaderboard. They see that `haiku/qa` scores 0.87 across 3 contributors. The one question that made them opt in — *"is **my** 0.79 on haiku/qa bad, or is my workload just harder? am I overpaying versus everyone else?"* — has no answer. They must eyeball their own digest against a merged row that already includes their own numbers, which drags the "peer" mean toward their own and makes a heavy contributor compare against itself.
- **Root cause**: `merge_leaderboard` is the only consumer of `CollectiveEntry`, and it deliberately dissolves contributor identity: `Acc::add` inserts `contributor` into a `BTreeSet` (`merge.rs:93-95`) that is read exactly once, for `n_contributors: a.contributors.len() as u32` (`merge.rs:244`). Per-contributor position within a bucket is computed and then thrown away. There is no exported function taking *both* an instance's own stats and the pooled entries — `mod.rs:66` exports only `build_digest` and `merge_leaderboard`, so no caller can ask a relative question.
- **Impact**: This is the moat feature. A global vendor-style table is commodity; *"your gpt-4o costs 2.3× the collective median for extraction and scores 0.04 below the peer mean (p25, 14 peers)"* is a per-tenant insight no competitor with single-tenant data can produce, and it is the natural upsell/retention hook for opting in.
- **Fix sketch**:
  1. Add `types::CollectiveComparison { provider, model, task_type, own_quality, peer_quality, peer_quality_ci95, quality_delta, own_cost_usd, peer_cost_usd, cost_ratio, own_p50/peer_p50, percentile: Option<f64>, n_peers, n_peer_cases, low_confidence, judge_mismatch: bool }`.
  2. Add `merge::compare_to_collective(own: &[ModelDigestEntry], entries: &[CollectiveEntry], exclude: Option<&str>, low_confidence_floor: u32) -> Vec<CollectiveComparison>`. Filter `entries` by `e.contributor_id != exclude` **first** (leave-one-out), reuse the existing `Acc` fold per `Key` so the peer aggregate is identical maths to the leaderboard, then join on `key_of(provider, model, task_type)`.
  3. `percentile` = rank of `own.quality` among the *per-contributor* qualities in the peer bucket (case-weighted), `None` when `n_peers < 3` — the same honesty posture as `quality_ci95`.
  4. Set `judge_mismatch` when the own entry's `judge_provider` is absent from the peer bucket's `judge_providers` set — reusing the existing incommensurability signal so a cross-judge delta is never presented as fact.
  5. Export from `mod.rs`; surface as `GET /v1/collective/compare` (excluding `st.collective.contributor_id`) and a `comparison()` renderer.
- **Trade-offs**: Leave-one-out exclusion depends on the hub-derived contributor id, so it works only for instances that push with a bearer key (anonymous contributors collapse to one identity and cannot be excluded cleanly — return `exclude: None` and label the row "includes your data"). A small hub means many buckets have `n_peers < 3` and can only show a delta, not a percentile. Deltas across mixed judges are not commensurable — the `judge_mismatch` flag must be rendered, not swallowed.

## 2. The collective is timeless — `received_at` is stored but never merged, so cross-tenant model drift is invisible

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/core/src/collective/merge.rs:207-225`; field defined at `crates/core/src/collective/types.rs:104`
- **Scenario**: A provider silently ships a quantized/updated endpoint behind the same model name. Every instance's quality on `gpt-4o/extraction` slips ~4% over two weeks. No single operator can distinguish that from their own prompt changes or noise, so nobody acts. The collective — the one observer with cross-tenant evidence — reports a single all-time mean of 0.84 and says nothing.
- **Root cause**: Time exists in the schema and is dropped in the maths. `CollectiveEntry::received_at` (`types.rs:104`) is never read by `merge_leaderboard`: the `Sample` struct (`merge.rs:31-46`) has no time field, and `groups.entry(key_of(&e.provider, &e.model, &e.task_type))` (`merge.rs:210`) keys buckets on identity alone, so a digest from March and one from today fold into the same `Acc`. Symmetrically, `build_digest` (`merge.rs:158-201`) collapses every `RunStat` an instance has ever recorded into one bucket with no period bound — `gather_run_stats` walks *all* runs ever — and `ModelDigestEntry` carries no window, so even a time-aware hub could not tell when the contributed cases occurred. `LeaderboardRow` has no time field either.
- **Impact**: Cross-tenant regression detection is the single insight pooled data uniquely enables and no tenant can buy elsewhere — "12 instances observed `gpt-4o/extraction` drop 0.04 in the last 14 days" is an alertable, headline-grade product event. It also fixes a live correctness problem: a stale contributor who benchmarked a six-month-old snapshot dilutes today's ranking forever, since nothing ages out.
- **Fix sketch**: Stage it, smallest useful step first.
  1. *Wire (schema v3).* Add `window_start` / `window_end: Option<DateTime<Utc>>` to `ModelDigestEntry` and `CollectiveEntry` (both `#[serde(default)]`, so v1/v2 keep merging as today); bump `DIGEST_SCHEMA_VERSION` to 3 in `mod.rs:33`, leave `MIN_SCHEMA_VERSION` at 1. Extend `RunStat` with the run's `started_at` and have `Acc` track min/max.
  2. *Recency-scoped merge.* Give `merge_leaderboard` a `since: Option<DateTime<Utc>>` that filters entries by `received_at` before folding — a few lines at `merge.rs:209`, no new estimator, and it immediately makes a "last 30 days" leaderboard possible from data the hub already has.
  3. *Digest bucketing.* Add `build_digest_windowed(stats, min_cases, period)` that keys on `(provider, model, task_type, period_bucket)` — reuse `Acc` unchanged — so an instance contributes per-week buckets instead of one all-time blob. k-anonymity must be enforced **per window** (a weekly bucket is thinner than an all-time one; expect fewer publishable rows).
  4. *Drift.* `detect_drift(entries, window, baseline)` → per key, compare the recent window's pooled mean against the prior baseline, and report only when the gap exceeds the pooled `quality_ci95` **and** ≥3 distinct contributors moved the same direction — sign-agreement across independent tenants is what separates a provider regression from one operator's prompt edit.
- **Trade-offs**: Real cost. Per-window bucketing multiplies rows by the number of windows and collides with k-anonymity — thin weekly buckets get withheld, so the drift signal is strongest exactly where volume is highest and may be absent for long-tail models; a coarser (monthly) window trades detection latency for coverage. A schema bump means mixed-version hubs merge windowed and all-time entries in one `Acc` — step 2 must treat a missing window as "all-time" and not silently attribute old cases to the recent period. Step 4's alerting risks false positives if the CI floor is trusted too far — the module already documents that the CI is a *floor* on true uncertainty (`merge.rs:4-18`), so the sign-agreement quorum is not optional.

## 3. New models fragment into thin per-date rows and get withheld — the alias table only collapses variants it was hand-taught

- **Severity**: Medium
- **Category**: automation
- **File**: `crates/core/src/collective/aliases.rs:44-53`
- **Scenario**: A provider ships `claude-x-5-20260501`, and three weeks later `claude-x-5-20260522`. Operators benchmark both. Because neither string is in `config/model_aliases.json`, the hub creates two separate rows, each carrying a third of the cases. Both fall under the k-anonymity floor and are dropped from the digest entirely (`merge.rs:182`). The leaderboard shows nothing for the newest model — precisely when operators most want collective evidence — and the only fix is a hand-edited config file shipped to the hub.
- **Root cause**: Collapse is table-driven and exact-match only: `self.models.get(&model.to_lowercase())` (`aliases.rs:50-53`) falls back to `model.to_string()` on a miss. The documented intent (`aliases.rs:5-8`) — "collapses provider synonyms and **dated model variants** to their family" — is only realized for dates someone enumerated by hand (`gpt-4o-2024-08-06` → `gpt-4o`). There is no rule for the *shape* of a dated variant, so the table is permanently behind the model release cadence, and the pass-through miss is silent (nothing counts or reports unmapped identities).
- **Impact**: Keeps the collective current on new models without a config release — exactly the models with the highest interest and the thinnest per-variant data, where merging fragments is the difference between a published row and nothing. Also removes standing hub-operator toil.
- **Fix sketch**:
  1. In `normalize`, after the table lookup misses, apply a conservative structural rule: strip a trailing date suffix matching `-\d{8}$` or `-\d{4}-\d{2}-\d{2}$` (and optionally a trailing `-v\d+$`), then re-check the table with the stripped stem before falling back. Keep it strictly post-miss so an explicit table entry always wins and existing behaviour is untouched.
  2. Gate it on an `auto_collapse_dates: bool` field (`#[serde(default)]`) in the JSON so a hub operator can turn the heuristic off and keep today's exact-match-only semantics.
  3. Return the fact of a rewrite (e.g. `normalize_reported(&self, ..) -> (String, String, bool)`) so the ingest path can count heuristic collapses — an unmapped-identity counter turns "the table is stale" into an observable signal rather than a silent miss.
  4. Tests: `claude-x-5-20260501` and `-20260522` collapse to one stem; `gpt-4o` is untouched; `gpt-4.1` / `llama-3.1-70b` are untouched (the rule must not eat version numbers); an explicit table entry still overrides.
- **Trade-offs**: A heuristic that mis-fires merges two genuinely different models into one row — the exact failure the module doc calls out ("a new model is never silently mis-merged", `aliases.rs:8`), which is why the rule must be narrow (trailing date only), off-switchable, and reported. Date-suffixed snapshots of the *same* family can genuinely differ in quality, so collapsing them trades per-snapshot resolution for a publishable row; that is the right trade at k=5 but should be documented on the leaderboard footnote.
