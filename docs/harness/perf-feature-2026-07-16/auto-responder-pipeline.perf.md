## 1. No admission control on investigations — every POST spawns an unbounded, billable `claude` child

- **Severity**: Critical
- **Category**: unbounded-growth / cost-hot-path
- **File**: `crates/responder/src/webhook.rs:59-78`, `crates/responder/src/pipeline.rs:17-27`
- **Scenario**: `receive` ends with `tokio::spawn(pipeline::handle_trigger(...))` on every well-formed spike/drop body, with no in-flight check, no concurrency cap, and no responder-side cooldown. Each accepted trigger reaches `claude::run` (`crates/responder/src/claude.rs:20-59`), which spawns a Claude Code child bounded only by `max_budget_usd` (default **$1.00**) and `timeout_secs` (default **240s**). The `Breaker` exists but is consulted only inside `act::run_act` — i.e. it gates the auto-fix *edit*, never the investigation that precedes it. The only dedup for the investigation is the **alerter's** cooldown in a different process (`crates/api/src/alerts.rs:331` `should_send_key("error-spike:{project}")`, `LIGHTTRACK_ALERT_COOLDOWN_SECS`, default 3600). The responder trusts that entirely: it is an unauthenticated endpoint, so a webhook retry, a restarted/duplicated API instance, a cooldown set low, or a hand-rolled `curl` each buy a fresh $1 run. Even in the happy path, N mapped projects spiking in one window fan out to N concurrent children — the map file has no bound on N.
- **Root cause**: The spawn is unconditional and detached. Nothing between "HTTP 200 accepted" and "fork a Node-based agent process" consults any counter, set, or semaphore. Cost control was placed on the *write* stage (ACT) but the *read* stage is what actually spends money and RAM, and it runs first and always.
- **Impact**: Per trigger: up to $1.00 of model spend, up to 240s wall-clock, and one Claude Code child (Node runtime, typically 200–500MB RSS) holding a repo working set. 20 duplicate or concurrent triggers ⇒ ~$20 and ~4–10GB RSS with zero backpressure; the host OOMs or thrashes long before any configured limit notices, because no configured limit is watching. A retry loop on the sender side is an open-ended billing leak. `kill_on_drop(true)` does not help — nothing drops the futures.
- **Fix sketch**: Put the gate in `webhook::receive` (or at the top of `handle_trigger`, before `classify`), reusing `AppState`:
  1. **In-flight dedup**: add `inflight: Mutex<HashSet<String>>` to `Breaker` (or a new `Admission` type in `state.rs`); insert `project_id` before spawn, remove in a guard on completion. If already present, log + return `StatusCode::OK` without spawning. This alone kills retry/duplicate-instance amplification.
  2. **Global cap**: an `Arc<tokio::sync::Semaphore>` sized ~2–4 in `AppState`; `try_acquire_owned()` before spawn, drop the permit when the task ends. Shed (log + 200) rather than queue — a 10-minute-old spike is not worth investigating.
  3. **Per-project investigate cooldown**: reuse the existing `Breaker::allow`/`record` shape with a new `investigate_cooldown_secs` default (~600s) so the responder is not dependent on a remote process's cooldown for its own bill.
  4. Optionally add `max_investigations_per_hour` mirroring `max_acts_per_hour` as a hard spend ceiling.
  Metric to watch: concurrent-children gauge and sum of `ClaudeRun.cost_usd` per hour (already parsed at `claude.rs:86` and currently discarded into the report only).
- **Trade-offs**: Shedding means a genuine second spike for the same project inside the cooldown gets no report — acceptable, since the alerter's own dedup already assumes that. Needs one `Mutex`/`Semaphore` in `AppState`; no async-lock complexity if the guard is released via a drop guard rather than manual removal (a panicking task must not leak the in-flight entry).

## 2. Enrichment over-fetches and filters client-side — the $1 investigation routinely runs with no context

- **Severity**: High
- **Category**: missing-filter / wasted-work
- **File**: `crates/responder/src/enrich.rs:8-42`, `crates/responder/src/pipeline.rs:49-55`
- **Scenario**: `recent_failures` requests `/v1/events?project={p}&limit={enrich_limit}` (default 20) and then discards every `status == "success"` row in a client-side loop, keeping at most 10 lines. `GET /v1/events` has **no status filter** — `EventFilter` (`crates/store/src/lib.rs:65-74`) exposes `since`/`until`/`provider`/`model`/`trace_id`/`name`/`cursor` only. So the responder can only ask for "the newest 20 events, whatever they are". An error spike is defined as `error_threshold` failures (default 5) inside `error_window` (default 300s) — it does **not** imply failures dominate recent traffic. A project doing 200 calls/min at a 3% error rate trips the spike while its 20 newest events contain ~0–1 failures. `recent_scores` has the same shape: `pipeline.rs:89` asks for 30, keeps ≤12, and filters by rubric client-side.
- **Root cause**: The filter the caller needs lives on the wrong side of the network. The server can only page newest-first, so the client pays to transfer and `serde_json`-parse full event rows into `Vec<Value>` and throws most away — and, worse, has no way to reach *older* failures without blindly raising `limit` (there is no loop over `next_cursor` here either, so raising the limit is the only lever and it scales the waste linearly).
- **Impact**: The wasted bytes/parse are small (20 rows). The expensive failure is downstream: when the window holds no failures, `context` becomes the literal string `"(no recent failing events found in LightTrack)"` and `error_prompt` (`investigate.rs:44`) hands Claude a spike with **only the single alert error string**. The pipeline then spends the full $1 budget and up to 240s on a near-blind investigation, and the resulting low-confidence diagnosis is what gates auto-fix (`pipeline.rs:60`). That is 100% of the run's cost buying a fraction of its value, on exactly the high-traffic projects where spikes matter most. Raising `enrich_limit` to compensate scales transfer/parse linearly and still cannot express "only failures".
- **Fix sketch**: Push the predicate server-side. Add `status: Option<String>` to `EventFilter` (`crates/store/src/lib.rs:65-74`) — `TraceFilter:94` already carries exactly this field, so the shape and the `"success"`/`"error"` vocabulary are precedented — thread it through `list_events_filtered` in the sqlite/pg/firestore backends and the `/v1/events` query struct, then have `enrich.rs:14` request `&status=error&limit=10`. The `events(project_id, ts)` scan stays index-served; the status predicate is a cheap filter. Interim fix with no API change: pass `&since={now - error_window}` and a higher limit so the fetch is at least scoped to the spike window, and page `next_cursor` until 10 failures are collected or the window is exhausted. Metric: fraction of investigations whose context is the empty-fallback string — that should approach 0.
- **Trade-offs**: Touches the store trait and all three backends (sqlite/pg/firestore) plus the API query struct — a wider diff than the responder. The `since`-window interim fix is responder-local and can land first. Adding a status filter to a keyset-paginated endpoint means a filtered page can return fewer rows than `limit` while more remain, so `next_cursor` semantics must stay "more rows may exist" rather than "page was full".

## 3. A fresh `reqwest::Client` per trigger, and its fallback silently has no timeout

- **Severity**: Medium
- **Category**: allocation / blocking-async
- **File**: `crates/responder/src/pipeline.rs:129-134` (callers at `:49-55` and `:88-90`)
- **Scenario**: `http_client()` is called inline at each enrichment site, building a whole new `reqwest::Client` — a fresh rustls `ClientConfig`, a fresh connection pool, a fresh resolver — for exactly one GET, which is then dropped. `Config` is already `Arc`-shared and `AppState` already carries shared handles (`state.rs:8-12`), so the seam for a shared client exists and is unused. Note `crates/api/src/alerts.rs` keeps a long-lived `self.http` and clones it — the correct pattern is already in the codebase, just not here.
- **Root cause**: Client construction is treated as free. It is not: rustls root-store setup plus pool/resolver allocation is milliseconds-scale, and building per request also guarantees a cold connection (full TCP + TLS handshake) every time, since the pool dies with the client. Separately, `.unwrap_or_default()` on line 133 discards a build error and substitutes `Client::default()` — **which carries no timeout at all**, silently dropping the 10s cap the line above just set.
- **Impact**: The construction and handshake cost (single-digit ms) is genuinely dwarfed by a 240s Claude run — on its own this is a micro-optimization and not worth reporting. What earns the severity is the `unwrap_or_default()` fallback: if the builder ever fails (rustls provider/root-store init being the realistic cause, and it would fail on *every* call since it is rebuilt every time), enrichment loses its 10s bound entirely. `recent_failures` then awaits a hung or black-holed LightTrack **forever** — there is no `tokio::time::timeout` around the enrich call, unlike the claude child at `claude.rs:50`. The detached task parks indefinitely, holding its permit/in-flight slot once finding #1 is fixed, and no report is ever written. A silent fallback that removes the only bound on a network await is the real defect; per-call construction is what makes it reachable on every request instead of once at boot.
- **Fix sketch**: Build the client **once** in `main.rs` and store it on `AppState` (`http: reqwest::Client` — `Client` is internally `Arc`, so `Clone` is cheap); thread it through `handle_trigger` into `enrich::recent_failures`/`recent_scores`, which already take `&reqwest::Client` and need no change. Replace `.unwrap_or_default()` with `?`/`expect` at startup so a builder failure is a loud boot error rather than a silent removal of the timeout. Delete `http_client()`. This also restores connection reuse across triggers for the same LightTrack host.
- **Trade-offs**: None material — it is strictly fewer allocations, a warm pool, and a fail-fast boot. Adds one field to `AppState` and one parameter through `handle_trigger`/`run_error`/`run_quality`.
