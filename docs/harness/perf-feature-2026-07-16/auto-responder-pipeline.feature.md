## 1. Responder cannot authenticate to LightTrack — enrichment is dead under enforced auth

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/responder/src/enrich.rs:14-21`, `crates/responder/src/config.rs:10-19`
- **Scenario**: An operator runs LightTrack with API keys on (the real deployment mode), maps a repo, and waits for a spike. The diagnosis arrives, but every report reads `(enrichment unavailable: bad response from LightTrack: ...)`. Claude Code investigates with nothing but the single error string from the alert — precisely the situation enrich.rs's own module doc says it exists to prevent ("so the investigator sees the real request/response shape, not just the one error string from the alert"). The failure is silent: enrichment is best-effort, so the pipeline reports success while running blind.
- **Root cause**: `Config` (config.rs:10-19) has no API-key field, and `recent_failures`/`recent_scores` issue bare `client.get(&url).send()` with no `Authorization` header. `crates/api/src/guards.rs:22-30` returns `Principal::Dev` only when `st.auth_mode == AuthMode::Dev`; under `AuthMode::Enforced` a keyless `GET /v1/events` is a 401, whose JSON error body fails to deserialize into `Vec<Value>` and falls into the `Err(e) => return format!("(enrichment unavailable: ...)")` arm. Enrichment therefore only ever works against a dev-mode API. The same applies to `/v1/scores` in `recent_scores` (enrich.rs:53).
- **Impact**: Restores the responder's core differentiator. A diagnosis grounded in 10 real failing events (models, timestamps, recurrence) versus one error string is the difference between "here is the code path and root cause" and a guess. It also unblocks every future read-back the loop needs (costs, traces, prompt versions), and makes the responder deployable outside a dev-mode API at all.
- **Fix sketch**:
  1. Add `api_key: Option<String>` to `Config`, read from `LIGHTTRACK_API_KEY` (fall back to `LIGHTTRACK_RESPONDER_API_KEY`) via the existing `env_opt` helper; log `key=on/off` in main.rs's startup banner alongside `email=`.
  2. Add a small `get_json(client, url, key)` helper in enrich.rs that sets `.bearer_auth(key)` when present, and — critically — checks `resp.status().is_success()` before `.json()`, so a 401 surfaces as `(enrichment unavailable: HTTP 401 — set LIGHTTRACK_API_KEY)` instead of an opaque deserialize error.
  3. While there: `/v1/events` already accepts `since`/`until` (`crates/api/src/events.rs:182-195`), so pass `since = now - spike window` to stop the enricher formatting stale week-old failures as if they were the spike, and drop the hardcoded `lines.len() >= 10` cap in favor of `cfg.defaults.enrich_limit` (currently `enrich_limit` sizes the *fetch* but the output is capped at 10 regardless).
  4. Optional follow-on: request `trace_id` per failing event and include it in the bullet lines so the investigator can pivot to `/v1/traces/{id}`.
- **Trade-offs**: None material. The key is optional, so dev-mode setups keep working unchanged. Adding a status check makes previously silent 401s visible, which may surface that existing deployments were never enriched — that is the point.

## 2. The reactive loop ignores cost/budget alerts — the product's core signal has no responder

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/responder/src/webhook.rs:59-78`, `crates/responder/src/pipeline.rs:17-27`
- **Scenario**: A prompt change quietly doubles input tokens per call. LightTrack does its job: the `cost_usd` limit breaches, and the alerter POSTs to the responder webhook. The responder logs `ignoring alert event='...' (no spike/drop payload)` and does nothing. Nothing crashed and no judge score fell, so neither existing pipeline fires — yet cost tracking is LightTrack's headline capability, and the one alert class the operator most wants a code-level "why" for is the one the loop drops on the floor.
- **Root cause**: `receive` (webhook.rs:61-75) dispatches on exactly two body keys, `spike` and `drop`; the `else` branch treats everything else as noise, and the comment names the ignored classes outright: "Breach / forecast / relay-dead alerts share this endpoint; only spikes/drops drive a run." `Trigger` (webhook.rs:45-48) has only `Error` and `Quality` variants, and `handle_trigger` (pipeline.rs:23-26) matches both exhaustively. The upstream signal already exists and is already delivered here: `crates/api/src/alerts.rs:177` (`notify` for breaches, keyed `project:metric:window:scope`) and `alerts.rs:276` (`notify_forecast`, pre-emptive budget-breach/margin-erosion alerts).
- **Impact**: Completes the reactive loop across all three things LightTrack measures — errors, quality, and cost. A cost regression has a code cause (model swap, prompt bloat, lost caching, a retry loop) that is exactly as diagnosable as an error, and the read-only investigator with `git log`/`git diff` access is well suited to "what landed recently that made this more expensive". Forecast alerts make it *pre-emptive*: diagnose the trend before the budget is actually blown. This is the highest-fit unbuilt feature in the crate.
- **Fix sketch**:
  1. Add `struct Breach { project_id, metric, window, current, threshold, ratio, action }` (shape per `crates/render/tests/render.rs:29-30`) and a `Trigger::Cost(Breach)` variant; extend `project_id()` and `receive`'s dispatch with a `body.get("breach")` arm. Gate on `metric == "cost_usd"` — a `calls`-metric breach is a traffic story, not a code one, and should stay ignored.
  2. Add `enrich::recent_costs(client, base_url, project, since)` hitting the existing `GET /v1/costs` (`crates/cli/src/main.rs:298`) plus the use-case rollup grouped by (name, provider, model) (`crates/api/src/events.rs:271-290`), formatted as a compact "what got more expensive" table.
  3. Add `investigate::cost_prompt(entry, breach, context)` mirroring `quality_prompt`'s structure: sections `Likely cause:` / `Evidence (file:line):` / `Recommended remedy:` / `Confidence:`, steering at the real candidates — model choice, prompt/template growth, missing prompt caching, retry amplification, context stuffing. Reuse the same UNTRUSTED-DATA fencing.
  4. Add `pipeline::run_cost` alongside `run_quality`, wired through the same `deliver(...)` with `kind = "cost breach"`. Diagnosis-only, `act_outcome = None` — a spend regression is a product trade-off call, not something to auto-edit.
  5. Ship step 1-4 for breaches first; forecast alerts (`ForecastAlert`) are a second, near-identical arm once the shape proves out.
- **Trade-offs**: Each breach costs an investigation (≤ `max_budget_usd`, default $1). Breaches are already cooldown-deduped upstream (`alerts.rs:420-431`), so the volume is bounded, but this genuinely widens the loop's spend surface — worth a per-metric opt-in (`cost_investigate: bool` on `ProjectEntry`, defaulting off) mirroring how `auto_fix` is gated.

## 3. No per-repo serialization or durable breaker state — concurrent triggers collide on one working tree

- **Severity**: Medium
- **Category**: workflow
- **File**: `crates/responder/src/webhook.rs:76`, `crates/responder/src/pipeline.rs:17-27`
- **Scenario**: A bad deploy trips an `error_spike` and, minutes later, a `score_drop` for the same project. Two detached tasks now run against the same checkout. If the project is `auto_fix`, the error task's ACT stage runs `git checkout -b lt-fix/...` (`act.rs:71`) and later `git checkout <orig>` (`act.rs:95`) **on the user's real working tree** while the quality investigation is mid-`Read`/`Grep` — that run silently reads a different branch than the one it reports on. Two projects mapped to the same `repo` path collide the same way. Separately, restarting the responder wipes the breaker's in-memory maps, resetting the hourly auto-fix cap to zero.
- **Root cause**: `receive` does `tokio::spawn(pipeline::handle_trigger(...))` (webhook.rs:76) with no in-flight registry, and `handle_trigger` goes straight from a map lookup to `run_error`/`run_quality` with nothing serializing on `entry.repo`. `AppState` (state.rs:8-12) holds only `cfg` + `breaker`, and `Breaker` (breaker.rs:12-15) is two in-process `Mutex`es with no persistence. The upstream alerter dedups per project *per alert kind* (`error-spike:{project}` at alerts.rs:331, a distinct key from score_drop's), so it will never suppress a cross-kind overlap — the responder is the only place that could.
- **Impact**: Makes the loop trustworthy when it matters most — a real incident is exactly when multiple alert kinds fire at once. Prevents wasted duplicate investigations (each up to `max_budget_usd`), stops one pipeline corrupting another's view of the repo, and makes the auto-fix cap survive a restart, which is the whole point of a circuit breaker.
- **Fix sketch**:
  1. Add `in_flight: Arc<Mutex<HashMap<String, ()>>>` to `AppState`, keyed on the **canonicalized repo path** (not project id — two projects can share a repo). In `handle_trigger`, try-insert after the project lookup; on a hit, log `already investigating {repo} — skipping` and return. Release via a guard struct's `Drop` so a panicking pipeline cannot wedge the key forever.
  2. Prefer coalescing over dropping for the same-kind case: keep the newest trigger per repo as a pending slot and run it once the in-flight one finishes, so the last alert of a burst still gets diagnosed.
  3. Persist the breaker: serialize `last_act` + `recent` to a small JSON file under `report_dir` on `record`, reload in `Breaker::new()`. Requires swapping `Instant` for `DateTime<Utc>` (`Instant` is not serializable and is meaningless across restarts) — a contained change, and breaker.rs's existing tests pin the semantics.
  4. Longer term, ACT should work in a `git worktree` rather than checking out on the user's tree, which removes the interference at the root; step 1 is the cheap guard until then.
- **Trade-offs**: Step 1 can drop a genuinely distinct second alert (mitigated by step 2). Step 3 adds a small disk write per applied fix and a schema to maintain. Step 4 is a real refactor of `act.rs` — list it, don't bundle it.
