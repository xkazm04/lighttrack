## 1. Forecast alerts only fire when a human happens to poll the endpoint

- **Severity**: High
- **Category**: automation
- **File**: `crates/api/src/forecast.rs:183-186` (fire site), `crates/api/src/alerts.rs:276-287` (`notify_forecast`)
- **Scenario**: An operator sets a $500/mo budget and wires `LIGHTTRACK_ALERT_WEBHOOK`. Spend starts climbing on a Friday. Nobody opens the dashboard over the weekend. On Monday the budget is already blown — and only the *live* breach alert (fired from ingest) ever reached Slack. The forecast that would have said "on track to breach in about 2 days" was computed **zero times**, because `build_alerts` + `st.alerts.notify_forecast(&alerts)` are reachable only from inside the `GET /v1/forecast` request handler. The whole pre-emptive promise ("warning *before* a budget is hit rather than after") is structurally a pull, while every other alert class in the system is a push.
- **Root cause**: `get_forecast` is the sole producer of `ForecastAlert`s (`crates/api/src/forecast.rs:183`). A repo-wide grep for a scheduler/ticker finds none — every `tokio::spawn` in `crates/api/src` is a *delivery* task inside `alerts.rs`, spawned from a request or ingest. Compare the live path: `Store::record_event_with_limits` (`crates/store/src/lib.rs:276-287`) evaluates rules on every ingest and pushes breaches/warnings automatically. Forecasting has no equivalent trigger. The dedup/cooldown machinery (`ForecastAlert::dedup_key`, `crates/api/src/forecast.rs:81-86`; `should_send_key`, `alerts.rs:439`) is already built for repeated evaluation — it is designed for a loop that does not exist.
- **Impact**: Turns forecasting from a dashboard curiosity into the product's differentiator. "We told you 3 days before you blew the budget" is the sentence that sells an LLM cost tool; today it only happens if someone was already looking, which is exactly when they didn't need telling. It also makes the existing cooldown, severity tiers, and email/ntfy/webhook fan-out actually pay off.
- **Fix sketch**:
  1. Extract the pure middle of `get_forecast` into `async fn evaluate_forecast(st: &AppState, project: &str, dim, horizon, lookback) -> Result<ForecastResponse, ApiError>`; the handler becomes `authenticate` + `resolve_read_project` + `Json(evaluate_forecast(..).await?)`. No behavior change.
  2. Add `crates/api/src/forecast.rs::spawn_forecast_sweeper(st: AppState)`: a `tokio::interval` task (period from `LIGHTTRACK_FORECAST_SWEEP_SECS`, default 3600, `0`/unset-on-non-alerting = disabled) that lists distinct projects that have enabled limit rules (a `SELECT DISTINCT project_id FROM limit_rules WHERE enabled = 1` store method — cheap, and it naturally scopes the sweep to operators who actually configured a budget), calls `evaluate_forecast` with the defaults, and drops the response on the floor — the `notify_forecast` side-effect is the point.
  3. Call it from `main.rs` after `build_router`, gated on `st.alerts.enabled()`.
  4. Guard the sweep with `MissedTickBehavior::Skip` and a per-project `spawn_db` so a slow store can't pile up ticks.
- **Trade-offs**: A background loop adds steady DB load proportional to `projects × lookback`; bounding it to projects with enabled rules and defaulting to hourly keeps it near-free. Alerts now arrive without a human present, so the existing cooldown (default 3600s) matters more — but it is already keyed correctly (`forecast:{project}:{kind}:{subject}`), so a sustained trend alerts once per cooldown, not once per tick.

## 2. A scoped budget is forecast against project-wide spend

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/core/src/forecast.rs:127-151`, `crates/api/src/forecast.rs:139-149` + `:218`, `crates/store/src/sqlite/forecast.rs:16-38`
- **Scenario**: An operator follows the documented pattern from `LimitScope` ("cap gpt-4o at $5/day", `crates/core/src/limits.rs:5-8`) and creates a rule scoped to `model=gpt-4o` with a $5/day threshold, while the project as a whole burns $40/day on cheap Haiku traffic. `GET /v1/forecast` reports that rule as `current: 40`, `projected_daily: 40+` and `eta_days: <breach now>` — a permanently-red row and a permanently-firing pre-emptive alert about a cap that has never been close. Conversely a project-wide *drop* in Haiku spend will show the gpt-4o cap as "safe" while gpt-4o climbs. The forecast surface is silently wrong for every scoped rule, which is exactly the rule shape a cost-conscious operator reaches for first.
- **Root cause**: The forecast path ignores `LimitRule::scope` at both ends. `forecast_budget` (`crates/core/src/forecast.rs:127`) matches only on `rule.window`/`rule.metric` and never reads `rule.scope`. In the API, the series is picked purely by metric — `LimitMetric::CostUsd => &cost_series` (`crates/api/src/forecast.rs:139-143`) — where `cost_series` came from `store.daily_usage(&proj, ..)`, an unscoped `GROUP BY substr(ts,1,10)` over all of `events` (`crates/store/src/sqlite/forecast.rs:22-25`). `current` is read from `store.usage_since(&proj, r.window.since(until))` (`crates/api/src/forecast.rs:218`) — the *unscoped* method — even though `usage_since_scoped` exists and the live limit path already uses it (`crates/store/src/lib.rs:286-287`: `match scope { None => usage_since, Some(s) => usage_since_scoped }`). The forecast is the one consumer of `LimitRule` that never learned about scopes.
- **Impact**: Makes scoped budgets — the per-model/per-provider/per-use-case caps that are the most actionable thing in the limits subsystem — forecastable and trustworthy. It also removes a class of permanent false-positive alerts that would train operators to ignore the forecast channel entirely, which would poison finding #1's value.
- **Fix sketch**:
  1. Store: add `daily_usage_scoped(project, scope: &LimitScope, since, until) -> Vec<DailyUsage>` in `crates/store/src/sqlite/forecast.rs` next to `daily_usage` — same SQL plus `AND {col} = ?4` where `col` is `provider` | `model` | `name` chosen from `scope.kind_str()` (never interpolate `scope.value()`; bind it). Mirror `daily_cost_by_dimension`'s existing kind→column dispatch style (`:49-52`). Add a `Store` trait method defaulting to the unscoped `daily_usage` so non-SQLite backends stay compiling and merely lose precision, exactly as `usage_since_scoped` already defaults (`crates/store/src/lib.rs:370-376`).
  2. API `gather`: build a `HashMap<(LimitScope, LimitMetric), Vec<f64>>` for the distinct scopes present among `raw.rules` (typically 0–3), and fetch `current` via `usage_since_scoped` when `r.scope.is_some()` — a two-line `match` mirroring `lib.rs:286-287`.
  3. API handler: series selection becomes `match (&r.scope, r.metric)` — scoped series when present, today's project-wide series when `None` (byte-identical for unscoped rules).
  4. Surface it: add `scope: Option<LimitScope>` to `BudgetForecast` (`crates/core/src/forecast.rs:108-121`, `skip_serializing_if = "Option::is_none"`) and render `LimitScope::label()` (or `all`) in the budgets table's `Budget rule` column (`crates/render/src/forecast.rs:67-87`) — the render already has the `scope_tag()` precedent from `LimitStatus`. Include the scope in the alert message and in `ForecastAlert::dedup_key` if a rule id alone ever stops being unique.
  5. Test: extend `crates/api/src/tests_forecast.rs` with a scoped-rule case — ingest heavy traffic on model A and light traffic on model B, assert the `model=B` rule's `current`/`projected_daily` track B only and that no breach alert fires.
- **Trade-offs**: One extra store query per distinct scope per forecast (bounded by rule count, and only for scoped rules). The `Store` trait grows a method, but the defaulting pattern is already established. Non-SQLite backends keep today's imprecise behavior until they implement it — an explicit, documented degradation rather than a silent one.

## 3. Forecasts are never persisted, so nobody can tell whether they were right

- **Severity**: Medium
- **Category**: half-implemented
- **File**: `crates/store/src/sqlite/forecast.rs` (whole file — read-only), `crates/api/src/forecast.rs:188-198`
- **Scenario**: LightTrack tells an operator "projected $45 next 30d" and "on track to breach in about 5 days". Two questions follow immediately and neither is answerable: *was it right last time?* and *is this forecast worse than yesterday's?* Every call to `GET /v1/forecast` computes from scratch and returns; the `ForecastResponse` — `generated_at` timestamp and all — is dropped the moment the JSON is serialized. An operator who wants to know whether the model over-predicts (it structurally does: `days_until_cumulative` is documented as a "conservative roll-off-free estimate that errs early", `crates/core/src/forecast.rs:123-126`) has no data to check against, and there is no way to see that the projected 30d spend has doubled week-over-week.
- **Root cause**: `crates/store/src/sqlite/forecast.rs` contains only two `SELECT`s (`daily_usage`, `daily_cost_by_dimension`) and no write path — no `forecast_snapshots` table, no insert, no history query. `get_forecast` ends at `Ok(Json(ForecastResponse { .. }))` (`crates/api/src/forecast.rs:188`) with no store write. The `generated_at: until` field is computed and shipped purely as response decoration. Note this is *not* an accident of the alert path: `ForecastAlert::dedup_key`'s in-memory cooldown (`alerts.rs:439-449`) is the only memory the subsystem has, and it lives in a `HashMap` that dies with the process — so even "did we already warn about this?" does not survive a restart.
- **Impact**: Three things unlock from one small table. (a) **Forecast accuracy**: join a snapshot's `projected_cost_next_7d_usd` against the actual 7-day sum from `daily_usage` and report a MAPE — turning "trust me" into a measured claim, and giving a principled way to tune `DEFAULT_ALPHA`. (b) **Trend-of-the-trend**: "your projected 30d spend is up 40% from last week" is a stronger alert than any point-in-time projection. (c) **Restart-durable alert dedup**, replacing the process-local cooldown map for forecast alerts.
- **Fix sketch**:
  1. Schema: `forecast_snapshots(id TEXT PK, project_id TEXT, generated_at TEXT, horizon_days INT, lookback_days INT, projected_daily_cost_usd REAL, projected_cost_next_7d_usd REAL, projected_cost_next_30d_usd REAL, trend_level REAL, trend_slope REAL, trend_n INT)` + `INDEX (project_id, generated_at)`. Keep it to the *spend headline* only — per-budget/per-margin rows are re-derivable and would balloon the table.
  2. Store: `insert_forecast_snapshot(&Snapshot)` and `list_forecast_snapshots(project, since, until)` in `crates/store/src/sqlite/forecast.rs`, following the module's existing `params!` + `fmt_ts` conventions.
  3. API: write the snapshot inside the existing `spawn_db` hop (or a fire-and-forget spawn — a failed snapshot must never fail the response), rate-limited to at most one row per project per hour so a dashboard on a 5s poll doesn't flood the table. Add `GET /v1/forecast/history?project=&days=` returning snapshots joined to realized actuals, with an `accuracy: { mape_7d, samples }` block.
  4. Render: a `forecast_history` report in `crates/render/src/forecast.rs` — a "Forecast vs actual" table (Date | Projected 7d | Actual 7d | Error %) using the existing `Table`/`money` helpers.
  5. Retention: a `DELETE FROM forecast_snapshots WHERE generated_at < ?` older than N days (default 90), run from the sweeper in finding #1 if built.
- **Trade-offs**: One new table and a write on a read endpoint — mitigated by the hourly per-project cap and by making the write best-effort. Accuracy reporting will publicly expose that the conservative month-window model errs early; that is a feature (it is already documented behavior), but the copy should frame it as "errs early by design" rather than shipping a bare error percentage that reads as a bug.
