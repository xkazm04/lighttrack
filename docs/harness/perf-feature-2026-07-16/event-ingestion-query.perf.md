## 1. Firestore admission check reads the entire rolling window — every single ingest

- **Severity**: Critical
- **Category**: unbounded-result
- **File**: `crates/store-firestore/src/events.rs:58-71` (reached from the default `Store::insert_event_checked`, `crates/store/src/lib.rs:281-293`; called per event from `post_event`, `crates/api/src/events.rs:163`)
- **Scenario**: Firestore-backed deployment, one project with a `cost/30d` (or any windowed) limit rule enabled — i.e. the normal configuration, since limits are a headline product feature. The project has been ingesting for a month and has 1M events in the window. Every subsequent `POST /events` runs admission control.
- **Root cause**: `usage_since` issues `rest.query(COLL, &filters, None, None)` — the `limit` argument is `None`, so `build_sq` (`crates/store-firestore/src/rest.rs:139-168`) emits a `structuredQuery` with **no `limit` and no `select` projection**. Firestore streams back every matching document in full, including the `input`/`output` payload strings, and the sums are then computed client-side in the Rust `for` loop. `evaluate_admission` invokes the closure once per distinct `(window, scope)` key, so N rule-windows multiply it. The module header's premise — "Firestore has no GROUP BY/SUM" — is stale: `runAggregationQuery` has supported `COUNT` since 2020 and `SUM`/`AVG` since 2023.
- **Impact**: Ingest cost is **O(events already in the window)** per event, so total cost over a window is **O(N²)** in reads, and Firestore bills per document read. At 1M in-window events, one ingested event = 1M doc reads ≈ **$0.30 at $0.03/100k reads**, to record an event whose own LLM call may have cost a fraction of a cent — the observability tool costs 100× the thing it observes. Latency is equally fatal: multi-MB (payloads included) of response body pulled synchronously inside `spawn_db` before the 200 is returned; p99 ingest goes to tens of seconds and the whole `spawn_db` pool saturates. This does not degrade gracefully — it falls over as soon as one project accumulates real history.
- **Fix sketch**: Add an aggregation path to `Rest` (`POST …:runAggregationQuery`) wrapping the same `structuredQuery` in `aggregations: [{count:{}}, {sum:{field:{fieldPath:"cost_usd"}}}, {sum:{fieldPath:"input_tokens"}}, {sum:{fieldPath:"output_tokens"}}]`, and rewrite `usage_since` to a single aggregation call returning `Usage` directly. Firestore bills an aggregation at 1 read per 1024 index entries scanned → ~1000× read reduction and no payload transfer. Requires a composite index on `(project_id, ts)`. Same treatment applies to `revenue.rs`, which mirrors this shape.
- **Trade-offs**: Aggregation queries need the composite index deployed before the code path goes live (add to the index config, else the query 400s). `SUM` over float `cost_usd` accumulates in a slightly different order than the current client-side loop, so summed values can differ in the last ulp — irrelevant against limit thresholds. If a deployment must stay on plain `runQuery`, the interim mitigation is passing a `select` projection of just the three summed fields plus a hard `limit`, but that only reduces the constant, not the O(N) — the aggregation query is the actual fix.

## 2. Firestore `cost_summary` reads the whole `events` collection, and the query window is silently discarded

- **Severity**: High
- **Category**: unbounded-result
- **File**: `crates/store-firestore/src/events.rs:30-56`
- **Scenario**: An operator opens the cost dashboard, or `GET /api/costs?since=<yesterday>` polls on an interval. Firestore backend, 1M events in the collection.
- **Root cause**: Two compounding causes. (a) `rest.query(COLL, &project_filter(project), None, None)` again passes `limit: None` with no projection, so the whole (project-filtered, or with no project filter the **entire**) collection is pulled down and grouped client-side into a `BTreeMap`. (b) The Firestore store does not override `cost_summary_windowed`, so the default (`crates/store/src/lib.rs:339-346`) **drops `since`/`until` on the floor** and delegates to unwindowed `cost_summary`. The narrow window the caller in `get_costs` (`crates/api/src/events.rs:249-262`) carefully parsed never reaches the query — asking for one day of costs reads all history.
- **Impact**: One dashboard load = one doc read per event ever recorded (1M reads ≈ $0.30, plus payload bytes over the wire and the full working set held in memory in `docs`), regardless of how narrow the requested window is. Unauthenticated-by-project reads (`project: None` → empty filter) scan every project's events at once. A polling dashboard turns this into a recurring per-minute charge. Bounded only by collection size, which grows forever — this is the query-side twin of finding #1.
- **Fix sketch**: Override `cost_summary_windowed` on `FirestoreStore` and have `cost_summary` delegate to it with `None, None`. Push the window into the filter list (`ts GREATER_THAN_OR_EQUAL`, `ts LESS_THAN`) alongside `project_id`. Because the grouping keys are low-cardinality, the efficient form is a `runAggregationQuery` per `(provider, model)` pair; the simpler first step that removes the unboundedness is to pass a `select` projection of the six fields the loop actually reads (`project_id, provider, model, input_tokens, output_tokens, cost_usd`) plus a sanity `limit`, which stops the payload transfer immediately. Needs a `(project_id, ts)` composite index.
- **Trade-offs**: Pushing `ts` bounds into the filter changes results for Firestore callers that pass `since`/`until` — correctly (today they are ignored), but any snapshot test pinning the current wrong-but-stable output will move. A `select` projection alone reduces bytes and memory but not the read count; only aggregation or a real `limit` does that.

## 3. `scrub` makes two full regex passes per rule and allocates for every string leaf, clean or not

- **Severity**: High
- **Category**: hot-path
- **File**: `crates/anon/src/lib.rs:48-62` (driven per event by `scrub_value`, `crates/api/src/redact.rs:100-113`, from `prepare_event`, `crates/api/src/events.rs:29`)
- **Scenario**: `LIGHTTRACK_REDACT_INGEST=all`. Every customer LLM call POSTs an event carrying its captured `input` (a chat array — often 10-50KB of prompt across many string leaves) and `output`. Redaction runs synchronously on the ingest path before admission.
- **Root cause**: Per rule, `scrub` runs `rule.re.find_iter(&out).count()` — a complete scan of the haystack purely to obtain a count — and then, if non-zero, `rule.re.replace_all(...)`, which internally re-finds every match: **two full passes where one suffices**. With 9 rules that is up to 18 scans of every string. Additionally `let mut out = text.to_string()` allocates a copy of every string leaf unconditionally, and each matching rule's `.into_owned()` allocates another. The overwhelmingly common case — clean text, `redactions == 0` — still pays 9 full regex scans plus one wasted allocation and copy per leaf, and the two costliest patterns are exactly the ones that never match: `\b(?:\d[ \-]?){13,19}\b` and `\+?\d[\d\s().\-]{8,}\d` compile to large bounded-repeat automata.
- **Impact**: ~2× the necessary regex work on the hottest path in the codebase, plus an allocation per string leaf per event. A 50KB payload with 40 leaves is ~450KB of regex scanning where ~50KB would do, and 40 avoidable `String` allocations, on every single ingest — an allocation storm at ingest rates, sized by customer prompt length rather than by anything we control. The single-pass `RegexSet` prefilter below collapses the clean-text case (the majority of leaves) from 9 scans to 1.
- **Fix sketch**: Two independent changes, both local to `scrub`. (a) Replace the count-then-replace pair with one `replace_all` using a counting replacer closure (`|caps: &Captures| { n += 1; placeholder }`), and keep the result as a `Cow` — `Cow::Borrowed` on no match means zero allocation. (b) Add a `OnceLock<RegexSet>` built from the same 9 patterns in the same order; call `set.matches(text)` once, and iterate only the rules it reports. Return `ScrubResult { text: text.to_string(), redactions: 0 }` early when the set reports nothing, so clean leaves cost exactly one pass. Rule order — load-bearing, per `ip_is_not_eaten_by_phone_or_cc` — is preserved by iterating `matches()` in index order and re-checking the set after each substitution, or more simply by using the set only as a skip-filter for the first pass over the original text and letting a rule that the set flagged run its normal `replace_all`.
- **Trade-offs**: The `RegexSet` prefilter is computed against the *original* text, but earlier rules rewrite the haystack (an `<EMAIL>` substitution can remove a later rule's match, and placeholder text can in principle create one), so the set can only be used to **skip** rules it says cannot match anything in the original — a rule it flags must still run and may legitimately find nothing. That is a conservative, no-behavior-change use. `RegexSet` adds one more compiled automaton to the `OnceLock` (startup cost, once). The existing property suite in `crates/anon/src/tests.rs` — 3000 rounds of generated PII plus the idempotence and false-positive guards — pins the observable behavior tightly enough to make this refactor safe; it should be run before and after unchanged.
