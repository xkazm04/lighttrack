# Feature Scout — Ingest Hardening & Idempotency

> Total: 3
> Critical: 1 | High: 1 | Medium: 1 | Low: 0

## 1. Ingest has no idempotency key — a retried write is indistinguishable from a bug

- **Severity**: Critical
- **Category**: capability-gap
- **File**: `crates/api/src/idempotency.rs:1-63`, `crates/api/src/events_batch.rs:129-132`, `crates/api/src/events.rs:161-168`
- **Scenario**: An SDK author writes the retry loop every telemetry client needs. A POST of 200 events times out at 2s (the Rust client's `reqwest` timeout, `clients/rust/src/lib.rs:56`) *after* the server committed the rows. The client retries. The second attempt returns HTTP 200 with every item `{"status":"invalid","reason":"conflict: …"}` — or, on the single-event path, a bare 409. The client cannot tell "you already have these, all good" from "these events were malformed and are gone forever". The only safe options are both wrong: retry (risk double-count) or drop (risk data loss). So the shipped SDK does neither — `let _ = req.send()` (`lib.rs:66`) discards the outcome and never retries at all.
- **Root cause**: Despite the module name, `idempotency.rs` implements *only* `SeenWebhooks`, and grep confirms its sole consumer is `billing.rs:66/90` (Polar webhook deliveries). No ingest route reads an `Idempotency-Key` header; `AppState.seen_webhooks` never reaches `events.rs` or `events_batch.rs`. The event PK is the *only* dedup mechanism, and it surfaces as an error: `events_batch.rs:131` maps a store `Conflict` to `BatchItem::Invalid { reason: e.to_string() }`, with the comment framing duplicate-id as a client mistake rather than a replay. The context's stated goal — "idempotency-key dedup so retried writes do not double-count" — is unbuilt on the ingest side.
- **Impact**: Safe retry is the precondition for an SDK that doesn't lose data. Every competitor (LangSmith, Langfuse, Helicone) ships a buffering client with retry+backoff; LightTrack cannot until the server defines replay semantics. Closing this also unblocks a real batching client, which today would be reckless to build: one dropped response = 500 lost or doubled events.
- **Fix sketch**:
  1. **Reframe duplicate-id as replay, not error.** Have the store's insert distinguish "PK collision, identical payload" from "PK collision, different payload". Same payload → return the original outcome as `Accepted { id, cost_usd, duplicate: true }` (batch) / 200 with `duplicate: true` (single). Different payload under the same id → keep 409, which is now a *true* conflict and a meaningful message.
  2. **Add `Idempotency-Key` for the request envelope.** Accept the header on both ingest routes; key an in-process replay cache on `(project_id, key)` holding the serialized `BatchResponse`. On a hit, return the stored response verbatim with `Idempotency-Replayed: true`. Reuse the exact `SeenWebhooks` shape (bounded `HashSet` + FIFO, `check_and_insert`/`forget`, fail-open on poison) — generalize it to `SeenKeys<V>` and let `SeenWebhooks` be `SeenKeys<()>`; both then live honestly in `idempotency.rs`.
  3. **Make the cache a fast path, not the guarantee.** Step 1's PK-based replay is the durable backstop across restart/eviction — exactly the two-layer composition `idempotency.rs:8-12` already documents for webhooks. Say so in the module doc.
  4. **Then** give the Rust client a retry-on-5xx/timeout loop keyed by a per-batch `Idempotency-Key`, and stop swallowing the response.
- **Trade-offs**: Step 1 needs a payload-equality check (hash the event, store it, or compare columns) — real store work. The replay cache is process-local, so a multi-replica deploy behind a load balancer gets cache misses; that's fine precisely because step 1 is the backstop, but the doc must say it. Turning today's 409 into a 200 is a behavior change for any client that depends on it — `tests_ingest.rs:519-540` pins the current 409 and would be rewritten to assert the same-payload/different-payload split.

## 2. No ingest-drop visibility: "why are my events missing?" has no answer

- **Severity**: High
- **Category**: developer-experience
- **File**: `crates/api/src/rejections.rs:1-12, 74-114`, `crates/api/src/events_batch.rs:85-98`
- **Scenario**: A team wires LightTrack into a service and sees 6,000 of 10,000 daily calls in the dashboard. The client is fire-and-forget and swallows every error. Nothing in the product can say where the other 4,000 went: were they 400s on an empty model? Skew-rejected? 413'd for body size? Never sent at all? The engineer's only recourse is adding logging to their own app and re-running — which is the moment the evaluation ends.
- **Root cause**: `RejectionLedger` is deliberately narrow: it counts *only* limit breaches, keyed `(project, metric, window, scope)` (`rejections.rs:35`), fed only from `events::record_rejection` on an enforcing breach. Every other drop is fire-and-forget. Validation failures are returned as a string and discarded (`events_batch.rs:97` builds `BatchItem::Invalid` and keeps no tally; `events.rs:156` maps to a 400 and returns). The 400s from `events_batch.rs:64` (empty batch) and `:68` (over `max_batch`) and the 413s from the body-limit layers leave no trace at all. The ledger's own doc — "an observability tool going blind exactly when limits bite is a product flaw" — makes the case; the argument just stops one category short.
- **Impact**: This is the single highest-leverage DX fix in the ingest surface, because it converts the worst debugging experience the product can produce (silent data loss, unfalsifiable) into a dashboard read. It is also a natural upsell surface — "we rejected 4,000 events, here's why" is the pitch for fixing the integration rather than churning.
- **Fix sketch**:
  1. Generalize the ledger to a **drop ledger**: widen the key to `(project, DropReason, detail)` where `DropReason` is an enum — `LimitBreach{metric,window,scope}` (today's case, unchanged), `InvalidModel`, `TsSkew`, `UnresolvedProject`, `DuplicateId`, `BatchTooLarge`, `BodyTooLarge`. Keep the exact `Entry` shape (count, est_cost, first_ts, last_ts), the 24h TTL prune, and the fail-open poison handling — all of it transfers.
  2. Record from the drop sites: `events_batch.rs:85` and `:97` (per-item invalid), `events.rs:156` (single-event 400), the batch guards at `events_batch.rs:63-72`. Body-limit 413s fire before the handler, so catch them in a thin middleware layer or accept that gap and document it.
  3. Expose `GET /v1/ingest/health?project=…` → `{ accepted_24h, drops: [{ reason, count, est_missed_cost_usd, first_ts, last_ts, sample_detail }] }`. The `/v1/limits/status` `rejected` block stays as-is (it's limit-specific and already documented); this is the wider view.
  4. Carry one **sample detail** per bucket (e.g. the offending model string, the skew seconds) — a count alone tells you *that* you're dropping, a sample tells you *what to fix*.
- **Trade-offs**: The ledger is process-local RAM and resets on restart — already true and already documented as acceptable (`rejections.rs:10-12`); the same honesty note must extend to the new endpoint, since "drops" reading zero after a restart must not be mistaken for "no drops". Sample details are attacker/PII-adjacent (a model string is low-risk; an arbitrary field would not be) — keep samples to a fixed allowlist of scalar fields, never event content. Cardinality is bounded by the enum, unlike a free-text reason key.

## 3. Batch item results are unaddressable and their errors are unclassifiable

- **Severity**: Medium
- **Category**: developer-experience
- **File**: `crates/api/src/events_batch.rs:29-46, 91-98, 129-132`
- **Scenario**: An SDK author writes the batch-response handler. For each non-accepted item they want to answer: which event was it, and should I retry it or drop it? Neither is available. `BatchItem::Invalid` carries **only** `reason: String` — no `id`, no `index` — so the item must be correlated by array position against the request. And `reason` is prose assembled from three unrelated sources (`"project_id is required"` at `:86`, a validator message at `:97`, a store `Display` at `:131`), so branching on it means substring-matching English. The library ends up logging `reason` and dropping the event, which is exactly the silent loss of finding #2.
- **Root cause**: `Accepted`/`Rejected` carry `id` but `Invalid` (`events_batch.rs:43-45`) does not — defensible in that a malformed item may have no usable id, but the *index* always exists and is never emitted. Order-preservation is the sole addressing mechanism, and it's load-bearing yet unstated in the response (only in the module doc and `tests_ingest.rs:432`). Separately, no variant carries a stable machine-readable code, while the sibling single-event path *does* have one — `tests_ingest.rs:249` and `:537` assert `body["error"]["code"] == "rate_limited"` / `"conflict"`. The batch path silently drops that taxonomy on the floor.
- **Impact**: Makes the batch endpoint programmatically usable rather than human-readable-only. It is the precondition for a client that can act per item — retry the transient, drop the malformed, surface the rest — which is what turns the batch route from an endpoint into a feature. Small, self-contained, and directly enables finding #1's retry loop.
- **Fix sketch**:
  1. Add `index: usize` to all three variants — populate from the loop counter at `:80` and from `valid_idx[k]` at `:133`. Positional correlation becomes explicit instead of load-bearing-but-undocumented.
  2. Add `id: Option<String>` to `Invalid` — the client-supplied id when present, `None` when the item is too malformed to have one.
  3. Add a stable `code: &'static str` to `Invalid` / `Rejected`, reusing the taxonomy the single path already returns: `"bad_request"`, `"conflict"`, `"rate_limited"`. This means `prepare_event` returning a typed error rather than `String` — a small refactor of `events.rs:25` that improves both paths and feeds finding #2's `DropReason` enum from the same source.
  4. Stop leaking `e.to_string()` from the store at `:131`. Map the error to its code and a controlled message; the raw `Display` is an internal detail on the product's most public surface.
  5. Document order-preservation and the tri-state `status` union in the OpenAPI/README, not only in the module doc.
- **Trade-offs**: Additive JSON fields, so existing consumers are unaffected — but the response grows for large batches (3 fields × 500 items); `index` is the only unconditional addition and it's small. Step 3's typed-error refactor touches `events.rs`, the single path, and `tests_ingest.rs:434`. Best done as one change with finding #2, which needs the same enum.
