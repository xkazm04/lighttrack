## 1. Queue health surface: the vanished-device alert never fires, because only the device triggers the sweep
- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/api/src/relay.rs:178-193`
- **Scenario**: The operator's laptop is closed for the weekend (or `lt-agent` crashed / the device key rotated). Apps keep enqueuing `xprice/reprice-summary` tasks. `docs/RELAY.md:39-41` promises that dead-lettering "alerts through the existing channels … from both paths: a failure report that exhausts the attempts, and **the pre-lease sweep that catches vanished devices**", and `docs/RELAY.md:155` sells the ~20h retry envelope + dead-letter alerts as the mitigation for the single-device SPOF. In reality the operator hears nothing — ever — and finds out days later when an app's results never landed.
- **Root cause**: `sweep_relay_dead()` is called from exactly one place: inside `lease_tasks`, at the top of the long-poll loop (`crates/api/src/relay.rs:180-184`). The sweep is driven by the very device whose absence it is meant to detect. No device polling ⇒ no lease request ⇒ no sweep ⇒ no `notify_relay_dead`. Worse, the sweep only touches `status='leased'` rows (`crates/store/src/sqlite/relay.rs:80-93`, `crates/store-pg/src/relay.rs:93-106`); a device that dies *between* runs leaves everything in `queued`, which no code path ever ages out or alerts on — those tasks are invisible forever, not merely late. There is also no device liveness record at all: `device` is stamped only as a side effect of a successful lease (`sqlite/relay.rs:108`), so "when did the device last check in?" is unanswerable, and the five shipped routes (`crates/api/src/main.rs:302-305`) expose no aggregate — `list_tasks` (`relay.rs:125-139`) returns rows with `limit` capped at 1000 and no counts, ages, or device view.
- **Impact**: Turns the documented SPOF mitigation from aspirational into real. The operator learns within minutes ("device silent 15m, 12 tasks queued, oldest 40m") instead of never, which is the difference between relay being trustworthy for app-facing work and being a thing you babysit. It also gives the ops dashboard the one relay widget it needs, and makes `wait_relay_task` callers diagnosable.
- **Fix sketch**: (1) Add a `relay_devices` last-seen record (or reuse a tiny KV): stamp `device`, `last_seen_at`, and last-lease result on **every** `lease_tasks` call, not just when tasks come back — the request itself is the heartbeat. (2) Move the sweep off the device's critical path: spawn a background tick in `main.rs` (~60s, alongside the existing alert plumbing) that calls `store.sweep_relay_dead()` + `st.alerts.notify_relay_dead(&dead)`, and drop it from the lease loop or keep it as a cheap fast-path. (3) Add a new alert event `relay_device_silent` — fire via `st.alerts` when `now - last_seen_at > LIGHTTRACK_RELAY_DEVICE_SILENT_SECS` (default ~900) **and** at least one task is `queued` and due; the existing `should_send_key` cooldown (`alerts.rs:296`) already de-dupes it. (4) Add `GET /v1/relay/health` (project key scoped / admin global) returning `{ counts_by_status, oldest_queued_age_secs, oldest_due_age_secs, devices: [{device, last_seen_at}] }`, backed by a `relay_queue_stats` store method (one `GROUP BY status` + `MIN(created_at)`) so it doesn't page rows.
- **Trade-offs**: A background tick means the sweep runs on every replica; both sweep impls are already single-statement `UPDATE … RETURNING` (`store-pg/src/relay.rs:93-106`), so concurrent sweeps are safe and the alert cooldown absorbs duplicate notifications. New store method needs a default "empty" impl for Firestore, matching the existing pattern.

## 2. Dead-lettered tasks are a one-way door: no retry, no cancel, and the idempotency key is poisoned forever
- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/api/src/relay.rs:53-97`
- **Scenario**: The alert fires: `relay task 'x' (xprice/reprice-summary) dead-lettered`. The cause was environmental — the action folder was mid-edit, the connector's callback host was down, the device was offline past the ~20h envelope. The operator fixes the cause in 30 seconds and then… has no way to run the task. The only recourse is hand-editing the production Neon DB. Worse, the app can't fix it either: re-calling `lt.relay_task("xprice/reprice-summary", {...}, idempotency_key="order-42")` returns the **dead** task, immediately and forever, so order-42 can never be processed again under its natural key. Symmetrically, an app that enqueues a 5-hour-retry task and then learns the order was cancelled has no way to withdraw it — the device will still burn a subscription-window run on it.
- **Root cause**: The relay router ships five routes (`crates/api/src/main.rs:302-305`) and none of them mutate a task's lifecycle from the app/operator side — `enqueue_task`, `get_task`, `list_tasks` (read-only), plus the two device-key routes. There is no `POST /v1/relay/tasks/:id/retry`, no `DELETE`/cancel, and `RelayTask.status` (`crates/core/src/relay.rs:17-19`) has no `cancelled` variant. The idempotency lookup is status-blind: `enqueue_task` calls `find_relay_task_by_key(&project2, &key)` (`crates/api/src/relay.rs:64-70`) and returns whatever it finds, and the store queries filter on `(project_id, idempotency_key)` only, with no status predicate (`crates/store/src/sqlite/relay.rs:50-56`; `crates/store-pg/src/relay.rs:61-71`). `docs/RELAY.md:53-54` frames the key purely as a dedupe device, never considering that a terminal-failed row keeps squatting on it. Cancellation has no lease-side story either: `lease` selects on `status`/`next_attempt_at` only (`sqlite/relay.rs:107-115`), so there's nothing for a cancelled state to key off — but equally nothing blocking one.
- **Impact**: Closes the loop that Phase 3's dead-letter alerts open. Today the alert is a notification with no action attached — the product tells you it broke and hands you a SQL console. A retry button (and its SDK/CLI equivalent) makes the ~20h envelope survivable and makes relay usable for money-adjacent app work, where "order-42 can never be retried" is a real business incident. Cancel additionally protects the scarcest resource in the design — subscription-window capacity (`docs/RELAY.md:125-127`) — from work that is already known to be pointless.
- **Fix sketch**: Three shippable steps. (1) **Retry**: add `POST /v1/relay/tasks/:id/retry` (project key, own project — reuse the `Principal::Project` ownership check from `get_task`, `crates/api/src/relay.rs:110-114`), backed by a `requeue_relay_task(id)` store method that, for a `dead` (or `succeeded`, if `force`) row, sets `status='queued'`, `attempts=0`, `error=NULL`, `next_attempt_at=now`, `lease_deadline=NULL` — SQLite as a plain `UPDATE … WHERE id=? AND status='dead'`, Postgres inside the existing `settle`-style transaction with `SELECT … FOR UPDATE`. Optionally accept a fresh `payload` for the "the params were wrong" case. (2) **Cancel**: add `status='cancelled'` to the `RelayTask` doc comment (`crates/core/src/relay.rs:17`) and `POST /v1/relay/tasks/:id/cancel`, which only transitions from `queued` (refuse `leased` — the device is already running; let the settle land, since `settle` already no-ops on non-leased rows, `sqlite/relay.rs:133-135`). `lease`'s `WHERE status='queued' OR status='leased'` predicate excludes `cancelled` for free. (3) **Un-poison the key**: make the idempotency lookup ignore terminal-failure rows — pass a status filter into `find_relay_task_by_key` so `dead`/`cancelled` rows don't collapse a fresh enqueue (or, more conservatively, keep returning them but let the new `retry` route be the documented escape hatch). Update `docs/RELAY.md:53-54` + the API table at `docs/RELAY.md:58-64`, and add SDK `retry_relay_task`/`cancel_relay_task` wrappers.
- **Trade-offs**: Step (3) changes an existing contract — an app that re-enqueues `order-42` after a dead-letter would silently get a *new* task rather than the dead one, which is almost certainly what it wants but is a behavior change worth a doc line and a test. Cancel adds a fourth terminal status that any dashboard/status consumer must handle. `attempts=0` on retry re-arms the full ~20h envelope; that's the point, but a runaway action could be retried indefinitely by an app loop — worth a `retry_count` column if that ever bites.

## 3. Fail-fast is missing on both ends: a typo'd `action_type` costs 20 hours to discover, and the device can't say "never retry this"
- **Severity**: Medium
- **Category**: workflow
- **File**: `crates/api/src/relay.rs:60-62`
- **Scenario**: A developer wiring a new xprice app calls `lt.relay_task("xprice/reprice-summary ", {...})` — trailing space, or `xprice/repricing-summary`, or the action folder simply isn't on the device yet. The enqueue returns `200 OK` and a healthy-looking `queued` task. The device leases it, fails to resolve the action in its library, settles `failed` — and the cloud dutifully requeues it for **5 hours** (`RELAY_DEFAULT_RETRY_INTERVAL_SECS`, `crates/core/src/relay.rs:15`), four times over. The developer learns their action name was wrong roughly a day later, via a dead-letter alert. Each of those four pointless attempts also stamped a `$1.00` flat-cost event (`crates/api/src/relay.rs:255-258`), so the cost dashboard now reports $4 of Claude spend for a typo that never invoked Claude.
- **Root cause**: Two halves of one missing idea. (a) The cloud validates `action_type` as "non-empty after trim" and nothing more (`crates/api/src/relay.rs:60-62`); the string is otherwise opaque until it reaches the device's library resolver. The cloud has no registry of which action types exist for a project, so it cannot reject an unknown one at the only moment the caller is still listening. (b) `RelayOutcome` offers exactly three settles — `Succeeded` / `Failed` / `Deferred` (`crates/core/src/relay.rs:79-91`) — and `Failed` unconditionally means "retryable": both stores branch only on `attempts >= max_attempts` (`crates/store/src/sqlite/relay.rs:147-158`; `crates/store-pg/src/relay.rs:169-187`), never on the *nature* of the failure. The device already knows "no such action in my library" is 100% deterministic and will fail identically on attempt 4, but the wire protocol (`crates/api/src/relay.rs:224-238`) gives it no way to say so. Note the design already has precedent for a non-standard attempt accounting path — `Deferred` hands the attempt *back* (`sqlite/relay.rs:159-169`) — so the retry ladder is clearly meant to model failure *kinds*; permanent failure is the missing rung.
- **Impact**: Turns a ~20h feedback loop into a sub-second one, which is most of the developer experience of onboarding a new action (`docs/RELAY.md:129-134` explicitly pitches "each xprice app is just a client" — this is the moment that promise is tested). Also stops fabricating flat-cost spend for runs that never touched the subscription, keeping the "solid usage overview from day one" claim (`docs/RELAY.md:87`) honest.
- **Fix sketch**: (1) **Cheap syntactic gate now**: in `enqueue_task`, validate `action_type` against the same shape the device's resolver enforces — non-empty segments, no `.`/`..`/absolute/backslash components, a conservative `[a-z0-9._-]` charset per segment — and reject with `ApiError::bad_request`. This is pure win with no new state and catches the trailing-space/`..` class immediately. (2) **Registry gate next**: let the device publish its action catalog on lease (an `actions: ["xprice/reprice-summary", …]` field on `LeaseReq`, stored next to the `last_seen_at` heartbeat from finding #1), and have `enqueue_task` reject an `action_type` absent from the last-known catalog — behind `LIGHTTRACK_RELAY_STRICT_ACTIONS` so it fails open until the operator opts in (a cold cloud with no catalog yet must not reject everything). Surface the catalog on the `GET /v1/relay/health` response. (3) **Permanent failure**: add `RelayOutcome::Failed { error, permanent: bool }` (or a fourth `Rejected(String)` variant), accepted as `"status": "failed", "permanent": true` in `ResultReq` (`crates/api/src/relay.rs:195-215`); when set, both store impls skip the retry ladder and go straight to `dead` with the error recorded. The device sets it for unresolvable-action / malformed-payload errors. Keep logging the `$1` event only when a Claude run actually occurred — i.e. suppress the event for `permanent`-rejected settles, the same way `deferred` is suppressed today (`crates/api/src/relay.rs:255`).
- **Trade-offs**: (2) couples the cloud to device state and needs the fail-open default plus an agent-side change, so it's the part to defer if the wave is tight — (1) and (3) are independently valuable and touch only the files under review plus the outcome enum. (3) widens `RelayOutcome`, a `pub` core type, so the Firestore default impls and the conformance suite need a pass; the wire field is additive and older agents omitting it keep today's behavior. A device that over-uses `permanent` loses the retry safety net — document it as "only for errors that are provably deterministic".
