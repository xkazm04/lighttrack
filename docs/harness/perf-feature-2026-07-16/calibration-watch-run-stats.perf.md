## 1. `previous_kappa` fetches 500 unfiltered scores per cycle and filters client-side

- **Severity**: High
- **Category**: hot-path / full-scan (client-side filter over a server-side page)
- **File**: `crates/runner/src/calibrate_watch.rs:167-179` (called from `run_cycle`, line 153)
- **Scenario**: A production project posts application scores continuously (`POST /v1/scores` is the app's normal scoring path). The sentinel runs hourly against the same project. Once the project accumulates >500 scores *newer than* the last calibration row — for a busy project, well under a day — the reserved-rubric row is no longer inside the newest-500 page.
- **Root cause**: `GET /v1/scores?project=…&limit=500` has no rubric filter (`crates/api/src/scores.rs:34-46` — the query struct exposes only `project` and `limit`; the store's `list_scores(project, limit)` at `crates/store/src/lib.rs:445` has no rubric predicate either). So the runner pulls the newest 500 rows across *all* rubrics and does `scores.into_iter().find(|s| s.rubric == reserved)` in the client. The needle is one row per cycle; the haystack is every score the app writes. Each row carries its full `reasoning` blob — for calibration rows that is the whole metrics JSON, and for app rows it is whatever the judge emitted — so this is a multi-hundred-KB JSON transfer + parse to extract a single `f64`.
- **Impact**: Two costs, the second worse than the first. (a) Every cycle serializes/transfers/parses ~500 score rows to read one number — wasted DB scan, wasted bandwidth, wasted serde. (b) Correctness-by-way-of-scale: past ~500 intervening scores the `find` misses, `previous_kappa` silently returns `None`, and `assess_drift` degrades to bar-only checking — `DriftLevel::Drift` becomes permanently unreachable and `Δκ` never prints. The feature's headline capability (catch a *gradual* slide that still clears the bar) quietly stops working on exactly the busy projects that need it, with no error and no log line. Note the API's `limit` is clamped to 1000 (`scores.rs:45`), so raising the constant only buys a 2× reprieve.
- **Fix sketch**: Add a `rubric` filter to the scores read path — `ListQuery { rubric: Option<String> }` in `crates/api/src/scores.rs`, threaded into `Store::list_scores` as an optional `WHERE rubric = ?` predicate across the three drivers — then have `previous_kappa` request `/v1/scores?project={pr}&rubric={reserved}&limit=1`. One indexed row, no client-side scan, and the drift signal stops depending on app write volume. Cheap interim (does not fix (b) at the tail, but caps the waste): keep the fetch but request `limit=1` behind the rubric filter only once the filter exists — without it, no client-side trick helps. Worth an index on `(project_id, rubric, created_at DESC)` to keep the filtered read a seek rather than a scan.
- **Trade-offs**: Touches the store trait and all three drivers (sqlite/pg/firestore) plus the API query type — additive and backward-compatible (absent `rubric` ⇒ current behavior), but it is a cross-crate change rather than a one-line runner edit. Firestore needs a composite index for the added equality+order.

## 2. Recurrence sweep pulls each benchmark's entire run history to read one timestamp (N+1, unbounded rows)

- **Severity**: High
- **Category**: N+1 / unbounded row fetch
- **File**: `crates/runner/src/recurrence.rs:56-78`, `latest_run_time` at `100-107`
- **Scenario**: `lt-runner serve` sweeps on `recur_interval`. A workspace with, say, 20 enabled projects × 10 recurring benchmarks each = 200 benchmarks. Each recurring benchmark runs on its interval, so its `benchmark_runs` history grows monotonically forever — an hourly benchmark is at ~8.7k rows after a year.
- **Root cause**: Two compounding issues. (a) N+1: the sweep does one `GET /v1/projects`, then one `/v1/projects/{id}/benchmarks` per project, then one `/v1/benchmarks/{id}/runs` per benchmark — 1 + P + B round-trips per sweep, serial and blocking. (b) Each of those B calls is unbounded: `list_runs` is `SELECT {RUN_COLS} FROM benchmark_runs WHERE benchmark_id = ?1 ORDER BY started_at DESC` with **no LIMIT** (`crates/store/src/sqlite/benchmarks.rs:83-92`; same shape in `crates/store-pg/src/benchmarks.rs:88`), and `/v1/benchmarks/:id/runs` exposes no limit param. `latest_run_time` then materializes the whole `Vec<BenchmarkRun>` — every row through rusqlite mapping, `run_from_raw`, JSON serialization, HTTP, and serde on the runner — and calls `.first()`, discarding all but row zero.
- **Impact**: Per sweep the runner transfers and parses the *entire lifetime run history of every recurring benchmark* to read B timestamps. At the scale above (200 benchmarks × ~8.7k rows × a row that carries report JSON) that is millions of rows and hundreds of MB of JSON per sweep, growing without bound as history accumulates — and it lands on the API's blocking DB pool, so it contends with live request traffic. The `has_job` short-circuit at line 69 only skips benchmarks that are *already in flight*, i.e. exactly the ones that aren't due — the steady-state path always pays. Cost grows as O(total historical runs), while the information needed is O(benchmarks).
- **Fix sketch**: Bound the read at the source. Add `limit: Option<usize>` to `list_benchmark_runs`'s query (default the current behavior, clamp like `jobs.rs:96`) and push it into `list_runs` as `ORDER BY started_at DESC LIMIT ?`; then call `/v1/benchmarks/{id}/runs?limit=1` from `latest_run_time`. That alone converts each fetch from full-history to one row and kills the unbounded growth. Better still, fold the N+1 away: expose the last-run timestamp on the benchmark list payload (a `LEFT JOIN LATERAL` / correlated `MAX(started_at)` per benchmark) so the sweep needs only 1 + P calls and no per-benchmark round-trip at all.
- **Trade-offs**: The `limit=1` fix is small and additive (API query param + store signature) and strictly reduces work — no behavior change, since only `.first()` was ever read. The join-based fix is the bigger win but changes the benchmark list response shape and adds a per-row subquery to a hotter endpoint; do the `limit=1` fix first, measure, then decide whether the N+1 still matters.

## 3. `inflight_bench_ids` truncates at 1000 jobs, making the idempotency guard fail open under backlog

- **Severity**: Medium
- **Category**: unbounded growth / self-amplifying enqueue
- **File**: `crates/runner/src/recurrence.rs:82-95` (guard consumed at line 69-73)
- **Scenario**: The job queue backs up — a worker dies, a provider rate-limits, a big dataset run stalls — and `queued` jobs exceed 1000. Sweeps keep running on `recur_interval` throughout.
- **Root cause**: The guard asks `GET /v1/jobs?status={queued,running}&limit=1000`, and the API hard-clamps `limit` to 1000 (`crates/api/src/jobs.rs:96`). Past that the response is silently truncated — no error, no `has_more`. Any `bench_run` outside the returned page is absent from the `HashSet`, so `inflight.contains(&b.id)` is false, `has_job` is false, and `is_due` (`recurrence.rs:38-51`) — whose whole in-flight branch depends on that flag — returns true for a benchmark that already has work queued. The sweep enqueues a duplicate.
- **Impact**: The failure is self-amplifying, which is what lifts it above a plain truncation bug: duplicate enqueues push the queue further past 1000, which truncates the page harder, which hides more in-flight jobs, which produces more duplicates on the next sweep. The module's stated invariant ("repeated sweeps never pile up jobs") inverts precisely when the queue is already distressed. Each duplicate is a real `bench_run` — LLM calls and provider spend — so the cost is money, not just CPU. Bounded in practice by `recur_interval` (sweeps are subsampled off the 5s claim loop, `serve.rs:36-41`, and the guard degrades only above 1000 queued jobs), which is why this is Medium rather than High. Secondary cost: two 1000-row job fetches with full `payload` JSON per sweep, of which only `bench_run` rows and one payload field are used.
- **Fix sketch**: Stop enumerating the queue to answer a per-benchmark question. Either (a) add a filtered count/exists read — `GET /v1/jobs?job_type=bench_run&status=queued,running&benchmark_id=…`, or a single grouped query returning distinct in-flight `benchmark_id`s for `bench_run` — so the result set is O(recurring benchmarks) not O(queue depth); or (b) if the page-scan shape is kept, paginate to exhaustion and, failing that, **fail closed**: when a page comes back full (`jobs.len() == limit`) the set is known-incomplete, so skip enqueueing this sweep and log, rather than enqueueing on incomplete knowledge. Fail-closed is the important half — a missed sweep costs one delayed run, a false-negative guard costs unbounded duplicate spend.
- **Trade-offs**: (a) is the real fix but needs a new job-query filter in the API + store. (b) is a few lines in the runner and removes the amplification immediately, at the cost of pausing recurrence while the queue is deep — arguably correct behavior anyway, since a >1000-job backlog is not a moment to add more work.

### Checked and deliberately not filed

- **`stats.rs` (all of it)** — `Summary::of` and `stability` are two-pass O(n) over a per-run score slice; `annotate_significance` is O(1) plus a few small allocations. The two-pass mean/variance is numerically *better* than a one-pass sum-of-squares and n is per-run case count (tens to low thousands). No hot-path cost; a Welford rewrite would be a micro-optimization with negligible impact.
- **`gate.rs`** — a 3-arm `match` on a `&str`. Nothing to optimize.
- **`watch()` re-judging the full golden set each cycle** (`calibrate_watch.rs:154-157`) — this is the feature, not a bug. Re-judging is the only way to observe judge drift; there is no incremental substitute, and the set is pinned and small. `prices` is correctly hoisted out of the loop (line 115), and `judge_set` already parallelizes at `jobs`.
- **`watch()` daemon cadence** (`calibrate_watch.rs:133`) — `sleep(interval)` runs *after* the cycle, so the effective period is `interval + cycle_duration` and the κ series is non-uniformly spaced. Real, but the drift decision compares consecutive samples rather than rates, so it costs accuracy of the time axis, not CPU or money. Not a performance finding; noted here for whoever owns the metric's semantics.
- **Sweep frequency** (`serve.rs:36-41, 82-87`) — already correctly subsampled off the 5s claim loop via `sweep_due`. The sweep's cost is finding #2's payload size, not its cadence.
