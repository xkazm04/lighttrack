## 1. Conformance suite doesn't exercise the default-bearing methods, so a wrong-answer default passes CI

- **Severity**: High
- **Category**: conformance-gap
- **File**: `crates/store/src/conformance.rs:23-37`
- **Scenario**: A cloud operator runs LightTrack on Postgres. Their dashboard filters events by provider and asks "spend in the last 24h"; the numbers look plausible and the Postgres conformance test is green — but the figures are wrong (see #2). CI gave no signal because the conformance suite never called the methods that carry silent defaults.
- **Root cause**: `run()` invokes only `events / projects_keys_limits / scores / prices / benchmarks / datasets / rubrics / jobs / admission / revenue / relay`. It never calls the trait methods that ship a default impl: `list_events_filtered`, `cost_summary_windowed`, `usage_since_scoped` (and scoped admission), `list_traces` / `list_traces_filtered` / `list_trace_events` / `list_trace_scores` / `get_trace`, `usecase_costs`, `daily_usage`, `daily_cost_by_dimension`, `tokens_by_dimension`, `customer_cost_by_model` / `_by_name`, the prompt registry, `get/update/delete_limit_rule`, and the collective leaderboard. I verified (grep over `crates/store-pg` / `crates/store-firestore`) that both cloud backends inherit *every one* of these defaults — Postgres doesn't even override the prompt registry. The conformance suite is the codebase's stated parity-enforcement mechanism ("so SQLite, Postgres, and Firestore can be held to identical behavior", `conformance.rs:1-4`), yet its blind spot is exactly the set of methods where parity actually diverges.
- **Impact**: This is the systemic root cause behind the whole class of parity bugs the parallel audits keep re-finding. A backend can silently return unfiltered rows, all-time cost for a windowed query, or empty traces and still pass conformance. Highest leverage: one suite extension turns every future silent-inheritance into a loud red test.
- **Fix sketch**: Add conformance coverage for the default-bearing methods, asserting the *filter/window actually applied*, not just a round-trip: (a) `list_events_filtered` with a provider filter returns only matching rows and a working `next_cursor`; (b) `cost_summary_windowed` with a `[since,until)` returns fewer calls than full history; (c) a scoped limit rule + `usage_since_scoped` counts only the scoped dimension; (d) `list_traces` / `get_trace` roll up an inserted multi-span trace; (e) `usecase_costs`, `cost_by_dimension` already partly covered by `revenue()` — extend to `tokens_by_dimension` and `customer_cost_by_*`; (f) a prompt create/version/label round-trip. Gate each optional-feature block the way `relay()` already does (skip on the "not supported" error) so feature-absent backends stay green while wrong-answer backends fail.
- **Trade-offs**: Cloud conformance runs only when a test env var points at a live DB, so these assertions won't gate every PR — but they convert "silently wrong in prod" into "fails the moment anyone runs the cloud suite," which is where these bugs are found today anyway.

## 2. Filtered/windowed read defaults return plausible-but-wrong data instead of erroring or degrading honestly

- **Severity**: High
- **Category**: backend-parity
- **File**: `crates/store/src/lib.rs:321-346` (`list_events_filtered`, `cost_summary_windowed`)
- **Scenario**: On Postgres/Firestore, `GET /events?provider=openai&since=…` returns the newest N events of *all* providers, unpaginated (`next_cursor` always `None`), and `cost_summary_windowed(project, since, until)` returns the project's *entire history* cost, ignoring the window. Both look like real answers.
- **Root cause**: The trait defaults silently drop their inputs — `list_events_filtered` delegates to `list_events` (ignores every `EventFilter` field and the cursor), `cost_summary_windowed` delegates to `cost_summary` (ignores `since`/`until`). This is a different, more dangerous default *policy* than the rest of the trait: the write defaults (`update_limit_rule`, `create_prompt`, `create_relay_task`…) return a clear "not supported" error; the read-only feature defaults return empty (`list_traces`, forecasting, `usecase_costs`); and `usage_since_scoped` degrades *conservatively* (over-enforces, documented `lib.rs:370-377`). Only these two return wrong-but-plausible values in the *safe-looking* direction — masking a windowed money query as all-time spend.
- **Impact**: Silent data corruption in the two surfaces most likely to drive decisions (cost/spend and event triage). No error, no empty result, no log — the exact failure mode this audit exists to surface. Affects both cloud backends today.
- **Fix sketch**: Make the defaults honest. For `list_events_filtered`, the correct-results default is feasible without keyset support: fetch `list_events` and apply the `EventFilter` predicates in Rust (loses cross-page keyset paging, but never returns rows the caller excluded) — or drop the default so unported backends fail to compile. `cost_summary_windowed` *cannot* be correctly derived from `cost_summary`, so it should be a **required** trait method (no default), using the same compile-error parity gate the codebase already trusts for writes. At minimum, if a silent default must remain, it should carry a runtime signal rather than pass as truth.
- **Trade-offs**: Removing a default forces Postgres/Firestore to implement (or explicitly `unimplemented!`/error) these before they compile — that's the point, and it's the pattern already used for the write side. The Rust-side filter fallback trades keyset pagination for correctness on unported backends, which is the right default direction.

## 3. Schema evolution is an ad-hoc per-backend ALTER list with no version table or migration runner

- **Severity**: Medium
- **Category**: migration
- **File**: `crates/store/src/sqlite/mod.rs:89-121` (`init_schema`)
- **Scenario**: A self-hoster upgrades LightTrack. A new column was added since their DB was created. On startup `init_schema` runs `CREATE … IF NOT EXISTS` (which skips their existing tables) plus a hand-maintained list of `ALTER TABLE … ADD COLUMN` statements, each wrapped in a check that swallows errors whose text contains `"duplicate column name"`. There is no `schema_version` row, no ordered migration files, and the Postgres/BigQuery schemas (`schema/*/001_init.sql`) evolve independently with no shared ledger.
- **Root cause**: Migrations are simulated by appending idempotent ALTERs and string-matching SQLite's error message. `schema/sqlite/001_init.sql` is numbered as if a migration series exists, but `002…` never materialized — every additive change instead accretes another ALTER in `init_schema` (already 6: `events.name`, three `limit_rules` columns, three `collective_entries` columns). The "already applied?" test is a fragile substring match on a backend-specific, potentially locale-dependent error string, and nothing records *which* schema version a DB is at, so there's no way to detect a partially-migrated or downgraded DB, and no parity between what SQLite ALTERs in vs. what the PG/BQ `001_init.sql` files declare.
- **Impact**: Operability/correctness risk that grows with every schema change: a renamed error string silently turns a real ALTER failure into "success"; a backend that forgets to mirror a column drifts undetected (a fresh parity gap of its own); and there's no migration story for the destructive/backfilling changes that inevitably come (the codebase already hit one such gap elsewhere — a `DEFAULT 0` column with no backfill). Not actively unsafe today, but the mechanism is at its scaling limit.
- **Fix sketch**: Introduce a minimal migration runner behind the store: a `schema_migrations(version INTEGER PRIMARY KEY, applied_at TEXT)` table, ordered `schema/<backend>/NNN_*.sql` files, and an `apply_pending()` that runs unapplied versions in a transaction and records each. Replace the string-matched ALTERs with `002_*.sql` migrations. Keep the per-backend SQL dialects but share the version ledger + runner contract so "what version is this DB, and are all backends caught up" is answerable. Add a conformance check that a freshly-initialized store reports the latest version.
- **Trade-offs**: A runner is more machinery than six tolerated ALTERs, and existing deployed DBs need a one-time baseline (stamp them at the current version so the runner doesn't re-run `001`). Worth it before the first migration that must backfill or drop data — which the current approach cannot express.
