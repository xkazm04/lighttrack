# Feature Scout — Responder Investigation & Actions

> Total: 3
> Critical: 0 | High: 2 | Medium: 1 | Low: 0

_Note on the stated roadmap hint: the crate docs' "auto-fix is deliberately out of scope for this first cut" is **stale** — `act.rs` implements a full gated auto-fix (opt-in `auto_fix`, `lt-fix/*` branch, breaker, test run, branch restore) and `pipeline.rs:60-70` wires it in. The remaining value is not "build auto-fix", it is **finishing the loop it starts** (finding 1) and **bounding the stage that is still ungated** (finding 2)._

## 1. The auto-fix branch is a dead end — no diff in the report, no push, no PR

- **Severity**: High
- **Category**: half-implemented
- **File**: `crates/responder/src/act.rs:79-104`, `crates/responder/src/git.rs:41-54`, `crates/responder/src/report.rs:51-72`
- **Scenario**: An operator gets the diagnosis email at 02:00. It says: `branch: lt-fix/systedo-dev-20260716T021500Z · applied true · tests passed`. That is all they get. To see *what changed* they must SSH to the machine running `lt-responder`, find the mapped repo, and `git show` the branch — because `act.rs:95` checks the tree straight back to `orig` and nothing ever leaves the responder host. If the responder runs anywhere but the reviewer's own laptop, a verified, test-passing fix is effectively invisible and unreachable. The most expensive artifact the product produces (a Claude-authored, test-green patch) is the one thing the report doesn't carry.
- **Root cause**: The ACT stage produces a commit and then discards every handle to it except the branch *name*. `git.rs` exposes only `is_clean`/`has_changes`/`current_branch`/`checkout_new`/`checkout`/`add_commit` — there is no `diff`, no `push`, no remote awareness at all. `ActOutcome` (`act.rs:17-24`) carries `branch/applied/tests/notes/cost_usd` but no patch text and no URL, so `render_act` (`report.rs:62-71`) and `render_act_html` (`report.rs:136-141`) can only print the metadata line. `notes = run.text` is Claude's *prose summary* of the change, not the change.
- **Impact**: Closes the investigate→fix→**review** loop. The diff in the email turns a notification into a reviewable unit of work; the PR link turns it into a mergeable one. This is what makes the responder feel like a teammate that opened a PR overnight rather than a bot that left a note about a branch on a server. It also makes `auto_fix: true` safe to actually opt into: reviewers can see the blast radius before trusting it.
- **Fix sketch**:
  1. `git.rs`: add `pub(crate) async fn diff(repo, from, to) -> Option<String>` (`git -C repo diff {from}..{to}`) and `diff_stat` (`--stat`). Cap the captured patch (e.g. 200 lines / 64 KB) with a truncation marker — an email body must not carry a 5 MB diff.
  2. `act.rs`: after `add_commit` (line 84) and *before* `checkout(&entry.repo, &orig)` (line 95), capture `diff_stat` + `diff` into new `ActOutcome` fields `diff_stat: Option<String>`, `diff: Option<String>`.
  3. `report.rs`: render the stat inline and the patch in a ```` ```diff ```` fence in `render_act`; in `render_act_html` reuse the existing `.md pre` style (`md_to_html` already renders fenced blocks, so wrapping the patch as a fenced diff string is enough — do not hand-roll `<pre>`).
  4. Optional step 2 — remote delivery, off by default: add `push: bool` to `ProjectEntry` and `git::push(repo, branch)` (`git -C repo push -u origin {branch}`). When set, push before restoring the branch and include the branch in the report.
  5. Optional step 3 — `pr: bool` on `ProjectEntry`: shell `gh pr create --draft --base {orig} --head {branch} --title ... --body {markdown report}`, capture stdout as the PR URL, and surface it as a link in both report bodies. Gate on `push` being true; treat `gh` absent as a skip with a reason, matching the existing `ActOutcome::skip` idiom.
- **Trade-offs**: Steps 4/5 move the responder from "never leaves the machine" to "writes to a remote", which is a genuine trust escalation — hence per-project opt-in, `--draft`, and never targeting `main` (it already only pushes `lt-fix/*`). Step 1-3 (diff in the report) has no material downside beyond email size, which the cap handles, and is worth shipping on its own.

## 2. Investigations are ungated and un-deduplicated — a flapping project bills full price on every webhook

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/responder/src/investigate.rs:23-25`, `crates/responder/src/breaker.rs:12-53`, `crates/responder/src/state.rs:8-12`
- **Scenario**: A provider outage or a bad deploy makes one project throw the same error continuously. The API fires a spike webhook each detection window. `classify` only diverts *transient* errors (`pipeline.rs:31-40`); anything it reads as `Class::Code` goes straight to `investigate::investigate`, which spawns `claude -p` bounded by `max_budget_usd` (default $1.0, `config.rs:114`). Ten windows in an hour = ten near-identical Claude runs, up to $10, ten near-identical Markdown files in `diagnoses/`, and ten near-identical emails to every recipient. The operator learns nothing after the first one, and the responder — a *cost-tracking* product — has become the biggest untracked cost in the room.
- **Root cause**: The `Breaker` is scoped exclusively to the ACT stage. Its own doc comment says so ("Circuit breaker for the auto-fix (ACT) stage"), `allow` is called only from `act.rs:57`, and `pipeline.rs::run_error` passes `breaker` *through* to `act` without ever consulting it before the far more frequently-executed `investigate` call at line 57. So the cheap-to-trigger, always-runs stage is unbounded while the rare, opt-in stage is bounded — exactly inverted. `run_quality` (`pipeline.rs:81`) doesn't even receive the breaker. Compounding it: `Breaker` state is `Mutex<HashMap>`/`Mutex<Vec>` of `Instant` (`breaker.rs:13-15`) held in a process-local `Arc` (`state.rs:10-11`), so every restart resets all cooldowns to zero, and `Instant` can't be serialized to fix that without a representation change.
- **Impact**: Makes the responder safe to point at a noisy production project — which is the only kind worth pointing it at. Dedup also directly improves the *signal*: one report per distinct failure, with a "seen 14× since 02:00" recurrence count, is more useful than fourteen reports, and the count itself is diagnostic information the product currently throws away.
- **Fix sketch**:
  1. Generalize `Breaker` from project-keyed to `(stage, key)`-keyed: `allow(stage: Stage, key: &str, cooldown, max_per_hour)` where `Stage` is `Investigate | Act`. Keep the existing per-key cooldown + rolling-hour cap logic verbatim — it is already correct and tested (`breaker.rs:60-77`); only the key changes. Existing ACT calls become `allow(Stage::Act, project, ...)`.
  2. Add a **fingerprint** for the investigate key: hash `project_id` + a normalized `spike.error` (lowercase, strip digits/UUIDs/hex/timestamps so `request 4f2a failed` and `request 9c81 failed` collapse). Same idea for `Drop`: `project_id` + `rubric`. Put it in a small `fingerprint.rs` with unit tests on the normalizer — this is the piece most likely to be subtly wrong.
  3. Add `investigate_cooldown_secs` (suggest 1800) and `max_investigations_per_hour` (suggest 6) to `Defaults`/`RawDefaults` (`config.rs:21-35`, `73-83`) and `Defaults::fallback`, following the existing `act_*` naming.
  4. In `pipeline.rs::run_error`, call `allow` after `classify` and before `enrich`; on `Err(why)`, log `'{project}': investigation suppressed ({why})` and return without spending. Thread `breaker` into `run_quality` and do the same. Record on dispatch (unlike ACT, which records only on *applied* — here the cost is incurred by the attempt, so an attempt must count).
  5. Track a per-fingerprint hit counter and surface it in the report header (`report.rs:20-32`) as `- recurrence: 14× since {first_seen}` — the payoff that turns suppression from a loss into a feature.
  6. Persist: back the breaker with a small JSON file next to `report_dir`, storing `chrono::DateTime<Utc>` instead of `Instant` so a restart doesn't reopen the floodgates. Load on boot in `state.rs`, write on `record`.
- **Trade-offs**: A suppression window can hide a *genuinely new* error that happens to fingerprint alike — mitigate by keying on the normalized message (not just project) and keeping the window well under an hour. Step 6 adds a file dependency; if it's unwanted, ship 1-5 and accept the restart reset, which is still strictly better than no gate at all.

## 3. `ProjectEntry.branch` is configured but never checked out — the diagnosis may describe code that isn't running

- **Severity**: Medium
- **Category**: half-implemented
- **File**: `crates/responder/src/investigate.rs:23-25`, `crates/responder/src/act.rs:53-71`, `crates/responder/src/git.rs:28-47`
- **Scenario**: An operator maps a project with `"branch": "main"`, reasonably reading that as "diagnose against what's deployed". A developer on that box left the clone on a WIP feature branch. A spike fires. `investigate` runs `claude -p` in `entry.repo` — on the WIP branch — and Claude confidently reports a root cause at `handler.rs:88` citing code that has never been deployed. The report looks authoritative and is wrong, and nothing in the output reveals which commit it read. Worse, with `auto_fix: true`, `act.rs:70` cuts `lt-fix/*` from that same arbitrary HEAD, so the fix branch is based off the WIP work.
- **Root cause**: `branch` is parsed (`config.rs:41-42`) and *printed* (`pipeline.rs:45`) — and that is its entire lifecycle. Grep the crate: no other reader. `investigate` passes only `&entry.repo` to `claude::run`, which does `cmd.current_dir(repo)` (`claude.rs:47`) against whatever HEAD happens to be. `act.rs` reads `current_branch` into `orig` purely so it can restore it (line 95); it never compares it to `entry.branch`. `git.rs` has `checkout`, so the mechanism already exists — it is simply never pointed at the configured branch. The report also records `model`, `cost`, and `ts` (`report.rs:20-32`) but no commit SHA, so a reader cannot detect the mismatch after the fact.
- **Impact**: Diagnosis provenance. An observability product's core promise is that what it tells you about production *is about production*; a root cause traced against unknown code quietly breaks that. Even the minimal version of this fix (record and display the SHA) converts a silent wrong answer into a verifiable one, and the full version makes `branch` mean what operators already assume it means.
- **Fix sketch**:
  1. `git.rs`: add `head_sha(repo) -> Option<String>` (`git -C repo rev-parse --short HEAD`).
  2. **Honesty first (cheap, ship alone)**: capture the SHA + actual branch before investigating, thread through `ClaudeRun` (or a small `RepoState` alongside it), and render `- repo: {branch} @ {sha}` in `report.rs::render` and the `<table class="meta">` in `render_html`. Every existing report becomes auditable.
  3. **Enforcement (opt-in)**: when `entry.branch` is `Some(b)` and `current_branch != b`, require `is_clean` (`git.rs:17`), `checkout(repo, b)`, investigate, then restore `orig` in the same always-restore discipline `act.rs:95` already models. If the tree is dirty, do **not** check out — investigate where you are and prefix the report with a warning: `⚠ configured branch '{b}' not checked out (tree dirty); diagnosed against '{actual}' @ {sha}`. Never silently disagree with the config.
  4. `act.rs`: cut the `lt-fix/*` branch from `entry.branch` when configured (`git checkout -b {branch} {base}`) rather than implicit HEAD, so the fix is based off the code it was diagnosed against.
  5. Consider `git fetch` before checkout when the branch should track a remote — flag it, and leave it off by default; a responder that mutates a developer's clone is a surprise.
- **Trade-offs**: Steps 3-5 mean the responder manipulates a checkout a human may be sitting in — the `is_clean` guard and always-restore keep that bounded, but it is real, so gate on `branch` being explicitly configured (it is `Option`, and today most entries won't set it). Step 2 is pure gain with no behavior change and is the piece to ship regardless of whether 3-5 are wanted.
