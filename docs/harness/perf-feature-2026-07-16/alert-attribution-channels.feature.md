## 1. Attribution names the biggest spender, never the thing that *changed*

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/api/src/alerts/attribution.rs:69-82` (`fetch`), `:88-122` (`compose`), `:159-171` (`rank`)
- **Scenario**: A team caps project `checkout` at $50/day. `gpt-4o` is their workhorse and is always ~60% of spend. On Tuesday someone ships a prompt change that triples `classify` calls on `gpt-4o-mini`, and the cap trips. The alert says: *"Top spenders (in this window): gpt-4o (summarize) 58% ($29.10), gpt-4o-mini (classify) 31% ($15.55), claude-sonnet 11%."* It said the exact same thing last week when the cap tripped for an unrelated reason, and the week before. The operator learns nothing and opens the dashboard anyway — which is what the alert existed to prevent.
- **Root cause**: `compose` only ever computes *share of the current window* (`rank`, `:159-171`, divides each contributor's cost by the window total). There is no second window and no comparison, so a contributor's rank is a function of its steady-state size, not of its change. `fetch` (`:77-79`) calls `store.cost_summary_windowed(Some(project), Some(since), None)` — the store's third parameter is `until: Option<DateTime<Utc>>` (`crates/store/src/sqlite/mod.rs:154-161`, SQL bound applied at `crates/store/src/sqlite/events.rs:471+`) and attribution passes `None`, using only the open-ended form. The bounded-window capability the diagnosis needs is already built and wired to the trait; attribution simply never asks for a prior window. `LimitWindow::lookback()` (`crates/core/src/limits.rs:105-116`) makes the previous comparable window a one-line derivation (`since - lookback .. since`).
- **Impact**: This is the difference between the alert this context is *for* ("spend spiked because X regressed") and a restatement of the cost dashboard's top-3 tile. Delta attribution is what makes an alert self-sufficient at 3am and is the specific thing Helicone/Langfuse cost alerts don't do well — a defensible wedge, not parity work.
- **Fix sketch**:
  1. In `fetch`, derive `prev_since = window.since(since)` and fetch a second rollup with the *bounded* call already supported: `cost_summary_windowed(Some(project), Some(prev_since), Some(since))`. One extra query, still inside the spawned delivery task, still zero cost on ingest.
  2. Extend `Contributor` with `prev_cost_usd: f64` and `delta_pct: Option<f64>` (`None` when the contributor is new — that's the "first seen this window" case and should read as such).
  3. Give `compose` a `prev_cost_rows: &[CostRow]` parameter and rank a *second* list by absolute cost growth (`cost - prev_cost`), keeping the existing share ranking. Report both: share answers "who is big", delta answers "what changed".
  4. `message_tail` leads with the delta line when any contributor grew materially: `" Biggest change: gpt-4o-mini (classify) +$12.40 (4.0x vs. previous 24h). Top spenders: …"`. Fall back to today's wording when there's no prior window (new project) — the existing empty-degradation contract at `:40-46` already models this.
  5. `to_json` (`:57-64`) gains `prev_cost_usd` / `delta_pct` per contributor so the responder and custom receivers can act on growth, not just size.
  6. Extend the fixture tests (`:216-285`) with a prior-window vector: the steady-big model must NOT be named as the change when a small model quadruples.
- **Trade-offs**: One extra store read per breach delivery, off the ingest path and already best-effort — cheap. Genuine seam: `usecase_costs` (`crates/store/src/lib.rs:352-358`) has **no `until` parameter`, so use-case-level deltas need a store signature change; model-level deltas (`CostRow`) work today with zero store changes. Ship model-level delta first, add the `until` param to `usecase_costs` in a follow-up. Note the default trait impl of `cost_summary_windowed` (`lib.rs:339-346`) *ignores* both bounds and delegates to `cost_summary(project)` — harmless here because attribution only runs on SQLite (`alerts.rs:57`, `attribution_db_from_env`), but a delta built on that default would silently compare a window against itself and report zero change. Guard by keeping attribution SQLite-gated as it is today.

## 2. Soft warnings — the only alert an operator can still act on — carry no attribution

- **Severity**: High
- **Category**: half-implemented
- **File**: `crates/api/src/alerts/channels.rs:75-83` (`deliver_warnings`), `:148-155` (`warning_message`)
- **Scenario**: A `Block`-action cap warns at 80%. The operator gets: *"project 'checkout' is approaching its CostUsd/Day limit — current 40.12 is 80% of threshold 50.00. No traffic has been blocked."* They have maybe an hour of headroom and no idea which knob to turn, so they either ignore it or go dig through the dashboard. An hour later the cap trips, ingest starts 429ing, and *that* alert — the one arriving after the damage — is the one that finally names the spenders.
- **Root cause**: `deliver_warnings` takes `(cfg, http, warnings)` only — no `attributions` parameter — and `warning_message(w)` (`:148-155`) formats purely from `LimitStatus` fields. Compare `deliver_breaches` (`:44-70`), which threads `attributions: &HashMap<String, Attribution>` and appends `a.message_tail()` via `breach_message` (`:168`). The attribution machinery is entirely `LimitStatus`-keyed and scope-aware (`Alerter::attribute`, `alerts.rs:205-232`, loops over `&[LimitStatus]` and keys on `alert_key()`) — it works on a warning row exactly as well as a breach row, because a warning *is* a `LimitStatus`. Nothing about the design excludes warnings; the parameter was just never threaded. The value is inverted from where it's spent: attribution is attached to the alert that arrives too late to act on, and withheld from the one that arrives in time.
- **Impact**: Turns the warning tier from a nag into the product's most valuable alert — the one where naming the driver still prevents the outage. Pre-emptive, actionable budget warnings are exactly what a team evaluating LightTrack against Langfuse would demo.
- **Fix sketch**:
  1. Widen the signature to `deliver_warnings(cfg, http, warnings, attributions: &HashMap<String, Attribution>)`, mirroring `deliver_breaches`.
  2. Widen `warning_message(w, attribution: Option<&Attribution>)` and append `attribution.and_then(|a| a.message_tail()).unwrap_or_default()` — the same one-line pattern as `breach_message:168`. `message_tail`'s "in this window" default scope wording (`:43`) already reads correctly for a warning.
  3. Include `"attribution": contributors` in the `limit_warning` webhook `extra` (`:78`), matching the breach payload at `:64`, so receivers get the same structured shape on both tiers.
  4. Caller side (`alerts.rs:259-273`) awaits the existing `me.attribute(&due)` before `deliver_warnings` — same spawned-task shape as `notify` (`:194-198`). Note: this touches `alerts.rs`, which a parallel audit owns; the two files above are where the gap actually lives and where the fix is load-bearing.
- **Trade-offs**: Warnings fire more often than breaches, so this adds a store read per warning delivery — bounded by the 3600s per-key cooldown (`alerts.rs:121`) and off the ingest path. Worth it: it's the tier where attribution pays.

## 3. Every alert lands in the same place — an enforcing breach can't page anyone

- **Severity**: Medium
- **Category**: integration
- **File**: `crates/api/src/alerts/channels.rs:44-70` (`deliver_breaches`), `:178-191` (`post_webhook`), `:193-206` (`post_ntfy`)
- **Scenario**: At 03:00 a `Block` cap trips on the production project. Ingest starts rejecting calls with 429s — the customer's LLM traffic is being dropped. LightTrack posts a Slack message to `#llm-costs`, the same channel that got a `bench_run` completion and a `limit_warning` twenty minutes earlier. Nobody is awake. The team finds out from their own error dashboard at 08:30, and the thing that would have woken them — PagerDuty — was never an option to configure.
- **Root cause**: `post_webhook` (`:178-191`) resolves exactly one URL (`cfg.webhook`) and posts a body shaped for Slack-or-Discord (`"text"` + `"content"`, `:180`) with the event name as a *field* (`"event": event`) rather than as a routing decision. Every `deliver_*` composer calls the same three sinks unconditionally (`:66-68`, `:78-81`, `:87-89`, `:109-111`, `:127-129`, `:143-145`) — there is no severity dimension in the transport at all. Meanwhile `LimitStatus.action` (`crates/core/src/limits.rs:119-128`) already distinguishes `Alert` (notify only) from `Throttle`/`Block` (**enforced at ingest admission — a breaching event is rejected with HTTP 429 and not recorded**), and `deliver_breaches` already knows this: it formats `action={:?}` into the message (`:171-172`) and reports `rejected_count` (`:54`, `:64`) — the count of *calls actually turned away*. The signal that separates a page from an FYI is present, computed, and serialized into the payload — and then thrown at whichever single webhook happens to be configured. `post_ntfy` hardcodes `Priority: high` and `Tags: warning` (`:198-199`) for *every* alert type including bench-run-adjacent traffic, so even the one channel with a priority concept is pinned to a constant.
- **Impact**: Cost caps that enforce are an availability control; a team can't adopt `Block` in production if the breach can't page. Adding PagerDuty (plus per-severity routing) is the difference between "cost dashboard with notifications" and "thing on our on-call runbook" — and it's what LangSmith/Helicone buyers ask about in the second meeting.
- **Fix sketch**:
  1. Introduce `enum Severity { Info, Warning, Critical }` in `channels.rs` and have each `deliver_*` declare its own: `deliver_bench_run` → `Info`; `deliver_warnings` → `Warning`; `deliver_breaches` → `Critical` when `b.action != LimitAction::Alert` (traffic is being rejected), else `Warning`; `deliver_error_spike` / `deliver_score_drop` / `deliver_relay_dead` → `Warning`.
  2. Thread `sev` through `post_webhook` / `post_ntfy` and map it onto ntfy's existing header knobs (`:198-199`): `Priority: urgent|high|default`, `Tags: rotating_light|warning|information_source`. Zero new config, immediately better triage on the one channel that already has the concept.
  3. Add `post_pagerduty(cfg, http, sev, dedup_key, msg, extra)` posting to Events API v2 (`https://events.pagerduty.com/v2/enqueue`) with `routing_key` from a new `LIGHTTRACK_ALERT_PAGERDUTY_KEY`. Fire it only for `Severity::Critical`. Use `LimitStatus::alert_key()` (`limits.rs:216-218`) as PD's `dedup_key` — it's already the stable per-rule identity the cooldown uses, so PD auto-resolves and re-fires correctly on the same rule. Mirror the existing best-effort error handling (`:186-190`): a down PD logs to stderr, never panics.
  4. Optional follow-up: a `LIGHTTRACK_ALERT_WEBHOOK_CRITICAL` that, when set, receives `Critical` events instead of the default webhook — one env var buys per-severity Slack channel routing without the per-project schema work `alerts.rs:10-11` documents as out of scope.
  5. Optional follow-up: Slack Blocks for breach alerts, so the attribution contributor list from finding #1 renders as fields rather than a run-on sentence. Gate on a `LIGHTTRACK_ALERT_WEBHOOK_KIND=slack` hint — the current body deliberately serves Slack *and* Discord *and* custom receivers off one shape, and that flexibility shouldn't be broken by default.
- **Trade-offs**: PagerDuty adds a third-party contract to a module whose current sinks are all "POST some JSON at a URL" — but the transport is already per-sink free functions with no shared state (module doc, `:1-4`), so `post_pagerduty` fits the established shape exactly. Severity mapping is a judgment call and someone will want `Alert`-action breaches to page too; the `LIGHTTRACK_ALERT_WEBHOOK_CRITICAL` escape hatch plus a future `LIGHTTRACK_ALERT_PAGE_ON` threshold covers that without hardcoding a policy.
