## 1. Pre-flight admission check — let a cap stop spend instead of rejecting the receipt

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/api/src/limits.rs:22-56`, `crates/api/src/limits.rs:211-228`
- **Scenario**: An operator sets "cap project `acme` at $50/day, action=block" after a prompt-loop bug burned $4k overnight. The next morning a retry storm hits. Every LightTrack ingest past $50 returns 429 — but each of those 429s happened *after* the OpenAI call completed and the money was already spent. The cap turned away the telemetry, not the spend. The operator ends the day at $900 with a `/v1/limits/status` page that cheerfully reports `throttled: true` and a rejected-traffic ledger estimating the money it failed to prevent.
- **Root cause**: Admission is evaluated inside `insert_event_checked` against a *completed* `LlmEvent` (`crates/api/src/events.rs:163`), i.e. the enforcement point is post-hoc by construction. `crates/core/src/limits.rs:119-121` concedes this: "Inline *pre-call* blocking still requires the future gateway/proxy mode." Meanwhile the machinery to answer the pre-call question already exists and is **unreachable from any client-usable entry point**: `evaluate_admission` (`crates/store/src/lib.rs:230`) folds a *candidate* contribution into scoped rolling usage and returns a verdict, but it is `pub(crate)` and only ever called with an event that is about to be inserted. The only exported surface is `limits_status` (`limits.rs:211`), which (a) folds in no candidate — `evaluate_project_limits:32-51` explicitly notes "no candidate event, so nothing is folded in", and (b) cannot filter to the scoped rules that would actually apply to *this* provider/model/use-case, so an SDK polling it gets a project-wide yes/no that is wrong for any scoped cap. There is no `check`-style route (`crates/api/src/main.rs:296-301` registers only CRUD + `/status`).
- **Impact**: This is the difference between "we noticed" and "we stopped it" — the exact thing the buyer is paying for. Today a `block` rule is a spend *detector* with a 429 costume. A pre-flight endpoint turns every SDK into an enforcement point without needing the gateway: the client asks "may I spend ~2k tokens on gpt-4o?", gets a verdict in one round trip, and skips the provider call. It also makes `Block` finally honest and gives the SDK something to back off *on* rather than back off *after*.
- **Fix sketch**:
  1. Promote `evaluate_admission` + `event_contribution` to `pub` in `lighttrack_store` (they are already pure over an injected `current_usage` closure — no new query shapes needed).
  2. Add `Store::check_admission(project, candidate: &LlmEvent) -> Result<Admission>` — the body is `insert_event_checked` minus the `insert_event` call. Backends inherit it; no per-backend work.
  3. Add `POST /v1/limits/check` in `crates/api/src/limits.rs`, ingest-scoped via `resolve_ingest_project` (same key class as `post_event`, not the admin gate). Body = a partial event: `{provider, model, name?, estimated_tokens?, estimated_cost_usd?}`. Price the estimate through the existing `st.prices` book exactly as `prepare_event` does, so a caller that only knows token counts still gets a cost verdict.
  4. Response: `{allowed, statuses[], reason?}` reusing `breach_reason` for the human string. **Do not** record a rejection or fire an alert here — a check is not traffic; the ledger in `rejections.rs` must keep counting only real turned-away ingest.
  5. Optional follow-on: an SDK helper that calls check → provider → ingest, so the cap becomes opt-in enforcement in one line.
- **Trade-offs**: A check is advisory — a client can ignore the verdict and call anyway, so this hardens cooperating callers, not hostile ones (the gateway mode remains the only true chokepoint). Two round trips per call unless the SDK caches the verdict for a short TTL; a TTL means a burst can slightly overshoot the cap, which is strictly better than today's total overshoot. Estimated cost is an estimate: under-estimating lets a call through that ingest will then 429 — acceptable, and the two paths share one evaluator so they never disagree on semantics.

## 2. Firestore backend silently discards `scope` and `warn_at` — a scoped cap widens to the whole project

- **Severity**: High
- **Category**: half-implemented
- **File**: `crates/store-firestore/src/limits.rs:11-21`, `crates/store-firestore/src/limits.rs:32-46`
- **Scenario**: An operator on the Firestore backend runs the documented "cap gpt-4o at $5/day" flow: `POST /v1/projects/acme/limits {"metric":"cost_usd","window":"day","threshold":5,"action":"block","scope":{"model":"gpt-4o"}}`. The API returns **201 with the scope echoed back** (`crates/api/src/limits.rs:95-110` builds the rule, validates it, and `Ok(Json(rule))` returns the in-memory struct — not what was stored). Every subsequent gpt-4o-mini, claude, and embedding call in the project starts 429-ing the moment total project spend crosses $5, because the rule came back from storage unscoped. The operator's narrow experiment became a project-wide kill switch, and the only evidence is `lighttrack limits list` showing Scope `—` on a rule they created *with* a scope.
- **Root cause**: `create_limit_rule` (`limits.rs:11-21`) writes exactly six fields — `id, project_id, metric, window, threshold, action, enabled`. `r.scope` and `r.warn_at` are never inserted. The read path is explicit about it (`limits.rs:41-44`): "warn_at / scope: not yet persisted by this backend (handoff) — defaulted so the shared LimitRule constructs", hardcoding `warn_at: None, scope: None`. Because `evaluate_admission` treats `scope: None` as *applies to everything* (`scope_matches` returns `true` for `None`, `crates/core/src/limits.rs:70`), the dropped field doesn't disable the rule — it **widens** it, and the widened rule is `action: block`. The `warn_at` loss is quieter but the same shape: the operator's early-warning tier is silently deleted, so the first signal is the 429 instead of the 80% heads-up. Two related gaps compound it: `Store::usage_since_scoped` defaults to `usage_since` (`crates/store/src/lib.rs:370-377`), so even a backend that *did* load a scope would count project-wide usage against it; and Firestore implements neither `get_limit_rule` (trait default → `Ok(None)`, `lib.rs:421`) nor `update_limit_rule`/`delete_limit_rule` (defaults → `Err`, `lib.rs:429-440`) — so `PUT` 404s at `crates/api/src/limits.rs:141-143` and `DELETE` 500s. The over-broad cap **cannot be edited or deleted through the product at all**; lifting it requires hand-editing Firestore.
- **Impact**: The two headline limit features of this context — scoped caps and soft warnings — are structurally dead on a shipped backend, and they fail *open into a worse state* rather than closed. Scoped caps are the natural upsell ("cap the expensive model, not the app"); shipping a backend where that silently becomes a project-wide block is a churn event, not a bug report. Closing this makes the Firestore backend actually at parity and makes finding #3's action semantics meaningful there.
- **Fix sketch**:
  1. Persist the two fields in `create_limit_rule`: `warn_at` as a nullable number, and the scope flattened to `scope_kind` + `scope_value` columns — `LimitScope::kind_str()` / `value()` exist precisely for this (`crates/core/src/limits.rs:22-35`), and `from_parts` (`limits.rs:38`) is the ready-made read side.
  2. In `limit_from`, rebuild via `LimitScope::from_parts(kind, value)`; absent kind → `None` (correct for genuinely project-wide rules, and old docs written before this change deserialize to today's behavior unchanged).
  3. Add `get_limit_rule` (doc fetch by id) and `update_limit_rule`/`delete_limit_rule` (put/delete returning `false` on unknown id) so a bad cap can be lifted through the API.
  4. Implement `usage_since_scoped` with a `scope_kind`-appropriate `EQUAL` filter on the events query, so a scoped rule reads its own dimension's total. **Order matters**: land this *before* step 1, or a newly-honored scope will be evaluated against project-wide usage — a different wrong answer.
  5. Extend the store conformance suite (`crates/store/src/conformance.rs:136-148` currently constructs rules with `warn_at: None, scope: None`, which is why this never tripped) with a scoped + `warn_at` round-trip case, so any backend that drops the fields fails CI instead of shipping.
- **Trade-offs**: Rules written before the change have no scope/warn_at fields; `from_parts`-on-absent → `None` handles them without a migration, and that is also their true meaning. Step 4 needs a composite Firestore index per scope dimension — cheap, but it is a console/deploy action, so a backend that hasn't got the index yet should surface the query error rather than fall back to `usage_since` (falling back is exactly the silent-widening bug in a new coat).

## 3. `Throttle` and `Block` are the same behavior — there is no rate-shaping tier

- **Severity**: Medium
- **Category**: power-feature
- **File**: `crates/core/src/limits.rs:119-143`
- **Scenario**: An operator wants a cap that bends rather than breaks: past $50/day, keep the critical `checkout-summarize` path flowing but shed the bulk backfill. `LimitAction` offers `Throttle` — which reads like exactly that — so they pick it. It hard-rejects every event past the threshold, identically to `Block`. They discover the two words are synonyms only by reading `enforces()`, and the product's own test suite documents the equivalence as intended (`only_throttle_and_block_enforce`, `limits.rs:376-380`).
- **Root cause**: `LimitAction::enforces()` (`limits.rs:140-142`) is `matches!(self, Throttle | Block)` and is the **only** consumer of the variant distinction — nothing downstream branches on `Throttle` vs `Block`. `LimitStatus::rejects_ingest()` (`limits.rs:201-203`) collapses them again, and `evaluate_admission` derives admission purely from that. So a three-valued enum encodes two behaviors: observe (`Alert`) and reject (`Throttle` == `Block`). The name is presented to operators verbatim — `crates/render/src/limits.rs:138` prints the action in the rules table, and `breach_reason` (`crates/api/src/events.rs:132`) echoes `action={:?}` into the 429 body — so the label promises shaping the engine never implemented. Worth noting the status table (`render/src/limits.rs:23-31`) has no Action column at all: a `❌ over` row there could be an observe-only `Alert` or a hard 429, and the operator cannot tell which from the surface they check during an incident.
- **Impact**: Hard-blocking is the wrong default for the buyer's real situation. At the moment spend runs away, an all-or-nothing cap forces a choice between "burn money" and "break production" — so in practice operators leave everything on `Alert`, and the enforcement feature goes unused. A real throttle tier (shed low-priority traffic, keep the rest) is the setting people will actually turn on, which is what makes the whole enforcement path commercially live. It composes directly with the scoped caps from #2: shape by `name`, keep the money path.
- **Fix sketch**:
  1. Give `Throttle` a distinct meaning by carrying a shape: `Throttle { allow_fraction: f64 }` (serde-defaulted to `0.0` so existing `"throttle"` rules deserialize to today's exact behavior — a hard reject — and nothing silently loosens).
  2. Replace `enforces()`'s boolean with a `Decision { Admit, Reject }` returned from a `decide(&self, ev) -> Decision`, so `Block` → always `Reject`, `Alert` → always `Admit`, `Throttle` → `Reject` for a deterministic `allow_fraction` slice. Key the slice on a stable hash of `ev.trace_id`/`ev.id`, **not** a RNG, so the same trace isn't half-admitted and the decision is reproducible in tests.
  3. Keep `rejects_ingest()` as the single collapse point the ingest path calls, now delegating to `decide` — no change at `crates/api/src/events.rs:111` / `:171`.
  4. Add an Action column to the status table (`crates/render/src/limits.rs:23-31`) and split the badge: `❌ over (blocking)` vs `❌ over (alert only)`. This is worth doing on its own even if the shaping tier is deferred — it is the surface an operator reads mid-incident, and today it cannot answer "is this cap actually stopping anything?".
  5. Deferrable follow-on: shape by priority rather than fraction (`allow_names: [..]`), once the `name` dimension proves out via scoped caps.
- **Trade-offs**: Fractional shedding makes the ledger fuzzier — the rejection ledger (`rejections.rs`) counts turned-away calls, and a partial throttle means "rejected" no longer implies "at the cap", so the `est_missed_cost_usd` column needs re-reading as a sample not a total. Step 4 alone is near-zero-risk and independently valuable; steps 1-3 change a public enum's serde shape, so pin the default-to-current-behavior test before touching it. If #1 lands, shaping is far more valuable applied at the pre-flight check (shed *before* spending) than at ingest — sequence accordingly.
