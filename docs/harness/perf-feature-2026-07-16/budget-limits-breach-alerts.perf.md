## 1. `evaluate_project_limits` full-scans the rolling window under the global store lock, stalling ingest

- **Severity**: High
- **Category**: recompute
- **File**: `crates/api/src/limits.rs:23-56`
- **Scenario**: A dashboard (or the CLI `limits status` view) polls `GET /v1/limits/status` every few seconds for a project with a `month` cost cap. The project has ~500k events in the 30-day window. Meanwhile ingest is running at hundreds of events/sec through `insert_event_checked`.
- **Root cause**: The status path calls `store.usage_since` / `usage_since_scoped` (`limits.rs:43-44`), which are `SELECT SUM(cost_usd), COUNT(*), SUM(tokens) FROM events WHERE project_id=?1 AND ts>=?2` ÔÇö a full aggregate over every in-window row (`sqlite/events.rs:519`). The incremental `UsageCache` (`sqlite/usage_cache.rs`) that exists precisely to make this `O(events since last check)` is reachable only from `SqliteStore::insert_event_checked` (`sqlite/mod.rs:127-132`); it is not exposed on the `Store` trait, so this caller has no way to reach it and recomputes from scratch. Worse, `SqliteStore::with` serializes on the store's single global `Mutex<Connection>` ÔÇö so each poll's multi-hundred-thousand-row scan **holds the same lock the ingest admission path needs**. The per-`(window, scope)` memoization at `limits.rs:35-49` dedups *within* one request only; nothing is cached across requests.
  Two aggravators visible from this call site: a `model`- or `provider`-scoped rule takes the `usage_since_scoped` branch, which has no covering index for those columns (only `(project_id, ts)` and `(project_id, name, ts)` exist), so it scans every in-window row and filters; and a project with 3 rules across 3 distinct `(window, scope)` keys pays 3 independent scans per poll.
- **Impact**: A 30-day window at 500k events is roughly 10ÔüÁÔÇô10ÔüÂ rows aggregated per distinct key per poll ÔÇö order tens to low hundreds of ms of lock-held CPU. At a 5s dashboard poll with 3 rules, that is a recurring multi-hundred-ms window in which *every* ingest request queues behind the mutex; ingest p99 becomes a function of dashboard polling and window size. Cost grows linearly with retained history, so it degrades silently as the DB fills. Metrics worth adding: histogram of store-lock hold time by call site, and `usage_since` rows-scanned.
- **Fix sketch**: Expose the incremental rollup on the `Store` trait ÔÇö e.g. `fn rolling_usage(&self, project, window, scope, now) -> Result<Usage>`, implemented on `SqliteStore` by locking `usage_cache` + `conn` and calling `UsageCache::usage` (the same call `events::insert_checked` makes), and defaulted to the existing `usage_since`/`usage_since_scoped` full scan for Postgres/Firestore. Then swap `limits.rs:42-48` to call it. The cache's own docs assert exactness vs. the full scan (property-tested in `sqlite/tests.rs`), so the status numbers stay identical while the poll drops to "fold in events since last read". Independently, add `idx_events_project_model_ts` / `_provider_ts` so scoped rules stop scanning.
- **Trade-offs**: The `UsageCache` assumes a non-decreasing `now`; the status path passes `Utc::now()`, same as ingest, so that holds. Reads through the cache now take the cache mutex as well as the connection mutex ÔÇö the lock order must match `insert_event_checked` (cache before connection) to stay deadlock-free. Cache memory grows with distinct `(project, window, scope)` keys, but status reads use the same keys ingest already populates, so this adds no new buckets.

## 2. Firestore backend re-queries `limit_rules` over the network on every single ingest event

- **Severity**: High
- **Category**: missing-cache
- **File**: `crates/store-firestore/src/limits.rs:23-30`
- **Scenario**: LightTrack deployed on the Firestore backend, ingesting a customer's monitored LLM traffic at, say, 20 events/sec. Every event goes through `Store::insert_event_checked`, whose default impl (`store/src/lib.rs:281-282`) opens with `self.list_limit_rules(&ev.project_id, true)?` ÔÇö Firestore does not override it.
- **Root cause**: `list_limit_rules` issues a fresh `rest.query("limit_rules", ÔÇĞ)` per call with no caching layer of any kind. Limit rules are configuration: they change on an operator's `POST/PUT/DELETE /v1/limits`, i.e. essentially never relative to ingest volume. Yet the ingest path pays a full synchronous Firestore REST round trip to re-read them for every event, inside the blocking `spawn_db` step the HTTP response waits on. The `only_enabled` variant also adds an `enabled EQUAL 1` filter alongside `project_id EQUAL`, which requires a composite index ÔÇö an unindexed deploy makes the per-event query fail or fall back badly.
- **Impact**: Per event, one RTT to Firestore (~20-80ms typical from Cloud Run, more cross-region) added directly to ingest latency, plus one billed doc read per rule. At 20 events/sec Ôëê 52M events/month ├ù 3 rules Ôëê 155M doc reads (~$56/mo at $0.036/100k) purely to re-read three rows that never change ÔÇö and the latency is the bigger tax: it can easily dominate the whole ingest handler. It also multiplies through `insert_events_checked`, whose Firestore default loops `insert_event_checked` per item (`store/src/lib.rs:307-308`), so a 100-event batch does 100 rule queries serially. Metric: count `limit_rules` queries/sec vs. events/sec ÔÇö the ratio should be Ôëê0, today it is ÔëÑ1.
- **Fix sketch**: Add a small TTL cache in `store-firestore/src/limits.rs` ÔÇö `Mutex<HashMap<(String, bool), (Instant, Vec<LimitRule>)>>` on the store, with a short TTL (5-30s, env-tunable) served by `list_limit_rules`, and an explicit invalidation of the project's entries from `create_limit_rule` / `update` / `delete`. Since all mutations flow through this process's own API handlers, invalidation makes the TTL a backstop rather than the primary freshness mechanism. This alone removes the per-event RTT. Also confirm the `(project_id, enabled)` composite index exists in the deployed Firestore config.
- **Trade-offs**: A rule edited directly in the Firestore console (out of band) takes up to one TTL to take effect ÔÇö acceptable for config, and bounded/tunable. Cache is per-process; with multiple API processes each converges within a TTL. Note this file also does not persist `warn_at`/`scope` (`limits.rs:41-44`), so scoped-rule fan-out isn't a factor on this backend yet ÔÇö the caching shape above stays correct when they are added.

## 3. Alerter dedup/window maps grow without bound; the relay-dead key is unique per task

- **Severity**: Medium
- **Category**: unbounded-growth
- **File**: `crates/api/src/alerts.rs:103-105, 296, 439-449`
- **Scenario**: A long-lived API process (weeks of uptime) serving many projects, with relay tasks dead-lettering occasionally and judge scores flowing across many `(project, rubric)` pairs.
- **Root cause**: `last_sent: Mutex<HashMap<String, Instant>>` is insert-only ÔÇö `should_send_key` (`alerts.rs:439-449`) inserts on every send and **nothing ever removes an entry**, not even once an entry is far past its cooldown and can no longer suppress anything. Most key families are bounded by config cardinality (`project:metric:window:scope`, `warn:ÔÇĞ`, `error-spike:{project}`, `bench-run:{benchmark}:{status}`), but `relay-dead:{t.id}` (`alerts.rs:296`) is keyed by a **unique task id**, so every dead-lettered task permanently adds an entry that can never dedup anything again ÔÇö strictly unbounded in a process-lifetime map. `error_windows` and `score_windows` have the same shape at the map level: `note_error` (`alerts.rs:348-360`) and `note_score` (`alerts.rs:392`) bound each *deque*'s length, but the per-project / per-`(project, rubric)` **entries themselves are never removed**, so an idle project's empty `VecDeque` lives forever.
- **Impact**: Bounded in rate, not in total ÔÇö each dead-letter costs ~100 bytes of permanently-retained heap. A deployment dead-lettering 1k tasks/day retains ~36MB/year of pure garbage, and the map is walked-by-hash under a mutex taken on the ingest path (`notify` ÔåÆ `should_send`), so the growth also gradually degrades cache locality of a lock every event touches. `score_windows` similarly retains ~20 f64s per rubric ever seen. Not fatal at typical scale, which is why this is Medium ÔÇö but it is a leak with no ceiling in a process expected to run for months.
- **Fix sketch**: Give `should_send_key` an amortized sweep: track an `Instant` of the last GC and, when it is older than (say) the cooldown, retain only entries whose `now.duration_since(t) < cooldown` (`map.retain(|_, t| now.duration_since(*t) < cooldown)`) ÔÇö an entry past its cooldown is behaviorally identical to an absent one, so this is semantics-preserving and fixes the `relay-dead` case exactly. Do the same in `note_error` / `note_score`: after evicting, `if dq.is_empty() { map.remove(key) }` (and for scores, drop entries not touched within some idle horizon). Guard the sweep so a `cooldown` of 0 (`zero_cooldown_always_sends` test) still behaves ÔÇö with a 0 cooldown every entry is immediately collectable, which is correct.
- **Trade-offs**: A periodic `retain` is O(n) under the mutex; amortized once per cooldown interval (default 3600s) it is negligible against the per-event lock acquisitions it shares with. No behavior change: only entries that can no longer suppress an alert are dropped.
