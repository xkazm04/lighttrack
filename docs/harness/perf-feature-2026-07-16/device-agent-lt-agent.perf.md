# Performance Optimizer — Device Agent (lt-agent)

> Total: 3

## 1. Command connector deadlocks the whole agent on a >64KB envelope
- **Severity**: Critical
- **Category**: subprocess
- **File**: `crates/agent/src/connect.rs:50-76`
- **Scenario**: An action with a `command` connector returns a schema-shaped result of any real size (a reprice table, a summary with per-item rows). `exec::deliver` builds the envelope with the full `result` **and** the full task `payload` embedded, so envelopes routinely exceed the OS pipe buffer (64KB on Windows, 64KB on Linux). The connector script is a normal "read stdin at the end / print progress as it goes" script.
- **Root cause**: `write_all` on the child's stdin runs to completion *before* `wait_with_output()` starts draining stdout/stderr. `stderr` is `Stdio::piped()`. Two independent ways to wedge: (a) the child writes >64KB to stderr while the parent is still blocked in `write_all` — the child blocks on stderr, the parent blocks on stdin, neither ever moves; (b) the child does not consume stdin until it has finished work, so the parent's `write_all` blocks on a full stdin pipe. Neither `write_all` nor `wait_with_output` has any timeout. This is on the serial path — `run::run` calls `exec::execute` inline (`run.rs:35`), so a wedged connector hangs the *entire* agent, not just one task.
- **Impact**: Permanent hang. The agent stops leasing from every source forever, so the whole device goes dark until a human notices. Worse, the cloud reclaims the task at lease expiry (default `lease_secs = 1800`, `config.rs:78-80`) and hands it to the next device/attempt, which re-runs the *full Claude Code call* — a paid, multi-minute LLM run repeated for a task the device already completed successfully. One malformed script converts into indefinite duplicated model spend plus a dead agent.
- **Fix sketch**: Drain concurrently and bound the wait. Take `child.stdin` and write it on a spawned thread (`std::thread::spawn(move || stdin.write_all(&bytes))`), while the main thread does `child.wait_with_output()`; join the writer afterwards and treat a `BrokenPipe` from the writer as non-fatal when the child exited 0. Add a wall-clock cap around the wait (poll `child.try_wait()` in a loop against a `Duration`, or use `wait-timeout`), and on expiry `child.kill()` + `bail!` so the task settles `failed` instead of stalling the loop. Cap stderr capture at the `truncate()` limit (500 bytes) rather than buffering it whole.
- **Trade-offs**: One extra thread per connector delivery (negligible — deliveries are serial and rare) and a new timeout knob to pick. The timeout turns a silent hang into a visible failed attempt, which is strictly the better failure mode given connectors are required to be idempotent.

## 2. Serial round-robin lease makes every source pay for every other source's latency
- **Severity**: High
- **Category**: polling
- **File**: `crates/agent/src/run.rs:24-61`
- **Scenario**: A device configured with several clouds (the module doc and `config.rs:15-18` both describe multi-source as the normal case). Any of: one source long-polls, or one source is unreachable/slow, or one source has a queue of work.
- **Root cause**: The `for client in &clients` loop leases from each source strictly in sequence on the single thread that also executes Claude runs, so all per-source wall clock is additive:
  - **Long-poll head-of-line blocking**: `wait_secs` is sent to every source (`run.rs:27`). With 4 sources at the server cap of 25s, a task arriving at the last source waits up to ~75s of *other sources' held polls* before it is even seen. The config comment ("keep it small — a held poll on one source delays the round-robin for the others") documents the defect rather than fixing it; keeping `wait_secs` small to dodge it re-introduces poll latency and wastes lease round-trips on every source.
  - **Dead-source stall with no backoff**: a failing lease is logged and the loop moves on (`run.rs:52`), with no per-source backoff or circuit breaker. Each attempt burns up to `CONNECT_TIMEOUT` (10s, `cloud.rs:14`) or `REQUEST_TIMEOUT` (60s). When any *other* source is producing work, `worked` is `true` (`run.rs:32`), so the `poll_secs` sleep is skipped entirely and the dead source is retried on every single round — inserting a 10–60s blocking stall between consecutive task executions of the healthy sources.
- **Impact**: Task pickup latency scales as O(sources) instead of O(1). A single unreachable cloud can more than halve throughput on a busy device: a 30s Claude run followed by a 10s connect timeout is 25% of wall clock spent waiting on a socket that will not open. Under a queue backlog the healthy sources' tasks age toward their own lease/deadline windows purely because of a peer's downtime.
- **Fix sketch**: Split leasing from execution. Spawn one thread per `Client` that leases with long-poll into a shared bounded `std::sync::mpsc::sync_channel(1)` (or a small per-source slot), and keep a single consumer thread executing + settling — preserving the deliberate one-Claude-run-at-a-time property while removing the additive lease latency. Add per-source exponential backoff on lease error (e.g. `poll_secs` doubling to a 5-minute ceiling, reset on success) so a dead cloud costs one probe per backoff window rather than one per round. Round-robin fairness is preserved by the consumer draining sources in rotation.
- **Trade-offs**: N+1 threads instead of 1 and a channel to reason about — modest for a loop already dominated by multi-second subprocess runs. The `once` drain semantics need reworking (all lease threads must report an empty round before exit). Backoff means a recovering source is picked up up to one backoff window late; capping the ceiling bounds that.

## 3. A fresh reqwest client (and TLS stack) is built for every connector delivery
- **Severity**: Medium
- **Category**: missing-cache
- **File**: `crates/agent/src/connect.rs:32-42`
- **Scenario**: Every task with an `http` connector — i.e. the common path — on a device draining a queue.
- **Root cause**: `reqwest::blocking::Client::builder()…build()` is called inside `deliver`, per task. Constructing a blocking client spins up its own tokio runtime and thread, and loading the TLS root store (~150 native/webpki roots) is done from scratch each time. The client is dropped immediately after the single POST, so its connection pool never survives to a second request — every delivery pays a full TCP + TLS handshake to the same callback host, with zero keep-alive reuse. `cloud.rs:30-35` builds its clients once at construction and shows the right pattern; the connector path just doesn't follow it.
- **Impact**: Per delivery, roughly a runtime spin-up plus root-store load plus a full TLS handshake (~2 RTT) that a pooled client would skip — order tens of milliseconds of CPU plus 50–200ms of network on a WAN callback. Against a multi-second Claude run the ratio is small, but it is pure waste on the serial path, it recurs on every task forever, and the same rebuild happens on failed attempts and retries. The cost is also unbounded in the wrong direction: the more the queue drains, the more handshakes.
- **Fix sketch**: Build one `reqwest::blocking::Client` at agent start (next to the `EngineConfig` in `run::run`, or in a `OnceLock<Client>` in `connect.rs`) and thread it into `deliver` as a parameter — the same client serves all connector hosts, since reqwest pools per-origin internally. Keep the existing connect/request timeouts on the shared builder. Header expansion (`expand_headers`) stays per-request, which is correct — `${ENV_VAR}` values must not be cached.
- **Trade-offs**: One extra parameter through `exec::deliver` → `connect::deliver`, and the pooled connections stay warm between tasks (a few idle sockets; bound with `.pool_idle_timeout(…)` if that matters). No behavioural change.
