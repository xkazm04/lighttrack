## 1. Whole-corpus leaderboard merge recomputed on every GET, with filters applied after the merge
- **Severity**: High
- **Category**: missing-cache / hot-path
- **File**: `crates/core/src/collective/merge.rs:207-252`
- **Scenario**: A hub (`LIGHTTRACK_COLLECTIVE_ACCEPT=1`) is the public leaderboard for many instances. `GET /v1/collective/leaderboard` calls `store.list_collective_entries()` then `merge_leaderboard(&entries, floor)` on **every request**, unconditionally. `crates/api/src/collective.rs:278-287` then throws most of the result away via `rows.retain(...)` for `task_type` / `provider` / `judge`. Ingest caps one contributor at `MAX_ENTRIES = 5000`; a hub with 1k contributors × ~50 buckets is ~50k entries, and a dashboard polling the endpoint (or any unauthenticated-ish read fan-out) re-does the whole thing per hit.
- **Root cause**: `merge_leaderboard` is a **pure function of the stored entry set** — its output only changes on ingest — but it takes no filter and exposes no memoizable form, so the API layer has no choice but to full-scan + full-merge + full-sort and filter afterwards. Cost per request is O(N log M) grouping (N entries, M buckets) plus O(M log M) sort plus the allocation load of finding #2. A `?task_type=qa` request that returns 20 rows still pays to merge all 50k entries across all 10 task types.
- **Impact**: Reads scale with corpus size × request rate instead of with the answer size. At 50k entries the merge is on the order of tens of milliseconds of pure CPU on a blocking `spawn_db` thread — per request, per reader, for a result that is byte-identical until the next ingest. Ingests are rare, reads are not: the cache hit rate here should be ~100%, and today it is 0%.
- **Fix sketch**: Two independent moves, both cheap:
  1. Add a filter to the core API — `merge_leaderboard_filtered(entries, floor, &MergeFilter { task_type, provider, judge })` that skips non-matching entries **in the grouping loop** (before `Acc::add`), so filtered reads never allocate or accumulate for buckets they will discard. The judge filter must still be applied post-group (it depends on the merged set), but provider/task_type prune at ingest into the map.
  2. Cache the merged result: since the output depends only on the entry set, hold `RwLock<Option<(u64 /*version*/, Arc<Vec<LeaderboardRow>>)>>` in `AppState`, bumped by `post_ingest`. Readers then filter a cached `Arc<Vec<LeaderboardRow>>` and merge zero times. `merge_leaderboard` stays pure; only the caller gains the cache.
- **Trade-offs**: The cache adds an invalidation edge (every write path touching `collective_entries` must bump the version — today only `post_ingest`, so the surface is one function). Filter-pushdown means the `n_contributors`/`n_models` header counts must be derived from the pruned set (the API already recomputes them over filtered rows, so this actually simplifies `collective.rs:293-307`). Metric to watch: p99 of `get_leaderboard` vs. row count, and a merge-per-second counter that should fall to ≈ ingest rate.

## 2. Per-entry String allocation storm in the grouping loop
- **Severity**: High
- **Category**: allocation
- **File**: `crates/core/src/collective/merge.rs:68-96,141-143,164-178,209-224`
- **Scenario**: Every `build_digest` and `merge_leaderboard` call — i.e. every leaderboard GET (finding #1) and every admin digest build over all projects' runs. Bites hardest exactly where the corpus is largest.
- **Root cause**: Four separate per-entry allocations that are almost always immediately dropped:
  - `key_of` (`:141-143`) allocates **three `String`s per entry** for `groups.entry(key)`. `BTreeMap::entry` needs an owned key only on *insert*; for the N−M entries that hit an existing bucket (the common case — grouping is the point) all three allocations are constructed, compared, and dropped.
  - `Acc::add` (`:93-95`) does `contributor.to_string()` on **every** sample before `BTreeSet::insert`, so a contributor with 50 buckets allocates its id 50 times, and re-contributions re-allocate an id the set already holds.
  - `Sample` takes `judge_provider: Option<String>` / `rubric_fingerprint: Option<String>` by value, forcing `e.judge_provider.clone()` and `e.rubric_fingerprint.clone()` at both call sites (`:174-175`, `:220-221`). `judge_providers` is a **closed 5-value vocabulary** (`anthropic|openai|google|unknown|mixed`, clamped by `canon_judge`) — the set can never exceed 5 members, yet we clone a `String` per entry to feed it.
  - `LeaderboardRow` construction (`:229`) does `a.judge_providers.iter().cloned().collect()` even though `a` is owned by the `into_iter()` closure and could be moved from.
  At N=50k entries this is ~200k+ transient heap allocations per request, plus retained memory: every `Acc` holds its own copy of each contributor id, so the `contributors` sets collectively retain N strings for the lifetime of the merge.
- **Impact**: The allocator becomes the dominant cost of the merge — this is a function whose actual arithmetic is a handful of FLOPs per entry. Expect the malloc/free pairs to outweigh the accumulation math several times over, and peak RSS during a merge to roughly duplicate the string content of the entire `collective_entries` table.
- **Fix sketch**:
  - Lookup-before-insert to kill the hot-path key allocation: `if let Some(a) = groups.get_mut(&borrowed_key) { a.add(..) } else { groups.insert(key_of(..), ..) }`. With a `BTreeMap` this needs a borrowed-key form; the clean version is to intern identities — keep a `HashMap<&str, u32>`-style interner (or `Rc<str>`) over the `entries` slice and key the map on `(u32, u32, u32)`, which also fixes finding #3.
  - `Acc::add`: take `contributor: Option<&str>` (already does) and gate the allocation — `if !self.contributors.contains(c) { self.contributors.insert(c.to_string()); }`. Better still, hold `BTreeSet<&'a str>` borrowed from the input slice: the entries outlive the merge, so no ownership is needed at all, and this drops the retained N-string duplication to zero.
  - `Sample`: change `judge_provider` / `rubric_fingerprint` to `Option<&'a str>` and have `Acc` hold `BTreeSet<&'a str>`, converting to `String` only in the ≤5 outputs per row. Both call sites then pass `e.judge_provider.as_deref()` — no clone.
  - `:229`: `std::mem::take(&mut a.judge_providers).into_iter().collect()` (or move out of the owned `Acc`) instead of `iter().cloned()`.
- **Trade-offs**: Borrowing ties `Acc`/`Sample` to the input slice's lifetime (`Acc<'a>`) — a mechanical signature change confined to this file; both public entry points already take `&[T]` and return owned types, so no API breakage. Measure with a criterion bench over a 50k synthetic entry set plus an allocation counter (e.g. a counting global allocator in the bench harness).

## 3. `BTreeMap<(String,String,String)>` grouping whose ordering is discarded by the subsequent sort
- **Severity**: Medium
- **Category**: data-structure / algorithmic
- **File**: `crates/core/src/collective/merge.rs:139-143,159,208,250`
- **Scenario**: Same hot path — the grouping loop in both `build_digest` and `merge_leaderboard` at 10k–100k entries.
- **Root cause**: Grouping uses a `BTreeMap` keyed on a 3-`String` tuple. Each `entry()` is O(log M) **tuple comparisons**, and each tuple comparison is up to three `String` comparisons (byte-wise memcmp over `provider`, then `model`, then `task_type`), which are pointer-chasing and cache-hostile. Crucially, the map's sorted order is then **thrown away**: `sort_by_quality` (`:199`, `:250`) fully re-sorts the output by quality desc. The BTree ordering buys exactly one thing — deterministic output for entries tied on `(quality, provider, model)` but differing in `task_type`, since `sort_by` is stable and the tiebreak chain (`:262`) omits `task_type`.
- **Impact**: ~log₂(M) × up-to-3 string compares per entry (≈15–50 string compares each at M=200–1000 buckets) where a `HashMap` would need one hash of ~40 bytes plus one bucket probe. Real, but bounded — it is a constant-factor win on the grouping loop, not a complexity class change (both are O(N log M) vs O(N)). Worth doing mainly because it composes with the interning fix in finding #2, after which the key is a `(u32,u32,u32)` `Copy` tuple and the comparison cost vanishes either way.
- **Fix sketch**: Swap `BTreeMap<Key, Acc>` for `HashMap<Key, Acc>` (or `HashMap<(u32,u32,u32), Acc>` with the interner from #2), and **restore determinism explicitly** by extending the tiebreak chain in `sort_by_quality` to include `task_type`: change the key closure to return `(f64, &String, &String, &String)` and add `.then_with(|| ta.cmp(tb))`. That makes the total order fully specified by the sort itself rather than incidentally by the map, which is the more honest design — the current determinism is an undocumented side effect of the container choice, and a future `HashMap` swap by someone else would silently make output order non-reproducible.
- **Trade-offs**: Requires the `sort_by_quality` signature change (4-tuple key) — trivial, two call sites in this file. Do **not** land the `HashMap` swap without the `task_type` tiebreak, or leaderboard row order becomes nondeterministic across runs for fully-tied rows and any snapshot test will flake. Benchmark the grouping loop in isolation before/after; if #2's interning lands first, this may measure as noise, in which case skip it and keep the `BTreeMap` (but still add the tiebreak).
