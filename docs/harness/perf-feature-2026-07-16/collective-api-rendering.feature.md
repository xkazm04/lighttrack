## 1. Contribution is all-or-nothing: no per-project opt-in, no consent record, no withdrawal
- **Severity**: Critical
- **Category**: capability-gap
- **File**: `crates/api/src/collective.rs:128-146,323-335`
- **Scenario**: An agency runs LightTrack across a dozen client projects. One project is a confidential eval for a customer under NDA; the rest are internal. An operator runs `lt collective contribute --hub https://hub.example` to help the network. Every project ships — there is no flag, prompt, or per-project switch that could have held the NDA'd one back. Later the customer asks "what of ours is on that hub, and take it down": there is no endpoint that answers either question.
- **Root cause**: `gather_run_stats` (`collective.rs:323-335`) walks `store.list_projects()? → list_benchmarks → list_benchmark_runs` unconditionally — every project, every benchmark, every run, with no scope filter anywhere in the chain. The only knob `get_digest` exposes is `DigestParams { min_cases }` (`:120-124`), a per-bucket size floor, not a scope control. `Collective::from_env` (`:58-76`) has `accept`/`allow_anon` for the **hub** side but *nothing* gating the **contributor** side: `LIGHTTRACK_COLLECTIVE_ID` only names the instance (`:59-62`), it does not authorize contribution, and an instance with it unset still builds a full digest under `anonymous`. Withdrawal exists in the store (`store/src/sqlite/collective.rs:49-55` `delete`) but is reachable only as a side effect of `post_ingest`'s delete-then-upsert (`collective.rs:229-236`) — to retract you must POST an empty digest to the hub and hope.
- **Impact**: This is the gate on every commercial deployment of the network. Per-project opt-in turns "I hope nobody runs that command" into a reviewable posture, and makes the digest preview an actual consent artifact instead of a curiosity. A withdrawal endpoint is table stakes for any pooled-data product facing a DPA. Both are cheap, and they are what converts the moat from "technically privacy-safe" to "signable".
- **Fix sketch**:
  1. Store: add `collective_opt_in INTEGER NOT NULL DEFAULT 0` to `projects`; expose `set_project_collective_opt_in(project_id, bool)` + carry the flag on `Project`. Default off — opting in is an act, not an inheritance.
  2. `gather_run_stats`: skip projects where `!p.collective_opt_in`, and count them. Return `(Vec<RunStat>, Included { n_projects }, Excluded { n_projects })`.
  3. `get_digest`: add a master switch `LIGHTTRACK_COLLECTIVE_CONTRIBUTE` on `Collective`; when unset, still build the preview but stamp the response `contributable: false` so `lt collective contribute` refuses to POST. Add `projects_included` / `projects_excluded` counts to `CollectiveDigest` (serde-defaulted, so v1/v2 hubs ignore them).
  4. `DigestParams`: add `project_ids: Option<String>` (comma-separated) to narrow a single push further — the intersect of opt-in ∧ requested.
  5. `render::collective::digest` (`render/src/collective.rs:60-79`): headline the scope — `### Contributable digest — N bucket(s) from 3 of 12 project(s), as \`c-abc\` (k≥5)` — so what leaves the building is legible *before* the POST, not inferred from row count.
  6. New `DELETE /v1/collective/contribution` on the hub: authenticate, derive the id via the existing `derive_contributor_id(&bearer)` (`:116-118`), call `store.delete_collective_entries`, return `{contributor_id, removed}`. Right-to-withdraw with the same key-derived identity guarantee ingest already has — ~20 lines against machinery that already exists.
- **Trade-offs**: The opt-in default flips existing behavior to contributing nothing until projects are marked — correct, but call it out in release notes or hubs will look like they died. Steps 1–3 are the shippable core; 4–6 can follow.

## 2. k-anonymity is enforced over cases but never over *sources* — single-contributor rows are published verbatim
- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/api/src/collective.rs:269-316`
- **Scenario**: A hub has one contributor benchmarking `cohere/command-r` on `summarization`. `GET /v1/collective/leaderboard?provider=cohere` returns a row whose quality, pass rate, cost/case, p50 and p95 belong to exactly one instance — and `n_contributors: 1` says so on the tin. Anyone with read access to the hub reads that operator's private eval results off the public leaderboard. The `?provider=` / `?task_type=` / `?judge=` filters (`:279-287`) make the isolation a one-request affair.
- **Root cause**: Both floors in `Collective` are case-count floors. `min_cases` (`:47-48`) drops under-k *buckets* at ingest (`:217-220`) — that anonymizes over cases within one contributor's bucket, which is not the same as anonymizing over contributors. `display_floor` (`:50-51`) only sets `low_confidence`, which the doc comment states plainly: "shown, not hidden". `get_leaderboard` computes `n_contributors` per row via the merge and *reports* it (`:293-302`) but never gates on it — no `retain(|r| r.n_contributors >= k)` exists anywhere on the serving path. A 5000-case single-source row is treated as maximally publishable precisely because it is large.
- **Impact**: The pitch is "compare against everyone else"; a row backed by one instance is not the collective, it is that instance's data on a billboard. A source floor is what lets a hub operator honestly tell a prospective contributor "your numbers only ever appear blended with at least K others" — the single most load-bearing sentence in getting the second, third and tenth contributor to join. Network effects are gated on this, not on merge quality.
- **Fix sketch**:
  1. `Collective`: add `min_contributors: u32` from `LIGHTTRACK_COLLECTIVE_MIN_CONTRIBUTORS` (default 2; `.max(1)` clamp mirroring `min_cases` at `:65-69`, so `1` explicitly opts a private/single-tenant hub out).
  2. `get_leaderboard`: after `merge_leaderboard` and **before** the filters, partition — `let (rows, held): (Vec<_>, Vec<_>) = rows.into_iter().partition(|r| r.n_contributors >= k)`. Suppressing before filtering matters: a filter must never be able to strip a row down to a lone source.
  3. `LeaderboardResponse`: add `held_back: usize` (rows suppressed for insufficient sources). Disclose the suppression rather than silently shrinking the board — the header counts are already computed over filtered rows (`:289-307`), so this fits the existing honesty convention.
  4. `render::collective::leaderboard` (`render/src/collective.rs:43-50`): when `held_back > 0`, append a note — `"N row(s) held back: fewer than K contributing sources."` — and reuse it in the empty-board branch (`:26-28`), which currently nudges you to contribute without explaining that one contributor will still show nothing.
  5. Extend `tests_collective.rs`: assert a 1-source row is absent at `k=2` and present at `k=1`; assert `?provider=` cannot resurrect it.
- **Trade-offs**: A fresh hub shows an empty leaderboard until two independent contributors overlap on a `(provider, model, task_type)` — a real cold-start cost. Step 4's disclosure is what keeps that from reading as "the product is broken", so it ships with the gate, not after. Default `2` is the weakest defensible K; hub operators handling sensitive evals will want `3`+.

## 3. `received_at` is stored, then never used: no freshness surface, no age filter, no retention
- **Severity**: High
- **Category**: half-implemented
- **File**: `crates/store/src/sqlite/collective.rs:57-67`
- **Scenario**: A contributor benchmarks `gpt-4o` in January and switches vendors. Eight months later their row still sits on the leaderboard — same weight, same authority, no marking — pooled with last week's data from someone still running the model. An operator reading `Confidence: 1,200 × 3` has no way to learn that two of those three sources went dark two quarters ago. There is no way to ask for "the last 90 days", and nothing ever prunes.
- **Root cause**: The column is fully wired and then abandoned. `upsert` writes `received_at` (`store/collective.rs:42`), `from_raw` parses it back (`:123`), a test pins the round-trip — and it dies there. `list` (`:57-67`) is `SELECT … FROM collective_entries` with no `WHERE`, no `ORDER BY`, no bound, and no `_since` sibling; `delete` (`:49-55`) keys on `contributor_id` only, so there is no time-based prune. `get_leaderboard` (`:276`) calls `list_collective_entries()` and never touches the field. `LeaderboardRow` carries no freshness member, so `render::model_row` (`render/src/collective.rs:107-133`) cannot show one and the footnotes (`:43-50`) — which are otherwise scrupulous about what each number means — say nothing about *when*.
- **Impact**: A leaderboard silently blending 18-month-old and 7-day-old evals is quietly wrong in the one way its readers cannot detect. Surfacing age costs a column and turns "how does my cost compare?" into a question with a defensible answer. Retention also caps unbounded table growth and gives the hub a data-minimization story — the collected-forever default is the other half of the DPA conversation from finding 1. (Distinct from the core-side merge-time drift finding: this is the serving boundary — a `since` predicate, an operator-visible age, and a prune path, all outside the merge engine.)
- **Fix sketch**:
  1. Store: `list_since(conn, cutoff: Option<DateTime<Utc>>)` — `WHERE received_at >= ?1`, using the existing `fmt_ts` for the bound (string ISO timestamps compare lexicographically, so no schema change); keep `list` as `list_since(conn, None)`. Add `prune_before(conn, cutoff) -> u64`, mirroring `delete`'s shape.
  2. `LeaderboardParams`: add `max_age_days: Option<u32>`; `get_leaderboard` maps it to a cutoff and calls `list_since`. Filtering at the store, before the merge, keeps `n_contributors` and the header counts honest for free — the counts already derive from `entries` (`:297-302`).
  3. `LeaderboardResponse`: add `oldest_contribution` / `newest_contribution` (`Option<DateTime<Utc>>`) over the entries backing visible rows — a per-board freshness envelope needing no `LeaderboardRow` change, so no core-side edit.
  4. Render: add the envelope to the header — `### Collective model leaderboard — 12 model(s), 3 contributor(s) · data from 2026-04-02 to 2026-07-14` — and a footnote when the span exceeds ~90 days, in the register of the existing "p50 is an approximate…" notes.
  5. Retention: `LIGHTTRACK_COLLECTIVE_RETENTION_DAYS` (default 0 = keep forever, preserving today's behavior); when set, `post_ingest` opportunistically calls `prune_before` inside its existing `spawn_db` block (`:229-236`) — no scheduler, and the hub is already doing a write there.
  6. Per-row age (`LeaderboardRow.oldest_case_at` + an `Age` column) is the natural follow-on but needs a core-side merge change — file it separately once the envelope proves the demand.
- **Trade-offs**: Steps 1–4 are additive and default to current behavior. Step 5's opportunistic prune ties deletion to ingest traffic, so a dormant hub never prunes — acceptable versus adding a scheduler, and worth a doc line. Step 3's envelope is coarser than per-row ages and will get asked about; ship it anyway, it is ~15 lines against a real gap.
