## 1. `daily_cost_by_dimension` full-scans the entire `events` table on every forecast request

- **Severity**: Critical
- **Category**: table-scan
- **File**: `crates/store/src/sqlite/forecast.rs:53-58`
- **Scenario**: Every `GET /v1/forecast` — a surface designed to be *polled* (`ForecastAlert::dedup_key` exists precisely so "a sustained forecast doesn't re-alert every poll"). One dashboard refreshing every 30s against a database holding a few million events runs a full-table scan of `events` each time, and does so while holding the store's global connection mutex.
- **Root cause**: The predicate is `WHERE (?1 IS NULL OR project_id = ?1) AND ts >= ?2 AND ts < ?3`. The `?1 IS NULL OR …` form is not sargable: SQLite cannot prove `project_id` is constrained at plan time, so it will not use `idx_events_project_ts`. That index is `(project_id, ts)`, and with the leading column unusable the `ts` range can't drive it either — and there is **no standalone `ts` index** in `schema/sqlite/001_init.sql`. The planner is left with a full scan of every row in `events`: all projects, all time, ignoring the `[since, until)` window entirely as an access path. On top of the scan, `json_extract(metadata, '$.customer_id')` is evaluated per scanned row — a JSON parse of the full metadata blob for every event ever stored, to serve a 14-day question.
  The contrast is right above it: `daily_usage` (line 24) uses plain `project_id = ?1 AND ts >= ?2 AND ts < ?3` and rides `idx_events_project_ts` cleanly. Only the dimension query pays this. Note also that the `NULL` branch is **dead** for this caller — `get_forecast` resolves the project up front and 400s without one (`crates/api/src/forecast.rs:105-106`), then always passes `Some(&proj)` (`crates/api/src/forecast.rs:223`). The whole index-defeating clause buys nothing on the forecast path.
- **Impact**: Work is `O(all events ever)` instead of `O(events in the lookback window)` — the ratio is the retention multiple. At 5M stored events with a 14-day window over 200k of them, that is ~25× the necessary rows scanned plus ~5M JSON parses, turning a sub-10ms indexed aggregate into a multi-second scan. Because `SqliteStore::with` serializes on one connection, that scan blocks **ingest admission** for its whole duration: a polled dashboard becomes backpressure on the write path, and latency degrades as the table grows rather than staying flat with the window. `cost_by_dimension` (`crates/store/src/sqlite/revenue.rs:100`) carries the identical clause and is called in the same `gather`, so a single request pays this twice.
- **Fix sketch**: Split the statement on the `project.is_some()` branch rather than encoding the optionality in SQL — emit `WHERE project_id = ?1 AND ts >= ?2 AND ts < ?3` when a project is given, and drop the `project_id` term entirely when it isn't. Both forms are sargable; the first uses `idx_events_project_ts`. Add an `EXPLAIN QUERY PLAN` assertion pinning `SEARCH events USING INDEX idx_events_project_ts` (the repo already uses this technique — see the `rowid`-scan test referenced in `usage_cache.rs:92-95`), so a future edit can't silently regress the plan. Apply the same split to `revenue.rs::cost_by_dimension`. If the project-agnostic (admin) path matters, add `CREATE INDEX idx_events_ts ON events(ts)` to cover it.
- **Trade-offs**: Two prepared-statement shapes instead of one interpolated string — a few lines of branching. No behavior change: both forms are semantically identical to today's query.

## 2. Per-dimension daily costs are materialized for every customer, then 50 are used

- **Severity**: High
- **Category**: allocation
- **File**: `crates/api/src/forecast.rs:154-161`
- **Scenario**: A project billing thousands of end-customers (the exact case per-customer margin forecasting exists to serve) asks for a 90-day lookback. `MAX_DIM_FORECASTS` caps the *output* at 50 keys — but the cap is applied at line 166, long after every key has been read out of SQLite and indexed into memory.
- **Root cause**: `daily_cost_by_dimension` has no `LIMIT` and no key filter, so it returns one row per `(day, customer)` pair — `days × distinct_customers`. `gather` collects all of it into `Vec<DailyDimCost>`, each row owning a heap-allocated `day` String and `key` String. The loop then re-allocates all of it again into a nested `HashMap<String, HashMap<String, f64>>`: `d.key.clone()` per row (line 156) plus `d.day.clone()` per row (line 160), building a complete per-day series for *every* key in the project. Only after that does `compute_margin` order the rows and `.take(MAX_DIM_FORECASTS)` pick 50 — at which point `dim_by_key.get(&row.key)` reads at most 50 of the entries. The rest were built purely to be dropped.
- **Impact**: With 5,000 customers × 90 days = 450k rows: ~900k String allocations from the row Vec, ~900k more from the map rebuild, and two live copies of the dataset at peak (tens of MB of transient heap, on a request path, in an allocation pattern that fragments). 99% of it is discarded — 4,950 of 5,000 keys are never read. It scales with the project's customer count, which is the dimension that grows fastest for a successful tenant, and it is paid on every poll. This stacks on top of finding #1: the same query is both scanning the wrong number of rows *and* returning far more of them than the response can use.
- **Fix sketch**: Choose the top-N keys *before* building the series. `compute_margin(&raw.revenue, &raw.costs, …)` already ranks the dimensions from the (much smaller) rollup, so the winning 50 keys are known independently of the daily breakdown. Reorder: compute `rows`, take the 50 keys into a `HashSet<&str>`, then build `dim_by_key` with a `if !wanted.contains(key) { continue; }` guard at the top of the loop — a two-line change that bounds the map to 50 entries. Better still, push the bound into SQL: add an optional `keys: &[&str]` parameter to `daily_cost_by_dimension` and filter with `json_extract(metadata, '$.customer_id') IN (…)` so the rows never cross the store boundary. Use `std::mem::take(&mut d.key)` / consume `raw.daily_dim` by value to drop the `.clone()` pairs on whatever does get built.
- **Trade-offs**: Requires computing `rows` before `dim_by_key` (a statement reorder within the same function — no new I/O; `compute_margin` reads `raw.revenue`/`raw.costs`, which `gather` already fetched). The SQL-side variant couples the store signature to the caller's cap; the `HashSet` variant avoids that and captures most of the win.

## 3. Every forecast is recomputed from scratch, and `usage_since` re-aggregates the full window instead of using the existing incremental cache

- **Severity**: Medium
- **Category**: missing-cache
- **File**: `crates/api/src/forecast.rs:202-234`
- **Scenario**: Several operators keep a forecast dashboard open, or an alerting loop polls `/v1/forecast` on a timer. Each request pays the full `gather`: `daily_usage` + `list_limit_rules` + up to three `usage_since` aggregates + `list_revenue_events` + `cost_by_dimension` + `daily_cost_by_dimension` — six-plus queries, all serialized on the one store connection, all recomputed even when nothing has been ingested since the last call.
- **Root cause**: Two compounding gaps.
  (a) **No memoization anywhere.** Despite the context brief's "persistence of forecast snapshots", there is no snapshot table in `schema/sqlite/001_init.sql` and no cache in the handler — the response is rebuilt per request. Most of the inputs are *immutable*: `daily_usage` buckets for every day before today can never change (events are append-only and bucketed by UTC day), yet all `lookback` days are re-aggregated each time. Only today's bucket is live.
  (b) **The gather-loop `usage_since` bypasses the incremental cache that already exists.** `crates/api/src/forecast.rs:218` calls `store.usage_since(…)`, which routes through `SqliteStore::usage_since` → `events::usage_since` (`crates/store/src/sqlite/events.rs:519-533`) — a `SUM`/`COUNT` over every event in the window. `usage_cache.rs` was built to kill exactly this cost ("re-aggregates the *whole* rolling window … a hundreds-of-thousands-of-row scan"), makes it `O(events since last check)`, and is proven exact by property tests — but it is wired only into `insert_event_checked`, not the read path. A `Month`-window rule therefore re-sums 30 days of events per forecast poll. The loop's `contains_key` guard correctly dedups to ≤3 windows, so this is bounded — but each one is a full-window aggregate.
- **Impact**: A dashboard polling every 30s runs ~2,900 redundant full-window aggregates/day, ~99% of which return a value identical or near-identical to the previous poll. On a Month-window rule over a busy project that is a 100k+ row scan per poll, each one holding the global store mutex and pushing latency onto concurrent ingest. Bounded by the window rather than by table size (unlike #1), which is why this is Medium — but it is pure repeated work on stable data, and it is the reason the cheap fix is attractive: the machinery already exists and is already tested.
- **Fix sketch**: Two independent steps, smallest first. (i) Route the read path through `UsageCache` — expose a store method that reads the same `usage()` the admission path uses, so the forecast's `usage_since` becomes a bounded `rowid > seen` delta scan. The cache lives behind the store mutex and is already exact-by-property-test; this is a wiring change, not new logic. (ii) Add a short-TTL memo in `AppState` keyed by `(project, dim, horizon, lookback)` holding the `RawForecast` (or the finished `ForecastResponse`) with a 30-60s TTL — the forecast is explicitly advisory ("about N days", not a guarantee), so minute-scale staleness is within its stated contract. Guard `st.alerts.notify_forecast` to fire only on a genuine recompute so a memo hit doesn't re-notify (the sink's cooldown already dedups on `dedup_key`, so this is belt-and-braces). If polling grows beyond a handful of clients, promote the memo to the real snapshot table the brief implies: persist per-day rollups on ingest and read the closed days from there, recomputing only today's bucket.
- **Trade-offs**: The TTL memo trades freshness for load — acceptable for a day-granularity trend model, but it means a just-ingested event may not move the projection for up to a minute; make the TTL a config knob and set it to 0 in tests (`tests_forecast.rs` ingests then immediately reads back, and would break against a warm memo keyed before ingest). Step (i) has no staleness cost at all — it is exact — and is worth doing regardless of whether (ii) lands.
