## 1. Resource templates cover 3 of 7 addressable entities — the prompt registry is unattachable

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/mcp/src/resources.rs:14-18` (`KINDS`), `:87-94` (`render_kind`)
- **Scenario**: A developer in Claude Code wants to say "review @lighttrack://prompt/proj-1/checkout-summarizer and tell me why v4 regressed". The client offers three template completions — trace, event, benchmark — so there is no way to attach the prompt, the dataset it is benchmarked on, the rubric that judged it, or the job that ran it. Every one of those must instead be pulled through a tool call whose output lands in the conversation as a transient tool result rather than as a durable, re-readable attachment the user picked. The prompt registry is the product's differentiating surface ("the agent manages prompts, gated by benchmarks") and it is precisely the thing that cannot be attached.
- **Root cause**: `KINDS` is a hardcoded 3-entry table (`trace`, `event`, `benchmark`). The read surface exposes seven single-entity fetches — `get_event`, `get_trace`, `get_benchmark`, `get_dataset` (`read.rs:138`), `get_rubric` (`read.rs:141`), `get_job` (`read.rs:143`), `get_prompt` (`prompts_tools.rs:101`) — and `lighttrack_render` already ships a renderer for **all seven** (`crates/render/src/lib.rs:47-58`: `datasets::detail`, `rubrics::detail`, `jobs::detail`, `prompts::resolved`). So the resource layer stops at three purely because the table says three; the resolve-and-render machinery behind it is already complete. `render_kind` mirrors the same three-arm match. The asymmetry is exact and mechanical: readable-by-tool but not attachable-by-URI.
- **Impact**: Roughly doubles the attachable surface for near-zero new machinery, and closes the gap on the flagship workflow (attach a prompt version + its benchmark + the run job, ask "what regressed?"). Attachment is also the cheapest context path: the user picks the entity once and the client re-reads it, instead of the agent burning a tool round-trip per entity per turn.
- **Fix sketch**:
  1. Add `("dataset", "/v1/datasets/", …)`, `("rubric", "/v1/rubrics/", …)`, `("job", "/v1/jobs/", …)` to `KINDS`, and the matching arms to `render_kind` (`"dataset" => "get_dataset"`, etc.). This is the whole change for three of the four — the `c.get(&format!("{prefix}{id}"))` path at `:58` already works for any `/v1/{kind}/{id}` shape.
  2. Prompts need a two-segment id (`project` + `name`), not a flat one. `parse_uri` already supports this: it splits only on the *first* `/` and keeps the remainder intact — the test at `:130` pins `lighttrack://trace/a/b` → id `a/b`. So `lighttrack://prompt/{project}/{name}` parses today; it needs a per-kind path builder rather than the single `{prefix}{id}` concat, e.g. extend `KINDS` with a formatter or special-case `prompt` to `/v1/projects/{project}/prompts/{name}`.
  3. Optionally accept a `?label=production` / `?version=N` suffix on the prompt URI, mapping to the selector `get_prompt_path` already builds (`prompts_tools.rs:132-140`), so a user can attach "whatever production points at right now".
  4. Extend `templates_cover_all_three_kinds` to assert one template per read-tool entity, so a future `get_*` tool that ships without a template fails the test.
- **Trade-offs**: The prompt kind breaks the uniform `{prefix}{id}` assumption and is the only kind needing a composite id — worth isolating in a small `path_for(kind, id)` helper rather than smearing a special case through `read`. Templates are advertisements, so adding them cannot break an existing client. Jobs are ephemeral, so an attached job URI may 404 after retention expiry — acceptable, and it flows through the normal error mapper.

## 2. Errors are prose-only — an agent cannot tell "retry this" from "this will never work"

- **Severity**: High
- **Category**: agent-experience
- **File**: `crates/mcp/src/errors.rs:7-39` (`map_error`, `guidance`)
- **Scenario**: An agent runs a nightly cost sweep and calls `query_events`. It gets back `isError: true` with the text `error: LightTrack API error — is the server healthy? (try the /health endpoint).\n\nupstream boom`. Should it back off two seconds and retry, or abandon the sweep? A 503 is almost always worth retrying; a 400 (`threshold must be > 0`) never is; a 429 depends entirely on whether the rule's action was `throttle` (wait) or `block` (stop, the budget is spent). All four arrive as an opaque English sentence with no status code, so the agent is left regex-matching prose it did not write — and `map_error` deliberately *strips* the `HTTP {code}: ` prefix at `:20`, so even the raw status is gone from the string by the time the agent sees it. In practice agents guess, and they guess wrong in both directions: retrying a 400 in a loop, or giving up on a transient 503.
- **Root cause**: `map_error` is a `&str -> String` transform. It parses out the status (`parse_http`, `:42-46`), branches on it in `guidance`, and then throws the structured knowledge away, re-encoding it as English. `tools::call` (`tools.rs:33, 56`) wraps the result in `tool_text(&map_error(&e), true)`, which per `rpc.rs:21` emits only `{content: [text], isError: true}` — no `structuredContent`, unlike the success path (`tool_rendered`, `rpc.rs:33`) which carries a full structured payload *and* declares an `outputSchema` (`schemas.rs`). Errors are the one surface with no machine-readable contract. Relatedly, `main.rs:88` maps *every* `resources/read` failure to JSON-RPC `-32602` (Invalid params) — a 503 from the API is reported to the client as the caller's bad argument, because the classification needed to pick a better code does not survive `map_error`'s return type.
- **Impact**: Turns error handling from prose-guessing into a contract. Unblocks correct retry/backoff in unattended agent loops and CI steps (`check_benchmark_gate` in a pipeline), and lets a client render errors properly instead of dumping a paragraph. This is the difference between "the agent self-corrects" and "the agent dead-ends" — the stated bar for this surface.
- **Fix sketch**:
  1. Introduce `pub(crate) struct MappedError { status: Option<u16>, kind: ErrorKind, retryable: bool, guidance: Option<&'static str>, body: String }` with `ErrorKind` = `Auth | NotFound | Validation | RateLimit | GateBlocked | Server | Transport`. `retryable` is true for `Server` and `Transport`; false for `Auth`/`Validation`/`NotFound`/`GateBlocked`; for `RateLimit`, derive it from the breach body (throttle → retryable, block → not).
  2. Make `classify(raw) -> MappedError` the primitive and keep `map_error(raw) -> String` as a thin `Display` over it, so every existing call site and all eight tests in this file keep passing unchanged.
  3. Add `tool_error(&MappedError)` in `rpc.rs` emitting `{content: [text], structuredContent: {status, kind, retryable, guidance}, isError: true}`. Point `tools::call`'s two error arms at it. Declare the error shape once in `schemas.rs` so it is discoverable.
  4. In `main.rs:88`, choose the JSON-RPC code from `kind` (`-32602` for `Validation`/a genuinely malformed URI, `-32603` for `Server`/`Transport`) instead of hardcoding `-32602`.
  5. Consider surfacing `Retry-After` from a 429 if the API sends it — that upgrades `retryable: true` into an actual wait duration.
- **Trade-offs**: `MappedError` is a new internal type but is contained to this module plus two call sites; the public string behavior is unchanged by construction. Classifying 429 throttle-vs-block means matching on the breach body, which is a coupling to the API's wording — mitigate by defaulting to `retryable: false` (the safe direction: stop rather than hammer a spent budget) when the body does not clearly say throttle.

## 3. `update_limit.scope` ships as an untyped object — the one nested param the schema module skipped

- **Severity**: Medium
- **Category**: half-implemented
- **File**: `crates/mcp/src/write_schemas.rs:1-4` (module contract), `write.rs:115` (the untyped `scope`)
- **Scenario**: An agent is told "cap gpt-4o at $5/day on project p1". It calls `update_limit` and must invent the shape of `scope` from a prose example embedded in a description string. It has three plausible guesses — `{"kind":"model","value":"gpt-4o"}`, `{"model":"gpt-4o","provider":"openai"}`, `{"model":"gpt-4o"}` — and only the third works. The first two come back as a serde deserialization 400, which (per finding #2) is prose the agent then has to interpret. This is exactly the failure `write_schemas.rs` exists to prevent: its own module doc says the schemas "mirror exactly what the API deserializes … so an agent fills the right field names instead of guessing from prose."
- **Root cause**: `write_schemas.rs` defines three helpers — `rubric_dimensions`, `benchmark_dataset`, `benchmark_targets` — covering `create_rubric` and `create_benchmark`. `update_limit`'s `scope` was left inline in `write.rs:115` as `{"type":"object","description":"dimension scope, e.g. {\"model\":\"gpt-4o\"} or …"}`: no `properties`, no `oneOf`, no `additionalProperties:false`. The real type is fully closed and knowable — `LimitScope` (`crates/core/src/limits.rs:11-18`) is a 3-variant externally-tagged enum (`Provider|Model|Name`) serializing as a single-key object, confirmed by its doc comment at `:9-10`. An externally-tagged enum rejects both zero keys and two keys, so the permissive `{"type":"object"}` schema actively *admits* payloads the API is guaranteed to reject. Compounding this, `create_limit` (`write.rs:95-104`) omits `scope` from its schema entirely even though `CreateLimitReq` accepts it (`crates/api/src/limits.rs:74`) — so the one-call form of "cap gpt-4o at $5/day" is unreachable and the agent must create a project-wide rule and then `update_limit` it into scope, briefly installing a cap it never intended.
- **Impact**: Makes per-model/per-provider/per-use-case budgets — the feature `LimitScope`'s doc pitches ("cap gpt-4o at $5/day … without touching other traffic") — reliably reachable in one call instead of by trial and error. Scoped budgets are the sharp end of the cost-control story; an agent that cannot address them falls back to blunt project-wide caps.
- **Fix sketch**:
  1. Add `pub(crate) fn limit_scope() -> Value` to `write_schemas.rs`, mirroring `LimitScope` as a closed choice:
     `{"type":"object","description":"…; omit for project-wide","oneOf":[{"required":["provider"],"additionalProperties":false,"properties":{"provider":{"type":"string"}}}, …same for model / name…]}`. Keep the three-variant list adjacent to `LimitScope` in spirit — a `// mirrors core::limits::LimitScope` comment matching the existing header's citation style.
  2. Reference it from `update_limit`'s schema in place of the inline object.
  3. Add `"scope": crate::write_schemas::limit_scope()` to `create_limit`'s schema and `"scope"` to its `post_with` body_keys (`write.rs:190-197`) — the API already deserializes it, so this is a one-line capability unlock.
  4. Add a test in the module's existing style asserting the three variants and that `additionalProperties` is closed, mirroring `benchmark_targets_use_system_prompt_not_prompt` which pins the same "don't let the agent guess the field name" property.
- **Trade-offs**: Not every MCP client's schema validator handles `oneOf` well; a tolerant fallback is a single object with all three optional properties plus `maxProperties: 1` and a description stating exactly one is required. `create_prompt_version.config` is the other untyped object (`prompts_tools.rs:76`) but is genuinely free-form by design (`crates/core` marks it "Free-form"), so it is correctly left open — not a gap.
