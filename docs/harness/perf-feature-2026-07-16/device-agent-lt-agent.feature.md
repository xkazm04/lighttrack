## 1. Rate-limit backoff is half-implemented: the reset time is parsed away and never used

- **Severity**: High
- **Category**: half-implemented
- **File**: `crates/agent/src/exec.rs:29-45,64,119-125` + `crates/agent/src/run.rs:24-54`
- **Scenario**: The operator's Claude subscription window is exhausted at 14:00 with 3 tasks leased (`max_batch = 3`) across two sources. The first task fails with `Claude AI usage limit reached|<epoch>`, settles `deferred` — and the agent immediately spawns the CLI for task 2, gets the same wall, then task 3, then rotates to the next source and does it again, and after `poll_secs` (15s) leases a fresh batch and repeats. For the ~5h until the window resets, the agent hammers the CLI every 15s and every one of those tasks bounces around the queue.
- **Root cause**: Two gaps compound. (a) `RunReport::deferred` (exec.rs:42-44) is built as `RunReport { status: "deferred", ..Self::failed(reason) }`, and `failed` hard-codes `retry_after_secs: None` (exec.rs:33). So the `retry_after_secs` field that `cloud.rs:72-87` faithfully serializes and that RELAY.md advertises ("settles `deferred` with an optional `retry_after_secs`") is **structurally impossible to ever populate** — the only constructor for a deferred report nulls it. The cloud therefore re-offers the task on its default schedule, not when the window actually reopens. (b) The agent keeps no local rate-limit state: `run.rs`'s loop branches only on `"succeeded"` vs everything else (run.rs:36-45) and never lets a `deferred` short-circuit the remaining batch, the remaining sources, or the next poll. The reset timestamp is *right there* — the CLI's usage-limit stderr carries it after a pipe, as the module's own test fixture shows (`"Claude AI usage limit reached|123"`, exec.rs:167) — and `rate_limited()` (exec.rs:119-125) throws it away, returning a bare `bool`.
- **Impact**: Closes the loop the retry design already promises. Tasks come back exactly when capacity exists instead of drifting on a blind 5h interval; the device stops burning hundreds of pointless CLI spawns (each carrying Claude Code's ~30k-token context overhead, which per RELAY.md consumes window capacity — so the futile retries actively poison the next window); and the operator's other sources aren't starved by a rotation spent hitting a known wall.
- **Fix sketch**:
  1. Change `rate_limited(&EngineError) -> bool` into `rate_limit_reset(&EngineError) -> Option<Option<u64>>` — `Some(epoch)` when the stderr matches and a trailing `|<digits>` parses, `Some(None)` for a match without one. Keep the existing substring set (`usage limit`/`rate limit`/`429`/`overloaded`).
  2. Give `RunReport::deferred` a `retry_after_secs` parameter and compute it as `epoch.saturating_sub(now)`, clamped to a sane ceiling (e.g. 6h) and floored to a default (e.g. 900s) when the epoch is absent.
  3. Add a `paused_until: Option<Instant>` to the loop in `run.rs`. On a deferred report: settle the current task, then settle every *remaining already-leased task* in the batch as `deferred` with the same `retry_after_secs` **without invoking Claude** (hand the attempts straight back — they cost nothing), and set `paused_until`.
  4. While `paused_until` is in the future, skip leasing on all sources and sleep to the earlier of the reset and `poll_secs`; log the resume time once. Under `--once`, exit rather than sleep out a 5h window.
- **Trade-offs**: The stderr format is a CLI implementation detail — hence step 1 degrading to a default when the epoch is missing, which is no worse than today. The pause is machine-global (correct: one subscription, one window), so a source whose tasks would have succeeded also waits — acceptable, since the CLI is the shared constraint. Clamping guards against a garbage epoch parking the agent for years.

## 2. The device never tells the cloud which actions it can actually run

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/agent/src/cloud.rs:57-69` + `crates/agent/src/exec.rs:47-53` + `crates/agent/src/actions.rs:60-83`
- **Scenario**: An xprice app ships a new `action_type` (or an app enqueues `xprice/reprice-sumary` — a typo) before the operator adds the folder to the gitignored library. The task is leased, fails instantly with `action: no action '…'`, and burns a real attempt. With defaults (`max_attempts = 4`, `retry_interval_secs = 18000`), the queue spends **~20 hours** re-serving a task the device provably cannot execute, then dead-letters it and pages the operator via `relay_task_dead`. Meanwhile every one of those leases displaced real work from the batch.
- **Root cause**: The library is the sole authority on what is executable — `actions::load` (actions.rs:60-83) resolves `<actions_dir>/<action_type>/` and the cloud, by design, knows only a name and params. But the lease request carries no capability information: `Client::lease` posts exactly `{ device, max, lease_secs, wait_secs }` (cloud.rs:64-67), and the cloud side matches on device identity alone (`store.lease_relay_tasks(&device, lease_secs, max)`, `crates/api/src/relay.rs:187`). The device therefore discovers the mismatch only *after* winning the lease, in `exec::execute` (exec.rs:48-52) — and the comment there makes the current choice explicit and deliberate ("retrying later is right — the user can add the action to the library between attempts"). That reasoning holds for a genuine race, but it makes an unsatisfiable task and a temporarily-missing one indistinguishable, and pays the 20h dead-letter envelope for both.
- **Impact**: Turns a 20-hour silent-failure envelope into an immediate, correct signal. The mechanism also unlocks the multi-device future RELAY.md defers ("multi-device enrollment … is future work"): once lease requests carry a capability set, heterogeneous devices — one with the xprice library, one with an internal one — fall out for free, because the queue can route by what a device *has* rather than by what it is *named*.
- **Fix sketch**:
  1. **Device-side only, ship first**: in `exec::execute`, split the load error. A `validate_action_type` rejection is permanently unsatisfiable → keep `failed` (fail fast, dead-letter fast). A *missing directory* → settle `deferred` with a `retry_after_secs` (reusing finding #1's plumbing) instead of `failed`, so the operator's grace period costs zero attempts. This preserves the comment's intent while removing its 4-attempt price.
  2. Add `actions::scan(actions_dir) -> BTreeSet<String>` — walk the library for directories containing `prompt.md`, reusing `validate_action_type` to skip junk. Call it at startup and print the count next to the existing banner (`main.rs:35-42`), which today reports `actions=<dir>` — a path, not a capability. That alone catches "the agent is pointed at the wrong folder" at boot.
  3. Send `actions: [...]` on the lease body (cloud.rs:64-67) and have the cloud filter due tasks by `action_type IN (…)` when present, omitted ⇒ today's behavior (backward-compatible with an older agent).
  4. Re-scan on each loop round (the library is small; a stat-walk is cheap) so an action added mid-run is picked up without a restart.
- **Trade-offs**: Step 3 alone touches the cloud lease query and `Store::lease_relay_tasks`'s signature across backends — that's why steps 1–2 are staged first and deliver most of the value device-side with no protocol change. Filtered leasing needs its own visibility: a task no device advertises would sit `queued` and silently miss its deadline, so it should still age into the existing dead-letter sweep rather than wait forever.

## 3. No way to author or test an action without a round trip through the cloud

- **Severity**: Medium
- **Category**: workflow
- **File**: `crates/agent/src/main.rs:20-29` + `crates/agent/src/exec.rs:47-95`
- **Scenario**: The operator is writing `actions/xprice/reprice-summary/`. To find out whether `prompt.md`'s `{{params.sku}}` resolves, whether `action.toml`'s model string is valid, whether the model's output actually conforms to `schema.json`, or whether the connector's `${API_TOKEN}` header is set, they must enqueue a real task from a real app with a real project key and watch the agent's stdout. Each iteration costs a cloud round trip and a subscription-window CLI run, and a typo in a `${…}` header is only discovered *after* Claude has run and the tokens are spent.
- **Root cause**: The CLI exposes exactly two knobs — `--config` and `--once` (main.rs:22-29) — and `--once` still only drains whatever the *cloud* has queued. Every ingredient for a local dry run already exists and is already decoupled from the network: `actions::load` + `actions::render` need only a `RelayTask` (a plain struct the module's own tests synthesize from a `json!` literal, actions.rs:162-167), and `exec::execute` takes `(&AgentConfig, &EngineConfig, &RelayTask)` with no cloud client anywhere in its signature. There is simply no entry point that constructs a synthetic task. The pieces are assembled for the cloud path only.
- **Impact**: Collapses the authoring loop from minutes-plus-a-cloud-task to seconds, and makes the library's failure modes (unresolved placeholder, missing env var, non-conforming schema output) diagnosable at author time. This is the operator's *only* real authoring surface — the library is gitignored, so it is written by hand on the device, with no CI to catch these.
- **Fix sketch**:
  1. Restructure the CLI into subcommands, keeping bare `lt-agent` ⇒ today's loop for compatibility.
  2. `lt-agent try <action_type> --payload '<json>' [--dry-run] [--no-connector]`: build a synthetic `RelayTask` (`id: "local-try"`, payload from the flag) and hand it to `exec::execute`. `--dry-run` stops after `actions::render` and prints the final prompt plus the resolved model/system/schema — zero tokens, and it surfaces the "resolved empty" placeholder warning `render` already emits (actions.rs:101). `--no-connector` runs Claude but skips delivery, for iterating on prompt quality without spamming the app.
  3. `lt-agent validate`: walk the library (reusing finding #2's `scan`), and for each action assert `prompt.md` exists, `action.toml` parses, `schema_file` resolves and is valid JSON, and every connector header's `${VAR}` is set — `expand_headers` (actions.rs:149-154) already returns a precise error for the last one; this just calls it at author time instead of post-run. Non-zero exit on any failure makes it usable as a pre-flight in the agent's own startup or a cron wrapper.
  4. Have `try` reuse `exec::execute` verbatim rather than reimplementing it, so the test path cannot drift from the production path.
- **Trade-offs**: Small surface addition (~a subcommand enum plus a validate walker); no change to the loop or the protocol. `try` executes real connectors by default, which can write to a live app — hence `--no-connector`, and `--dry-run` as the safe default suggestion in `actions/README.md`.
