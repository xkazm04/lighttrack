# Performance Optimizer — Prompt Registry & Versioning

> Total: 3

## 1. Runtime fetch of the latest version loads the entire version history

- **Severity**: High
- **Category**: hot-path
- **File**: `crates/api/src/prompts.rs:213-229` (store: `crates/store/src/sqlite/prompts.rs:98-107`, `crates/store-firestore/src/prompts.rs:67-71`)
- **Scenario**: `GET /v1/projects/:pid/prompts/:name` with neither `?version=` nor `?label=` — documented as "the hot path apps call at startup", and the default path of the MCP `get_prompt` tool (`crates/mcp/src/prompts_tools.rs:132-140` appends no selector unless the agent passes one). A mature prompt has tens-to-hundreds of versions, each carrying a full prompt template (`content` is commonly 2–20 KB).
- **Root cause**: To compute the latest version number the handler calls `store.list_prompt_versions(&id)` and then `.iter().map(|x| x.version).max()`. `list_prompt_versions` is `SELECT id, prompt_id, version, content, config, note, created_at … ORDER BY version DESC` — every column of every row. So the whole history is read off disk, allocated into `String`s, JSON-parsed (`val_or_null` on `config` per row), and then all of it but a single `u32` is dropped. The chosen version is then fetched *again* via `get_prompt_version`. There is no `latest`-shaped accessor on the `Store` trait (`crates/store/src/lib.rs:520-557`), so no backend can answer this cheaply.
- **Impact**: Per fetch-latest request, roughly `N × avg_content_bytes` of I/O, allocation and JSON parsing where 1 row is needed. At 100 versions × 8 KB that is ~800 KB read + parsed to serve ~8 KB — a ~100× amplification that grows linearly and permanently with every version cut (the table is append-only, so this only ever gets worse). On the Firestore backend it is worse than CPU: `list_prompt_versions` is an unlimited `query` with no `limit`, so it is **N billed document reads plus N × content egress per request** — a startup fetch from a fleet of app instances multiplies that by the fleet size.
- **Fix sketch**: Add `fn get_latest_prompt_version(&self, prompt_id: &str) -> Result<Option<PromptVersion>>` to the `Store` trait (default impl delegating to `list_prompt_versions` so other backends compile unchanged). SQLite: `SELECT {VERSION_COLS} FROM prompt_versions WHERE prompt_id = ?1 ORDER BY version DESC LIMIT 1` — served directly by the existing `idx_prompt_versions_pid(prompt_id, version)` index. Firestore: reuse the existing `rest.query("prompt_versions", &filters, Some(("version", true)), Some(1))` shape already used by `get_prompt_version`. Then in `get_prompt` the no-selector branch becomes one store call that also returns the row, collapsing today's two round trips into one.
- **Trade-offs**: One new trait method plus two backend impls. Behaviour is identical (both paths mean "highest version"). None material.

## 2. No cache on the registry read path, despite content being immutable and version-keyed

- **Severity**: High
- **Category**: missing-cache
- **File**: `crates/api/src/prompts.rs:194-238` and `:288-294`
- **Scenario**: Every `GET …/prompts/:name` — the fetch endpoint apps and agents call at startup and on every prompt refresh — goes to the store twice: `load_prompt` (`get_prompt` by `(project, name)`) then `get_prompt_version(prompt_id, version)`. Nothing in `AppState` (`crates/api/src/state.rs:18-38`) holds prompts, even though it already carries a hot-swappable `Arc<RwLock<PriceBook>>` and an in-process idempotency cache — the pattern exists, it just isn't applied here.
- **Root cause**: `PromptVersion` rows are immutable by design ("a registry edit is a new version, never an in-place overwrite" — `crates/core/src/prompt.rs:7-10`), so a `(prompt_id, version)` pair is a perfect, never-invalidating cache key, and every one of those reads is re-derived from the store on every request. The two calls are also strictly sequential (the second needs `prompt.id`). On Firestore both are `query` calls with filters — `get_prompt_version` (`crates/store-firestore/src/prompts.rs:54-65`) queries on `(prompt_id, version)` rather than doing a `get_doc`, so it needs a composite index and is billed and latency-priced as a query even though the row is uniquely identified.
- **Impact**: 2 sequential store round trips per fetch, ~0 of which are logically necessary after the first. On Firestore that is 2 billed queries + 2 network RTTs (typically 20–60 ms each) on an app's startup-critical path; a hot `production` label served to a fleet re-pays this for every instance, every restart, forever. With finding #1 unfixed the same request also pays the full-history scan.
- **Fix sketch**: Two tiers, cheapest first. (a) Set `Cache-Control` on the response in `get_prompt`: `public, max-age=…, immutable` when `?version=` was explicit (truly immutable) and a short `max-age` (e.g. 30–60 s) for `?label=`/latest, plus an `ETag` of `{prompt_id}:{version}` so refetches are 304s — this alone removes most fleet traffic with no server state. (b) Add an `Arc<RwLock<HashMap<(String,u32), PromptVersion>>>` (or a small LRU) to `AppState` keyed by `(prompt_id, version)`, populated in `get_prompt` — no invalidation logic is needed because the key is immutable; `create_prompt_version` only ever adds. The label→version pointer stays uncached (it is mutable via `promote`), so a promotion is still reflected on the next request. Separately, make Firestore's `get_prompt_version` a `get_doc` by keying `prompt_versions` documents on `{prompt_id}:{version}` instead of `v.id`, dropping a query and a composite index.
- **Trade-offs**: (a) a label promotion takes up to `max-age` to reach clients — bound it to seconds and it is well within what a label rollout tolerates; use `no-store` if a caller needs strict read-after-promote. (b) unbounded memory if unbounded — cap it (LRU, or bound entry size) since `content` is user-supplied. The Firestore doc-key change is a data-layout change and needs a migration or a read-fallback for existing docs.

## 3. Cutting a new version re-reads the whole history to compute `max + 1`

- **Severity**: Medium
- **Category**: hot-path
- **File**: `crates/api/src/prompts.rs:137-154` (`next_version`, `crates/api/src/prompts.rs:318-320`)
- **Scenario**: `POST …/prompts/:name/versions`, and the MCP `create_prompt_version` tool (`crates/mcp/src/prompts_tools.rs:169-188`), which is the agent-facing write path — an agent iterating on a prompt hits this repeatedly.
- **Root cause**: Same shape as finding #1 on the write side: `list_prompt_versions(&id)` pulls every column of every version — all `content` strings, all `config` JSON parsed via `val_or_null` — and `next_version` reduces the whole `Vec<PromptVersion>` to `max(version) + 1`.
- **Impact**: The full history read + parsed per version cut (~800 KB at 100×8 KB), and on Firestore N billed reads per write. Bounded to the write path, which is far colder than the fetch path — hence Medium, not High. It also holds a `spawn_blocking` worker for the duration of the scan, and the read-then-insert is not atomic, so two concurrent cuts can both compute the same `next` and one loses to the `UNIQUE (prompt_id, version)` constraint (correctness note, not the perf finding).
- **Fix sketch**: Reuse the `get_latest_prompt_version` accessor from finding #1 — `next = latest.map(|v| v.version).unwrap_or(0) + 1` — an index-only `LIMIT 1` lookup. Even cheaper, add `fn max_prompt_version(&self, prompt_id: &str) -> Result<u32>` backed by `SELECT COALESCE(MAX(version), 0) FROM prompt_versions WHERE prompt_id = ?1`, which touches the index and no row bodies at all. Keep `next_version`'s unit tests by having them exercise the pure `max + 1` arithmetic.
- **Trade-offs**: None material — identical semantics (`next_version` is already order-independent `max + 1`, not `count + 1`).
