# Performance Optimizer — Judge Pairwise, Parsing & Provider Pool

> Total: 3
> Critical: 1 | High: 1 | Medium: 1 | Low: 0

## 1. Round-robin judge cost is quadratic in targets with no cap, estimate, or confirmation
- **Severity**: Critical
- **Category**: combinatorial-blowup
- **File**: `crates/runner/src/pairwise.rs:115-135` (game enumeration + judging), with the per-game call fan-out in `crates/engine/src/pairwise.rs:112-125`
- **Scenario**: A user runs a benchmark with 8 targets over 100 cases — a perfectly ordinary "which prompt variant do I ship?" sweep, and only two more targets than the 6 that feel routine. Nothing in the CLI path warns them before the first call goes out.
- **Root cause**: `games` is the full unordered round-robin per case: `n_c · C(n_t, 2)` = **O(n_t² · n_c)**. Each game then costs *two* judge calls, because `pairwise_via` deliberately judges both A/B orders for debias (`engine/src/pairwise.rs:122-123`). So provider calls = `n_c · n_t · (n_t − 1)` — 8×100 ⇒ 28 pairs × 100 cases × 2 = **5,600 judge calls**. That is the floor: `sample_parsed` (`parse.rs:94-106`) adds a repair re-ask per unparseable sample (up to 2× again), `generate` adds a schema-less retry on 4xx (`providers.rs:90-99`), and `with_retry` adds up to 3 attempts each (`retry.rs:35-47`) — a worst-case ceiling of ~12 provider calls *per game*. The candidate set is never sampled, capped, or truncated; `run_pairwise_matrix` only guards `targets.len() < 2` (line 80).
- **Impact**: At a haiku-class ~$0.01/judge call, the 8×100 sweep is ~$56 of judge spend on the happy path, and 4 targets → 8 targets is a **4.7× cost jump for a 2× target list** — the surprise is structural, not a mistake the user can see coming. The cost total is only printed *after* every call has been paid for (line 153-157). At `--jobs 8` and ~15s/call, that same sweep is also ~3 hours of wall clock.
- **Fix sketch**: Before step 1, compute `n_games = n_c * n_t*(n_t-1)/2` and `n_judge_calls = 2 * n_games`, price it against the `prices` book (`price_gen_cost` already takes provider/model/token estimates), and print an estimate. Add a `--max-games` / `--budget-usd` bound: when the enumerated set exceeds it, either abort with the estimate or subsample the game list deterministically (seeded shuffle over `games`, truncate) — round-robin win-rate stays unbiased under uniform pair sampling, it just widens the confidence interval. Cheapest complementary win: for `n_t > ~4`, offer a Swiss/ladder schedule (`O(n_t log n_t)` games) instead of full round-robin.
- **Trade-offs**: A cap means fewer games ⇒ noisier win-rates; that must be surfaced in the printed ranking and in the posted `report` JSON (e.g. `sampled: true`, `games_possible`). An estimate gate adds a prompt/flag to a currently one-shot command.

## 2. One failed game discards every dollar already spent on the whole matrix
- **Severity**: High
- **Category**: wasted-llm-call
- **File**: `crates/runner/src/pairwise.rs:139-148`
- **Scenario**: A 6-target × 50-case run (750 games, 1,500 judge calls) is 95% done. Game #712's judge hits a 429 that outlives the ~600ms of total backoff `with_retry` allows (`retry.rs:11-13`: 200ms + 400ms), or returns text that fails both the first parse and the repair re-ask in a way that surfaces as `EngineError::Parse` from `assemble` (`engine/src/pairwise.rs:82-87`).
- **Root cause**: The fold does `let o = outcome?;` inside the loop over `outcomes`. But `parallel_map` at line 128 is **eager** — it has already run and paid for *all 750 games* before the fold begins. The `?` therefore propagates out of `run_pairwise_matrix` after the money is spent, skipping `tally`, the printed matrix, and — critically — `post_run` (line 159), which is the only thing that records `gen_cost + judge_cost` into the run ledger. Nothing is retained: not the 749 good verdicts, not the generation cost from step 1, not even a printed total.
- **Impact**: A single transient failure at the tail of the run destroys ~$15 of judge spend plus the entire generation spend, and the cost never lands in the benchmark-run totals — so the tracker under-reports real money burned, on a tool whose purpose is cost tracking. The user's only recourse is to re-run the whole matrix (paying twice), and the same class of failure can hit again. Failure probability grows with `n_games`, so this bites hardest exactly where the run is most expensive.
- **Fix sketch**: Mirror the generation path's own posture — step 1 already tolerates per-cell failure (`Err(e) => { eprintln!(...); GenCell { output: None, .. } }`, lines 101-105). Do the same here: fold `Err` into a `judge_errors: u32` counter, `eprintln!` the game, and skip it from `winners`/`games` (drop the pair from `tally` rather than scoring it a Tie — a Tie is a fabricated verdict). Report `judge_errors` in the printed line and in the `report` JSON, and always reach `post_run` so the spend is recorded. If a hard error must abort (auth), post the partial run first, then return the error.
- **Trade-offs**: Standings computed over a partial game set are slightly less comparable across targets; the dropped-game count must be printed and persisted so a lopsided failure distribution is visible rather than silently skewing win-rates.

## 3. A game's two debias orders are judged serially, capping usable concurrency at `games.len()`
- **Severity**: Medium
- **Category**: serial-await
- **File**: `crates/engine/src/pairwise.rs:112-125` (`pairwise_via`), driven from `crates/runner/src/pairwise.rs:128-135`
- **Scenario**: The most common pairwise shape — 2 targets, 1–5 cases ("is my new prompt better than the old one?") — yields 1–5 games. With `--jobs 8`, `parallel_map` clamps to `jobs.clamp(1, n.max(1))` (`pool.rs:17`), so at most 1–5 threads ever run, and the remaining workers sit idle.
- **Root cause**: `sample_parsed(&gen, 0, &p1, …)?` then `sample_parsed(&gen, 1, &p2, …)?` are two fully independent provider calls — `p2` is built at line 121 before either runs, and nothing in order 2 consumes order 1's result; `assemble` only joins them afterwards. The `?` on line 122 also means an order-1 hard error skips order 2, which is correct, but the common path pays both latencies back-to-back. Concurrency exists only *across* games in the runner, so it cannot compensate when `games.len() < jobs`.
- **Impact**: Every game costs `latency(order1) + latency(order2)` instead of `max(...)`. At ~15s/judge call, the 2-target/1-case run takes ~30s instead of ~15s — a 2× wall-clock penalty on the flow users hit most often, on a box with 7 idle workers. For large matrices (`games ≫ jobs`) throughput is unaffected, which is why this is bounded rather than High.
- **Fix sketch**: `pool::parallel_map` is already the right tool and is already `pub(crate)` in the engine. In `pairwise_via`, replace the two sequential calls with `let mut rs = parallel_map(2, 2, |i| if i == 0 { sample_parsed(&gen, 0, &p1, parse_pairwise) } else { sample_parsed(&gen, 1, &p2, parse_pairwise) });` then `assemble(rs.remove(0)?, rs.remove(0)?)`. Order-indexed slots keep the result deterministic, and `gen` is already `Fn` — it needs a `Sync` bound added to `pairwise_via`'s signature (the real `generate` closure satisfies it; the test `fake` at `pairwise.rs:178-190` does too). The nested pool means peak in-flight calls become `jobs × 2` — so halve the `jobs` passed to the outer `parallel_map` at `runner/src/pairwise.rs:128`, or thread a shared semaphore through, to keep the provider-facing concurrency ceiling unchanged.
- **Trade-offs**: Nested pools make the effective concurrency limit less obvious and risk doubling provider load if the outer `jobs` isn't adjusted — which matters given retry has no shared circuit breaker. Two extra threads per game is negligible against network latency, but the `Sync` bound is a small API ripple through `pairwise_via`'s callers.
