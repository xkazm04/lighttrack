## 1. Simulator can only reprice revenue — cost and usage are frozen, so "what if the model price changes / usage doubles?" is unanswerable

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/core/src/margin_sim.rs:82-127` (esp. `let sim_margin = sim_rev - r.llm_cost_usd;` at :111), `crates/api/src/revenue.rs:157-168`
- **Scenario**: A founder hears Anthropic cut Haiku's price 40%, or sees signup projections doubling next quarter, and asks the CFO-grade question the product is positioned to answer: "what does that do to my margin?" `/v1/margin/simulate` cannot express either question. It only accepts `price_per_mtok` and `flat_monthly` — both **revenue** levers. The operator falls back to a spreadsheet, which is exactly the tool LightTrack exists to replace.
- **Root cause**: `compute_margin_simulation` takes `SimAssumptions { price_per_mtok, flat_monthly, window_days }` — there is no cost-side or volume-side input. Cost enters only as `r.llm_cost_usd`, lifted verbatim from `compute_margin`'s real windowed rollup, and `sim_margin = sim_rev - r.llm_cost_usd` holds it fixed by construction. The module header states this as intent ("real windowed cost stays fixed"). `tokens` is likewise consumed only as a revenue multiplier (`hypothetical_revenue`, :131-138) — the same token counts that would drive a re-costed scenario are already in the function and are never used on the cost side. Notably `TokensByDimension` carries **only** a summed `tokens: i64`, with no provider/model split and no prompt/completion split, so even the data plumbed in cannot support a model-level reprice today.
- **Impact**: Closes the gap between the two-thirds of the pitch that doesn't work and the third that does. Model-price shocks are the single most common margin event in an LLM business, and a usage multiplier turns the tool from a rear-view report into a planning instrument. This is the differentiator vs LangSmith/Langfuse/Helicone, who stop at cost reporting — a competitor can copy revenue repricing in a week; a re-costing engine wired to the price book is a moat.
- **Fix sketch**: Three independently shippable steps.
  1. **Usage multiplier** (cheapest, no new store work): add `usage_multiplier: Option<f64>` (default 1.0) to `SimAssumptions`. Scale both sides — `sim_tokens = tokens * m`, and `sim_cost = r.llm_cost_usd * m`. This is defensible for a pure per-token cost model and answers "usage doubles" immediately. Guard `m > 0.0` in `SimAssumptions::new` alongside the existing either-price check.
  2. **Cost-side reprice**, needs a richer token feed: extend `TokensByDimension` to `{ key, provider, model, prompt_tokens, completion_tokens, tokens }` (SQLite's `tokens_by_dimension` in `crates/store/src/sqlite/revenue.rs:119` already reads the same events and merely sums; the split is present upstream). Then accept `cost_overrides: Vec<{ provider, model, prompt_per_mtok, completion_per_mtok }>` and recompute `sim_cost` per key by re-pricing that key's per-model token split, defaulting to today's real cost for any model without an override.
  3. **Model-swap what-if** falls out of (2) for free: "reprice all `gpt-5.4` traffic at `claude-haiku-4-5`'s rates" is a `cost_overrides` entry sourced from `pricing.rs`'s existing book rather than typed by the operator. Surface `simulated_cost_usd` on `SimRow` alongside the existing `llm_cost_usd` so the response shows both sides moving; keep the sort on `simulated_margin_usd` ascending.
- **Trade-offs**: Step 1 is a real simplification — linear cost scaling ignores caching, batch discounts, and tiered provider pricing; label the field honestly in the response so the operator knows it is a straight-line estimate. Step 2 widens a store trait method and needs the two non-SQLite backends to follow (see finding 3) or their simulations degrade further. `SimRow` gains fields — additive only, no breaking change. Steps are strictly ordered by cost: ship 1 alone if step 2's store work can't be funded now.

## 2. FX conversion is lossy and one-shot — the original amount and rate are never persisted, so revenue converted at a wrong/missing rate can never be corrected

- **Severity**: High
- **Category**: capability-gap
- **File**: `crates/billing/src/fx.rs:115-139` (`to_usd`, `shared_fx`), `crates/core/src/revenue.rs:66-70`, `crates/api/src/revenue.rs:85-94`
- **Scenario**: An operator runs for two months with EUR and GBP customers, but `config/fx_rates.json` has no GBP rate. Every GBP invoice is stored at 1:1 — £480K booked as $480K — and `/v1/margin` correctly warns "unconverted currencies present: GBP". The operator does the right thing and adds a GBP rate. **The warning disappears and the historical numbers stay wrong.** The margin surface now reports fabricated revenue with full confidence and no caveat — strictly worse than before the fix, because the one signal that the figure was approximate is gone.
- **Root cause**: A chain of three linked gaps, all mechanical.
  1. **The `converted` flag is computed and universally discarded.** `to_usd` returns `UsdAmount { amount_usd, converted }`, but every call site takes `.amount_usd` and drops the flag: `stripe.rs:135`, `stripe.rs:159`, `polar.rs:152`, `polar.rs:217`. `UsdAmount::converted` has no consumer in the codebase outside its own tests — a structurally dead field.
  2. **The inputs to the conversion are not persisted.** `RevenueEvent` (`core/src/revenue.rs:66-70`) stores `amount_usd` and `currency` but neither the original `minor_units` nor the rate applied. The conversion is destructive: £480K and $480K are byte-identical once stored, so no backfill is *possible* — the information required to recompute is gone, not merely unindexed.
  3. **The read path infers convertibility from the *current* table, not from what happened at ingest.** `unconverted_currencies` (`api/src/revenue.rs:85`) calls `fx.is_convertible(&r.currency)` against today's book. Adding a rate silently reclassifies historical 1:1 records as trustworthy. This is what turns (1) and (2) from an accuracy limitation into a *false* clean bill of health.
  Separately, `shared_fx()`'s `OnceLock` (`fx.rs:136-139`) pins rates for process lifetime — a long-running server converts January and June revenue at the same rate, and a rate edit needs a restart. There is no rate-provider integration and no `as_of` history, so even a correct book is a single point-in-time snapshot.
- **Impact**: This is money-truth on the product's headline number. Margin is revenue minus cost; if revenue is silently wrong for any non-USD customer, every downstream surface — `/v1/margin`, the simulator, the trend, the below-breakeven roster — inherits the error, and finding 1's whole value proposition rests on it. Persisting the rate also unlocks a genuinely differentiating view: FX-neutral margin ("did my margin move, or did the euro?"), which no competitor offers because none of them model revenue at all.
- **Fix sketch**: Ordered so each step stands alone.
  1. **Stop the bleeding — persist the provenance.** Add to `RevenueEvent`: `amount_minor: Option<i64>`, `fx_rate: Option<f64>`, `fx_converted: bool` (serde-default so existing rows decode). Populate from the `UsdAmount` the adapters already receive — this finally consumes the dead `converted` flag. Cheap, and it makes every *future* record correctable.
  2. **Make the warning honest.** Switch `unconverted_currencies` to filter on the persisted `fx_converted == false` rather than re-deriving from the live table. The caveat then reflects what was actually done to the money and stops vanishing when a rate is added.
  3. **Backfill command.** With (1) in place, add a `lighttrack fx backfill` bin that re-reads `amount_minor` + `currency`, applies the current book, and rewrites `amount_usd`/`fx_rate`/`fx_converted` for records where `fx_converted == false`. Pre-(1) rows are unrecoverable — report their count so the operator knows the blast radius rather than discovering it later.
  4. **Rate freshness** (largest, optional): a dated rate book (`rates_by_date`) plus an ECB/openexchangerates fetcher on a schedule, converting each record at *its own* `ts`. `shared_fx`'s `OnceLock` becomes an `ArcSwap` so a refresh lands without a restart.
- **Trade-offs**: Steps 1-3 are additive and low-risk; step 3 mutates historical revenue figures, so it must be explicitly invoked, never automatic — a margin number changing under a CFO without a command is worse than one that is wrong-but-flagged. Step 4 introduces a network dependency into a currently hermetic, file-seeded design; keep the JSON file as the fallback so the offline story survives. Note that a dated book means a record's `amount_usd` depends on ingest-time rate lookup, so a late-arriving webhook must convert at its event `ts`, not `now`.

## 3. Firestore silently reports every customer as catastrophically unprofitable in the simulator — missing token/daily-cost methods fall through to no-op trait defaults

- **Severity**: Medium
- **Category**: half-implemented
- **File**: `crates/store-firestore/src/revenue.rs:1-6` (header claim), `:49-76` (`cost_by_dimension` — the pattern to copy); absence of `tokens_by_dimension` / `daily_cost_by_dimension` in `crates/store-firestore/src/lib.rs:220-239`
- **Scenario**: A Firestore-backed operator calls `/v1/margin/simulate?price_per_mtok=5`. Every row returns `tokens: 0` → `simulated_revenue_usd: 0` → `simulated_margin_usd` = negative full cost, and rows sort by that. The response is a confident, well-formed table stating that repricing would bankrupt them. There is no error, no empty result, and no caveat — the operator cannot tell this apart from a real answer. `/v1/margin/trend` degrades more visibly to a revenue-only series (cost flat zero), which reads as "margin is pure profit".
- **Root cause**: `LtStore::tokens_by_dimension` (`crates/store/src/lib.rs:603-611`) and `daily_cost_by_dimension` (`:394-402`) both default to `Ok(Vec::new())`. SQLite overrides both (`sqlite/mod.rs:188`, `:389`); Firestore overrides neither, so it inherits the no-ops. This file's own header asserts the opposite — *"This keeps the backend behaviorally identical to SQLite/Postgres, which the store conformance suite now asserts — without it, this backend would silently inherit the trait's no-op defaults."* For these two methods that claim is **false**, and the predicted silent inheritance is exactly what happens. `api/src/revenue.rs:190-194` documents the gap as "other backends return an empty simulation", which is also inaccurate: `compute_margin_simulation` builds its rows from the *cost* key set (`margin_sim.rs:106`), which Firestore **does** populate — so rows are returned, just with zeroed tokens. The wrong-looking-right outcome comes from that mismatch, and the code comment in `margin_sim.rs:92-93` ("token keys are a subset of cost keys ... so every token key already has a row") quietly assumes a populated token feed.
- **Impact**: Makes the two flagship margin surfaces trustworthy on the Firestore backend instead of confidently wrong, and removes a documentation claim that would mislead the next maintainer into assuming conformance coverage exists. Also a prerequisite for finding 1 step 2 — a richer token feed is worth little if only one of three backends serves it.
- **Fix sketch**:
  1. Implement `tokens_by_dimension` in this file by cloning `cost_by_dimension` (:49-76) exactly — same `rest.query("events", ...)`, same client-side `ts` window filter, same `metadata_key(m, field)` grouping — but summing `prompt_tokens + completion_tokens` via `ff64`/`fi64` instead of `cost_usd`. Wire it in `lib.rs` next to `cost_by_dimension` at :231.
  2. Implement `daily_cost_by_dimension` the same way, bucketing by `ts.format("%Y-%m-%d")` into a `BTreeMap<(String, Option<String>), f64>` and emitting `DailyDimCost`. Note `margin_trend.rs:90-96` already defends against boundary-day spill, so a slightly loose window filter is safe here.
  3. Add both to the store conformance suite (`crates/store/src/conformance.rs`) so the header's claim becomes true and the next backend cannot regress silently.
  4. Correct the stale comment at `api/src/revenue.rs:190-194` — "empty simulation" understates the failure mode.
  5. Consider a store capability flag so the API can caveat rather than fabricate when a backend genuinely lacks the feed — the general fix for a no-op default that is indistinguishable from real data. Postgres has the same two holes (`store-pg/src/lib.rs`) and would be covered by the same conformance test.
- **Trade-offs**: Both methods scan the project's whole `events` collection client-side, inheriting `cost_by_dimension`'s existing cost/latency profile — no new architectural debt, but no improvement either; Firestore's lack of `GROUP BY` makes this unavoidable without a rollup collection. Steps 1-2 are mechanical; step 5 is a design change worth separating.
