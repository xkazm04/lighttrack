> Total: 3

## 1. Span tree renders anonymous, undebuggable nodes — no step name, no error text, no drill-down id

- **Severity**: Critical
- **Category**: capability-gap
- **File**: `crates/render/src/traces.rs:115-152`
- **Scenario**: An operator gets an alert that trace `tr-9f2` failed and runs `lighttrack traces tr-9f2`. The agent made 11 calls — a planner, three tool-arg generators, a retrieval reranker, a summarizer, two retries. The markdown comes back as eleven bullets that all read `- ✅ `anthropic/claude-sonnet-4-5` · @1420ms +830ms · 4,102/311 tok · $0.0180`, one of them with a ❌ and *no reason why*. The operator cannot tell which step is the planner, cannot read the error, and has no id to feed into the event-detail view to see the payload. They fall back to grepping app logs — the exact thing the tracer was supposed to replace.
- **Root cause**: `render_node` reads only `status`, `usage.input/output`, `cost_usd`, `offset_ms`, `latency_ms`, `provider`, and `model` off the span's event. Three fields that already exist on the ingested event and are already serialized into the `/v1/traces/:id` payload are never touched:
  - `event.name` — `crates/core/src/event.rs:132-137` documents it as "Optional use-case / call-site name (e.g. \"summarize-email\")", the *unit the rollup groups by*. It is the only thing distinguishing one span from another in an agent run where every call hits the same model. The renderer ignores it entirely.
  - `event.error` — `event.rs:152-153`. The glyph flips to ❌ via `status_glyph(status)` but the message that says *why* is dropped. `crates/render/src/events.rs:96-98` proves the field renders fine one layer down (`- **Error:** {err}`); the trace tree just never asks for it.
  - `event.id` — the trace test asserts it round-trips (`crates/api/src/tests_traces.rs:102`, `spans[0]["event"]["id"] == root`), so it is present in the payload, but the tree prints no handle the operator can paste into the single-event detail view to get `input`/`output`.
  
  `event.operation` is likewise unused, though `events.rs:73-76` renders it. The tree is a cost/latency report wearing a debugger's shape.
- **Impact**: Turns the trace view from "a table of anonymized LLM calls" into an actual debugging surface. This is the single thing that makes the markdown trace competitive with a LangSmith/Langfuse waterfall: named steps, inline failure reasons, and a drill-down path (`tree → event id → event detail → payloads`). Without it the whole trace-tree feature answers "how much did this request cost" but not "what went wrong", which is the question that gets someone to open the tool at 2am.
- **Fix sketch**:
  1. In `render_node`, build the label as `name` when present, else `provider/model` (mirroring `event.rs:132-137`'s documented fallback contract). Keep the model as a trailing dim segment when a name exists, so both are visible: `` `plan-next-step` · anthropic/claude-sonnet-4-5 ``. Append `(operation)` when it isn't the `chat` default, matching `events.rs:73-76`.
  2. Append the short event id to each bullet (`· `e-8fa2…``) via the existing `trunc`; that is the operator's copy-paste handle into the event-detail view. Consider emitting it as the last cell so line width stays stable.
  3. When `status != success`, emit an indented sub-bullet with the error: `{indent}  - ⚠ {trunc(error, 160)}`. Import `opt_s` from `crate::md` (already exported — `events.rs:5-7` uses it) and guard on non-empty exactly as `events.rs:96` does. A sub-bullet rather than an inline suffix keeps the tree scannable when the message is long.
  4. Extend the `tree_renders_nested_spans_and_scores` fixture (`traces.rs:184-216`) with a named span and an errored child, asserting the name, the error text, and the id appear.
- **Trade-offs**: Line length grows; mitigate with `trunc` budgets (name ≤ 28, model ≤ 24, error ≤ 160). The error sub-bullet can leak provider-side detail into a rendered doc — but the redaction boundary is upstream at ingest (`Redactor`), and `events.rs` already renders `error` unguarded, so this adds no new exposure class.

## 2. "Whole-trace" scoring judges only the root span's payload — the agent's actual work is never seen

- **Severity**: High
- **Category**: half-implemented
- **File**: `crates/runner/src/score_traces.rs:136-154`
- **Scenario**: A team runs `score-traces --rubric "did the agent answer the user's question and use tools correctly?"` against a research agent. The root span is the planner call: its input is the user question, its output is a JSON tool plan. Every retrieval, every synthesis, and the final user-facing answer live in child spans. The judge dutifully scores the *plan*, reports `0.9 pass=true`, and the team ships believing their agent is graded. Nothing ever judged the answer. Worse, on agents whose root span is instrumented without payload capture, `let output = match root.and_then(|e| text_field(e, "output")) { Some(o) => o, None => continue }` silently skips the trace — the cycle prints `cycle: scored 0 trace(s)` forever with no diagnostic, and the operator has no signal distinguishing "nothing to score" from "your root spans have no output".
- **Root cause**: `run_cycle` reaches into `detail.pointer("/spans/0/event")` and judges that single event's `input`/`output`. The module doc (`score_traces.rs:1-13`) is candid — "judges each sampled trace's **root exchange** (the root span's input/output — the whole-request in/out)" — but the parenthetical equates the two, and `crates/api/src/traces.rs:117-119` compounds it by calling the root anchor "the whole-request judgment". They are only the same thing for single-span traces, i.e. exactly the traces that don't need a *trace* feature. The tree that would let the runner do better is already built and already fetched: `Trace::from_events` (`crates/core/src/trace.rs:99-138`) assembles the full chronologically-ordered forest, and `run_cycle` already holds the whole `detail` payload with every span's `input`/`output` in it — it just reads index 0 and throws the rest away. `/spans/0` is also fragile on multi-root traces, where `build_forest` (`trace.rs:179-216`) legitimately emits several roots for dangling-parent or cycle input (`trace.rs:319-340`).
- **Impact**: This is the feature's headline claim — "let the runner score whole traces rather than isolated generations". Today it scores an isolated generation that happens to be first. Assembling a real trace transcript makes the judge able to catch the things only visible across steps: a tool call that returned an error the agent then ignored, a retry loop, a planner/executor contradiction, an answer that drifted from the retrieved evidence. That is the differentiated capability versus per-call scoring, and it is one function away.
- **Fix sketch**:
  1. Add `fn trace_transcript(detail: &Value, budget: usize) -> Option<(String, String)>` in `score_traces.rs`. Walk `spans` depth-first (the payload's own nesting order is already chronological per `trace.rs:104`), emitting one block per span: `[{depth}] {name or provider/model} ({status})` then its `input`/`output` via the existing `text_field`, plus `error` when present.
  2. Judge `input` = the root span's input (the originating user request), `output` = the transcript. This keeps `build_judge_prompt`/`run_rubric_judge`'s two-slot contract (`score_traces.rs:224-252`) untouched — no engine change.
  3. Cap it: per-span `trunc` (~2k) and a whole-transcript budget (~24k chars) with a `…{n} spans elided…` marker, so a 200-span trace can't blow the judge's context or its bill.
  4. Change eligibility from "root has an output" to "*any* span has an output", which un-skips the agents currently invisible to the loop. Keep the `continue` for genuinely payload-free traces but count them and print `skipped {n} trace(s) with no captured payloads` at cycle end — that line alone closes the silent-zero mystery.
  5. Gate on a `--transcript` / `--scope=root|trace` flag defaulting to `root` for one release, then flip. The scored rubric label is unchanged, so the idempotency check (`trace_already_scored`, `score_traces.rs:256-267`) keeps working across the switch — note that means traces scored under root-scope won't be re-judged under trace-scope; document that, or namespace the label.
  6. Fix the anchor to `Trace::root_event_id()` semantics rather than raw `/spans/0` indexing, and fix the two doc comments that conflate root exchange with whole request.
- **Trade-offs**: Judge token cost rises roughly with span count — real money, which is why the char budget and the sampling knobs (`--sample-every`, already there) matter. Long transcripts can dilute a narrow rubric; structured rubrics (`Judge::Structured`) are the mitigation. The default-flip is a behavior change to a scoring policy, so scores across the boundary aren't strictly comparable — worth a note in the cycle output.

## 3. The waterfall data is computed, then rendered as bare numbers — no bars, no hot-span attribution

- **Severity**: Medium
- **Category**: power-feature
- **File**: `crates/render/src/traces.rs:115-146`, `crates/core/src/trace.rs:26-31`
- **Scenario**: A trace takes 14 seconds and costs $0.31. The operator wants the two facts that matter — *which span ate the time* and *which span ate the money* — and instead reads 11 bullets of `@1420ms +830ms · $0.0180`, doing arithmetic in their head against a `duration_ms` in the header. With overlapping spans (which `trace.rs:379-390` explicitly models) the mental math doesn't even work: summed latency ≠ wall time, and nothing on the page says which span is on the critical path.
- **Root cause**: The core did the hard part and the renderer didn't spend it. `TraceSpan` carries `offset_ms` and `latency_ms` precisely so "a consumer can place the bar `[offset_ms, offset_ms + latency_ms]` without digging into `event`" (`trace.rs:37-43`) — the comment names the bar; nothing draws one. `TraceTotals::total_latency_ms` (`trace.rs:26-31`) exists so compute and wall time can be contrasted, and `tree` does surface both (`traces.rs:61-65`, "1.50s wall · 200 ms compute") — but a bare pair of numbers doesn't tell you *where* the gap is. Every span's cost is printed absolutely, never as a share of `totals.cost_usd`, which the renderer has in hand two lines up (`traces.rs:74-76`).
- **Impact**: This is what "markdown-native waterfall" means. A monospace bar column turns eleven bullets into a shape you read in one glance — serial chains, parallel fan-outs, and idle gaps become visually obvious, which is the single thing the web competitors' waterfall views actually buy you. Cost share turns "$0.0180" into "58% of this trace", which is where an optimization actually starts.
- **Fix sketch**:
  1. Add `fn bar(offset_ms: u64, latency_ms: u64, total_ms: u64, width: usize) -> String` in `traces.rs`: scale both to `width` (~24 cols), emit `·` padding then `█` for the busy region, minimum one cell so a fast span never vanishes. Guard `total_ms == 0`.
  2. Render it in a fixed-width backticked column per node so the bars line up under proportional-font markdown: `` `····████········` ``. Keep the `@{offset}ms +{lat}ms` text — the bar is for scanning, the numbers for citing.
  3. Compute each span's cost share against `totals.cost_usd` and append `({pct}%)` when the share is material (say ≥10%); skip it when totals are zero or absent.
  4. In the header block, add a **Hot spans** line: the top 2–3 spans by `latency_ms` and by `cost_usd`, named via finding #1's label logic. That is a flat walk of the same tree the renderer already recurses.
  5. Optionally flag the wall-vs-compute gap explicitly — `1.50s wall · 200 ms compute (87% idle)` — which is the tell for a slow tool call or a queue between LLM steps, and it's a subtraction on two values already in the header.
- **Trade-offs**: Indentation plus a bar column can get cramped at depth; cap the drawn depth (bars flatten past ~4 levels) or drop the bar width to ~16 cols. Bars assume monospace — true in every terminal and in fenced/backticked markdown, which is why the column must be backticked rather than bare. Deep-nesting indentation slightly misaligns bar starts; wrapping the whole tree in a fenced block instead of a bullet list would fix alignment at the cost of losing markdown nesting — worth a look but a bigger change than the rest of this finding.
