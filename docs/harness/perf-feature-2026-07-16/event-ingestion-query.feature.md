## 1. Event query surface silently degrades to "unfiltered, unpaged" on the production backends

- **Severity**: High
- **Category**: half-implemented
- **File**: `crates/store/src/lib.rs:321-331`, `crates/store/src/lib.rs:352-358`, `crates/store-pg/src/events.rs:12-14`, `crates/store-pg/src/events.rs:66-86`, `crates/store-firestore/src/events.rs:24-28`
- **Scenario**: A team runs LightTrack on Postgres (the only backend that survives real traffic — SQLite serializes every call through one locked connection). They call `GET /events?project=p1&provider=openai&since=2026-07-01T00:00:00Z&limit=50`, get 50 rows back with a 200, and start reasoning about their OpenAI spend. The rows are *the 50 newest events of any provider, from all of history*. No error, no warning, no `X-Next-Cursor` — just a plausible-looking wrong answer. Same for `GET /usecases`, which returns `[]` and renders as an empty LLM-Overview table, indistinguishable from "you have no named calls".
- **Root cause**: `Store::list_events_filtered` has a default body that **drops the filter and the cursor on the floor**: `Ok(EventPage { events: self.list_events(project, limit)?, next_cursor: None })` (`lib.rs:321-331`). Neither `store-pg` nor `store-firestore` overrides it, so all seven `EventFilter` fields are inert there. `Store::usecase_costs` defaults to `Ok(Vec::new())` (`lib.rs:352-358`) — also un-overridden. Compounding it, the pg write path structurally cannot support the `name` dimension: `COLS` (`store-pg/src/events.rs:12-14`) has no `name` column, the `INSERT` never binds it, and `from_row` hardcodes `name: None` (`:180-181`) — so the use-case a customer set via `track(..., name=...)` is **destroyed at ingest** and unrecoverable later. `crates/api/src/events.rs:211-239` calls the trait method unconditionally and shapes the response as if the filter applied. The doc comments are honest ("backends that haven't ported… compile unchanged"), but that honesty lives in rustdoc, not in the API response.
- **Impact**: The two capabilities the context is named for — *query by project* and *roll into a cost summary* — are dev-backend-only in their filtered/windowed form. Silent wrong answers are the worst failure mode for an observability product: the whole value proposition is that the numbers are trustworthy. A prospect evaluating against Langfuse/Helicone on Postgres sees a filter UI that lies and an empty use-case table. This also silently defeats scoped limit rules (`LimitScope::Name` can never match a NULL `name` on pg) and the alert attribution path (`crates/api/src/alerts/attribution.rs:80` `unwrap_or_default()`s an always-empty rollup).
- **Fix sketch**:
  1. **Stop lying first (small, ships today).** Add `Store::capabilities() -> StoreCaps { filtered_events: bool, usecase_rollup: bool, windowed_costs: bool }`, defaulting to all-false, `true` on SQLite. In `get_events`, when the caller passed any constraining filter/cursor and the store lacks `filtered_events`, return `501 Not Implemented` with the backend name rather than a wrong 200. Same for `get_usecases`. A loud gap beats a quiet lie.
  2. **Add the `name` column to Postgres.** Migration `ALTER TABLE events ADD COLUMN name TEXT` + index `(project_id, name, ts)` mirroring SQLite; extend `COLS`, the `INSERT`, and `from_row`. Do this *before* step 3 — until `name` is stored, no pg rollup can be correct, and every event ingested in the meantime is permanently missing its use-case.
  3. **Port `list_filtered` to pg.** It's a mechanical translation of `crates/store/src/sqlite/events.rs:131-203` to sqlx bind syntax; `encode_cursor`/`decode_cursor` (`:207-225`) are backend-agnostic and should move to `crates/store/src/codec.rs` for reuse. Port `cost_summary_windowed` and `usecase_costs` the same way.
  4. **Make the gap un-reintroducible.** `crates/store/src/conformance.rs` already exists as a cross-backend suite — add filter/cursor/rollup cases to it so any backend claiming `filtered_events: true` must actually filter.
  5. Firestore: keep the `501` (composite-index cost is real), or implement the subset its single-field indexes serve for free (`provider`/`model`/`name` equality + `ts` range).
- **Trade-offs**: Step 1 turns today's silent-wrong into a visible 501 — that is a *behavior change* someone may be unknowingly depending on, and it will look like a regression in a demo. It's still right: an observability tool that returns confidently wrong numbers is worse than one that admits a gap. The pg migration is additive and backfill-free (`name` is nullable; historical rows stay NULL, which the SQLite rollup already folds under "unnamed"). Firestore composite indexes cost storage and write latency — hence the 501 escape hatch.

## 2. Redaction is unprovable and skips the field most likely to hold raw PII

- **Severity**: High
- **Category**: compliance
- **File**: `crates/api/src/redact.rs:71-83`, `crates/api/src/redact.rs:34-50`, `crates/api/src/events.rs:25-40`, `crates/core/src/event.rs:152-153`
- **Scenario**: LightTrack is in a security review for an enterprise deal. The buyer's DPO asks three questions: *(a)* "Prove this stored event was scrubbed." *(b)* "Show me redaction is enabled for our project, and show me the change log for when someone turned it off." *(c)* "What exactly do you redact?" Today the answers are: *(a)* you can't — a clean event and a redacted event are byte-identical in the DB; *(b)* SSH to the box and read `LIGHTTRACK_REDACT_INGEST`, and there is no log because it's an env var; *(c)* nine hardcoded regexes, and there's a leak (below). Deal stalls on the compliance questionnaire.
- **Root cause**: Three distinct gaps in what is otherwise a solid scrubber (`crates/anon/` is well-built — the property suite in `tests.rs:234-284` proves the no-leak and idempotence invariants at 3000 rounds; that quality is exactly why the *product* wrapper around it is worth finishing).
  - **No audit trail.** `redact_event` returns a count and `prepare_event` throws it at stderr: `eprintln!("[REDACT] project={pid} event={} redacted {redacted} PII span(s)", ev.id)` (`events.rs:30-32`). Nothing is stamped on the event. Contrast the neighbouring `mark_cost_source` (`events.rs:45-57`), which stamps `metadata.cost_source` precisely so downstream consumers "can trust or discount it" — redaction provenance is the same problem and got the weaker answer.
  - **`error` is never scrubbed.** `redact_event` (`:76-81`) walks `ev.input` and `ev.output` only. But `LlmEvent.error` (`core/src/event.rs:152-153`) is a free-text string populated from provider error responses — and providers routinely echo the offending request back in them (`400 invalid_request_error: … in messages[0].content: "…jane@example.com…"`). So the exact path where a call *failed* — the path a debugging engineer is most likely to open, and the one carrying an unvalidated provider blob — bypasses redaction entirely, while the module docstring says raw PII "never lands in the DB". `tags` and `source` are likewise unscrubbed strings. The docstring does exempt `metadata` explicitly; `error` is not mentioned at all, which reads as an oversight rather than a decision.
  - **Config is deploy-time-only.** `Redactor::from_env` (`:34-50`) reads the env once at boot into an immutable `Mode`. No API to read or change it, no per-project self-service, no record of who changed it or when — for a control that is itself the audit subject.
- **Impact**: PII handling is a genuine enterprise-sales gate, and gates are cleared with *evidence*, not with a feature that works. The regex pass is already the hard part and it's done and proven; what's missing is the paperwork layer around it — comparatively cheap, and it's what converts "we scrub PII" from a claim on a landing page into a line item a DPO can sign off. The `error` gap is worse than a missing feature: it's a leak underneath a stated guarantee, and a redaction promise with a hole in it is a liability the way a missing promise isn't.
- **Fix sketch**:
  1. **Close the `error` leak first** — it's a five-line change and the only item here that is actively harmful: in `redact_event`, scrub `ev.error` (and `ev.tags`) through the same `scrub_value` path. Add a test asserting a provider-echo error string is scrubbed. Then reconcile the module docstring with reality: state exactly which fields are in scope and which (`metadata`) are deliberately exempt and why.
  2. **Stamp provenance**, following `mark_cost_source`'s established pattern: `metadata.redaction = { "spans": 3, "rules": "v1", "at": "<ts>" }`, written whenever redaction *ran* — including `spans: 0`, since "we scrubbed it and it was clean" and "we never scrubbed it" are different compliance facts and only the stamp distinguishes them. Now `GET /events/{id}` answers question (a) by itself.
  3. **Version the rule set.** Add a `RULES_VERSION` const to `crates/anon`; include it in the stamp. When rules change, the stamp says which vintage scrubbed a given event — the difference between "our redaction improved" and "our historical corpus is of unknown provenance".
  4. **Promote config to data.** Move the mode to a per-project store row (default: inherit the env global, so no deployment breaks), expose `GET/PUT /projects/{id}/redaction`, and write each change to an append-only audit log (actor, before, after, ts). That answers (b) and turns the env var into a bootstrap default rather than the only control plane.
  5. Ship a `docs/PII.md` enumerating each rule, its placeholder, and what is explicitly *out of scope* (free-text names/places — already correctly acknowledged at `redact.rs:14-15`). That's the questionnaire answer to (c), pre-written.
- **Trade-offs**: Step 2 writes into `metadata`, which the code correctly treats as client-owned — reuse `mark_cost_source`'s exact guard (`events.rs:50-56`: never clobber non-object, non-null metadata) and accept the same "we don't stamp weird-metadata events" limitation, or the redaction stamp becomes a data-loss bug in the name of compliance. Step 4 puts a store read on the ingest hot path — cache it in `AppState` with the same refresh cadence as the price book, which already solves this shape. Steps 1–3 are self-contained and worth doing even if 4 is deferred.

## 3. You cannot ask the event log "what failed?" — but you can ask a trace

- **Severity**: Medium
- **Category**: capability-gap
- **File**: `crates/store/src/lib.rs:62-74`, `crates/api/src/events.rs:182-195`, `crates/store/src/sqlite/events.rs:294-317`
- **Scenario**: PagerDuty fires: error rate on `summarize-email` is up. The engineer opens LightTrack and wants the obvious thing — *the failed calls for this use-case in the last hour*. `EventFilter` has no `status`. So they either page through `?name=summarize-email&since=…` 1000 rows at a time filtering client-side (the render layer already computes a status glyph per row — `crates/render/src/events.rs:31-37` — proving the field is right there, just not queryable), or they pivot to the trace list, which *can* filter by status, and hope the failure was traced. The alerting side already knows this dimension matters: `st.alerts.record_error(ev)` (`events.rs:93-95`) feeds error-spike detection, so the product detects error spikes and then can't show you the errors.
- **Root cause**: `EventFilter` (`store/src/lib.rs:66-74`) carries `since/until/provider/model/trace_id/name/cursor` — no `status`, no `min_cost`, no `min_latency_ms`, no `tags`. `TraceFilter` (`:91-97`), built later, carries `status` *and* `min_cost`. The asymmetry is the tell: the trace surface got triage filters because the need was obvious at the aggregate level, and the event surface — where the same need is more acute, since an event is where you actually land — never got them backfilled. The SQLite `list_filtered` (`sqlite/events.rs:131-203`) is a clean `conds`/`args` builder where each filter is a 4-line block; `list_trace_summaries_filtered` (`:294-317`) already demonstrates exactly the status/min-cost predicates on the same table, in `HAVING`. At the event level they're plain `WHERE` clauses — strictly easier.
- **Impact**: Debugging and cost triage are the two jobs an LLM-observability tool exists for, and the two highest-value queries — "show me the failures" and "show me the expensive calls" — both require client-side filtering over a paged API. Against LangSmith/Helicone, whose filter bars lead with status, this reads as an unfinished product on the very first click of a demo. It's also a perf problem wearing a feature costume: filtering 10k rows client-side to find 12 failures means shipping 10k rows, each with full `input`/`output` payloads.
- **Fix sketch**:
  1. Add `status: Option<String>`, `min_cost: Option<f64>`, `min_latency_ms: Option<u64>` to `EventFilter`; thread through `EventsParams` (`api/src/events.rs:182-195`) — the struct is plain serde, so this is additive and backward-compatible.
  2. In `sqlite/events.rs:131-203`, three more `conds.push(...)` blocks — `status = ?`, `cost_usd >= ?`, `latency_ms >= ?`. Accept `status=error` as a convenience alias for "not success" (matching `TraceFilter`'s semantics at `:300-304`, so the two surfaces mean the same thing by the same word — a small consistency that pays off the first time someone builds a UI over both).
  3. Add a composite index `(project_id, status, ts)` so the failure query is index-served rather than a scan-and-filter; the existing `idx_events_project_ts` naming convention carries over.
  4. Tests mirror `sqlite/tests.rs:185-204` (the existing per-filter cases) — same shape, new columns.
  5. Follow-on, only if asked for: a `fields=` projection param so a triage listing can skip `input`/`output` payloads. Worth noting the option; not worth building on spec.
- **Trade-offs**: Each filter is another index decision, and `(project_id, status, ts)` is low-cardinality on its middle column — it earns its keep only because "not success" is the rare side, which is exactly the query. `min_cost`/`min_latency_ms` won't be index-served and will scan the project+window slice; that's acceptable at the window sizes this endpoint serves, and it's what the trace surface already does. Keep the JSON body a bare array (the render/CLI contract at `render/src/events.rs:9-52` depends on it) — new capability, unchanged shape.
