# Performance Optimizer — Trace Trees & Span Scoring

> Total: 3
> Critical: 0 | High: 2 | Medium: 1 | Low: 0

## 1. `score-traces` refetches every sampled trace's full span tree, one blocking round-trip at a time, to read one root span
- **Severity**: High
- **Category**: n-plus-one
- **File**: `crates/runner/src/score_traces.rs:136-154`
- **Scenario**: A `score-traces` daemon cycle over a busy project. `collect_sampled` already paged the trace list (1–2 requests) and returned up to `limit` `TraceSummary` rows. The loop then issues one *serial, blocking* `GET /v1/traces/{id}` per sampled trace. Each response is the entire trace detail — every span, each carrying its full `input`/`output` payload — plus every score in the trace. All of it is deserialized into `serde_json::Value`, and then only `/spans/0/event` and `scores` are actually read.
- **Root cause**: Two compounding issues in one loop. (a) N+1: the list endpoint gives no payloads, so the runner must go back per trace; there is no projection (`?spans=root`) or batch endpoint to ask for just the root exchange + scores. (b) The fetch is sequential on `reqwest::blocking` inside a plain `for`, so wall time is `N × RTT` even though the *judging* right below it is deliberately parallelized via `parallel_map(.., p.jobs, ..)`. Bytes moved per cycle are `O(sampled × spans_per_trace × payload_size)` while the loop's information need is `O(sampled × 1 span)`.
- **Impact**: At `limit=200`, `sample_every=4` (≈50 sampled), 40 spans/trace, 8 KB avg in+out per span → ≈16 MB of JSON fetched, parsed into `Value`, and dropped per cycle, to extract ≈400 KB of root exchanges — roughly a 40× over-fetch. At a 30 ms RTT the serial fetch phase alone costs ≈1.5 s of pure wall time before a single judge call starts, and on a deep trace (agent loop, 500+ spans) one response can be tens of MB. On the settle-window default (120 s) with a short `--interval`, this is a permanent load on the API and DB.
- **Fix sketch**: Two independent wins, either alone helps. (1) Parallelize the detail fetch with the `parallel_map(eligible.len(), p.jobs, ..)` helper already imported in this file — build the `Vec<Eligible>` from a parallel fetch+filter pass rather than a serial `for`, keeping the existing fetch-order posting. (2) Cut the payload: add a query param to `GET /v1/traces/:id` (e.g. `?spans=root`) that returns only the root span's event and the scores, and have `collect_sampled`'s consumer request it — the runner needs nothing else. (2) subsumes the deep-trace blowup; do it if only one is affordable.
- **Trade-offs**: (1) raises concurrent read pressure on the API — it is bounded by the existing `--jobs`, and these are indexed reads, so it should be acceptable. (2) adds a param to the endpoint contract and a projection branch in the store; the default shape stays unchanged so render/CLI are untouched.

## 2. `GET /v1/traces/:id` has no span cap — one deep trace materializes the whole tree with every payload
- **Severity**: High
- **Category**: hot-path
- **File**: `crates/api/src/traces.rs:102-115`, `crates/api/src/traces.rs:177-183`
- **Scenario**: A runaway agent loop (or a long-lived session reusing one `trace_id`) accumulates thousands of spans under a single trace. Someone opens that trace in the UI/CLI — or `score-traces` (finding #1) touches it. `load_trace` → `store.get_trace` → `list_trace_events` runs `SELECT … FROM events WHERE trace_id = ?1 ORDER BY ts ASC` with **no LIMIT**, and `Trace::from_events` builds the whole forest.
- **Root cause**: Every sibling read path caps its result set — `list_traces` does `q.limit.unwrap_or(50).min(1000)` (line 80), `/v1/events` pages — but the trace detail path takes no bound at any layer: not the handler, not `get_trace`, not `list_trace_events`. Cost is `O(spans)` in DB rows, but the constant is the killer: each row carries the event's `input`/`output` blobs, and the trace is held in memory in ~three simultaneous forms — the `Vec<LlmEvent>` from the store, the `Trace` tree that consumes it, and the `serde_json` serialization buffer for the response body. `Trace::from_events` is itself well-behaved (`build_forest` is a linear two-pass owner-map build, not the classic re-scan-per-node quadratic), so the tree assembly is not the problem — the unbounded input is.
- **Impact**: A 20 k-span trace at 8 KB of payload per span is ≈160 MB of row data → several hundred MB of peak RSS for one request, on a `spawn_db` blocking thread, plus a response body no client wants. A handful of concurrent requests for such a trace OOMs the process. The blast radius is a single unauthenticated-by-project id away: `authorize_trace` runs *after* `load_trace` (line 108-109), so the whole trace is loaded and paid for before a cross-tenant caller is rejected with 403.
- **Fix sketch**: Push a bound through the detail path: add a `max_spans` (config default, e.g. 5 000) to `list_trace_events` / `get_trace`, applied as `LIMIT` on the indexed `trace_id` query (`idx_events_trace` already serves it), and surface a `truncated: bool` on `Trace` so the render layer can say "showing first N of M spans". Independently, reorder `get_trace` to fetch the trace's `project_id` cheaply first (see #3's root lookup) and authorize *before* materializing the tree, so a forbidden request costs one row read.
- **Trade-offs**: A cap makes the detail view lossy for pathological traces — acceptable, since no UI renders 20 k spans usefully, and `truncated` keeps it honest rather than silently short. The authorize-first reorder costs one extra small query on the success path.

## 3. `POST /v1/traces/:id/score` builds the entire trace tree to read one event id
- **Severity**: Medium
- **Category**: hot-path
- **File**: `crates/api/src/traces.rs:143-154`
- **Scenario**: Every whole-trace score write — including each one `score-traces` posts at the end of its cycle (`score_traces.rs:168`). `score_trace` calls `load_trace(&st, &id)`, which loads *all* the trace's events, sorts them, computes `totals_of`, `distinct_models`, and the full span forest. It then uses exactly two fields: `trace.project_id` (for authorization + the score row) and `trace.root_event_id()`. When the body supplies `event_id`, `root_event_id()` is never even called (line 154) — the entire rollup is computed and dropped.
- **Root cause**: There is no cheap "identity of this trace" store call, so the write path reuses the read path's full materialization. `root_event_id()` is just `self.spans.first().map(|s| s.event.id)` — the id of the earliest parentless span — which a single indexed query can answer directly. Cost is `O(spans)` rows + tree build per write where `O(1)` suffices.
- **Impact**: Same over-fetch profile as #2 on the *write* path, so it inherits the unbounded-span exposure: scoring one deep trace pulls its whole payload set. On ordinary traces (20–50 spans) the waste is modest per call — a few hundred KB and a tree build on a `spawn_db` thread — but it is paid on every score, and `score-traces` makes this the steady-state write path. It also roughly doubles the trace's DB work per cycle, since #1 already fetched the same trace's detail moments earlier.
- **Fix sketch**: Add a narrow store method, e.g. `get_trace_root(trace_id) -> Option<(String /*project_id*/, String /*event_id*/)>`, implemented as `SELECT project_id, id FROM events WHERE trace_id = ?1 AND (parent_span_id IS NULL OR parent_span_id NOT IN (SELECT span_id FROM events WHERE trace_id = ?1)) ORDER BY ts ASC LIMIT 1` (or, simpler and matching the common shape, order by `ts ASC LIMIT 1` and accept the earliest span as the anchor — which is what `spans.first()` resolves to for any well-formed trace). Have `score_trace` use it for both `authorize_trace` and the anchor, keeping `load_trace` for `get_trace` only. Preserve the 404-on-unknown-trace behaviour by mapping `None` the same way.
- **Trade-offs**: The root-detection predicate must stay consistent with `build_forest`'s rule (dangling/self-parent → root) or a scored anchor could drift from the tree's displayed root and break `trace_already_scored`'s idempotency match in `score_traces.rs:256-267`. The `NOT IN` subquery variant is exact but costs a second index probe; the `ts ASC LIMIT 1` variant is one row read and agrees with `spans.first()` whenever the earliest span is a root — true for every trace the ingest path produces. Worth a test pinning the two definitions together.
