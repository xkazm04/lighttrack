# Performance Optimizer — Collective API & Rendering

> Total: 3
> Critical: 1 | High: 1 | Medium: 1 | Low: 0

## 1. `/v1/collective/leaderboard` loads the entire `collective_entries` table per request; filters never reach SQL

- **Severity**: Critical
- **Category**: no-pushdown / unbounded-result / table-scan
- **File**: `crates/api/src/collective.rs:269-316`, `crates/store/src/sqlite/collective.rs:57-67`
- **Scenario**: A hub with real adoption. `MAX_ENTRIES = 5000` entries are accepted *per contributor* (`collective.rs:319`), so 500 contributors ⇒ up to 2.5M rows. Every `GET /v1/collective/leaderboard` — a non-admin endpoint any authenticated caller (and the CLI/MCP render path) can hit at will — runs `store.list_collective_entries()`, which is a bare `SELECT … FROM collective_entries` with **no WHERE, no LIMIT, no aggregation**. `?provider=anthropic&task_type=qa` may narrow the answer to a handful of rows, but the full corpus is still read, decoded, and materialized first.
- **Root cause**: The filter parameters (`task_type`, `provider`, `judge`) are applied in Rust *after* `merge_leaderboard`, via `rows.retain(…)` at `collective.rs:279-287`. The store exposes only a nullary `list()`, so the handler has no way to express a predicate. Each row is fully decoded twice — `map_raw` builds a `Raw` with 5 owned `String`s, then `from_raw` moves it into a `CollectiveEntry` and `parse_ts` parses the `received_at` text timestamp — for every row in the table, including the ~99% about to be discarded. Worse, the schema already ships `idx_collective_model ON collective_entries(provider, model, task_type)` (`schema/sqlite/001_init.sql:247`), a perfect covering prefix for the `provider`/`task_type` filters — **no query in the codebase ever uses it**. The index is paid for on every ingest write and never read.
- **Impact**: Per request at 2.5M rows: a full table scan plus ~2.5M `CollectiveEntry` allocations (~150–200 B each ⇒ **~400–500 MB of transient heap**) and 2.5M RFC-3339 timestamp parses, all to serve a filtered response of maybe 20 rows. The store is a single `Mutex<Connection>` (`crates/store/src/sqlite/mod.rs:49`), so the scan **holds the process-wide DB lock**, stalling ingest and every unrelated endpoint. Two concurrent leaderboard requests double the peak RSS; a handful will OOM the process. This is the classic full-corpus-scan-for-a-small-response, and it is reachable by any reader.
- **Fix sketch**: Push the filter and the aggregation into SQL. Widen the store trait to `list_collective_entries(&self, f: &CollectiveFilter) -> Result<Vec<CollectiveEntry>>` with optional `provider` / `task_type` / `judge_provider`, and build the `WHERE` clause from the present fields (bound params, never interpolation) so the `idx_collective_model` prefix drives `provider [+ model + task_type]` lookups. Add `CREATE INDEX … ON collective_entries(task_type)` and `(judge_provider)` if task-only / judge-only filters prove hot. Note the `judge` filter is post-merge over `judge_providers` (a merged set), so it can be pushed only as a pre-filter on the per-entry `judge_provider` column — semantically equivalent here, since a merged row lists a judge iff some contributing entry carried it. Independently, cap the unfiltered path: `ORDER BY n_cases DESC LIMIT ?` plus a `COUNT(*)`-backed `has_more`, so an unfiltered call can never be unbounded. Metric to watch: rows-scanned vs rows-returned per request, and p95 handler latency vs `collective_entries` row count.
- **Trade-offs**: Widening the store trait touches the non-SQLite default impls (`crates/store/src/lib.rs:684-701`) and the `SqliteStore` call sites, plus the tests that call `list_collective_entries()` with no args. Pre-filtering before the merge changes nothing about the merged values for `provider`/`task_type` (both are part of the merge key), but the `contributors` and `n_models` header counts must then be derived from the same filtered set — which they already are. Bounding the unfiltered response is a wire-visible change and needs a documented default limit.

## 2. Ingest commits one transaction per entry, holding the global DB mutex for thousands of round-trips

- **Severity**: High
- **Category**: hot-path / write-amplification
- **File**: `crates/api/src/collective.rs:229-236`, `crates/store/src/sqlite/collective.rs:14-46`
- **Scenario**: A contributor POSTs a full digest to `/v1/collective/ingest` — up to `MAX_ENTRIES = 5000` buckets. The handler runs a delete followed by a bare `for e in &entries { store.upsert_collective_entry(e)?; }` loop.
- **Root cause**: Each `store.upsert_collective_entry(e)` goes through `SqliteStore::with(…)` (`sqlite/mod.rs:446-447`), which **re-acquires the `Mutex<Connection>` and executes in autocommit** — one implicit transaction, one WAL commit, and one durability barrier per row. It also re-prepares the (long, 15-parameter) upsert SQL on each call rather than reusing a cached statement. The codebase already has the right pattern next door: `revenue.rs:51` uses `conn.unchecked_transaction()`.
- **Impact**: At ~0.5–2 ms per autocommit on real disk, a 5000-entry digest costs **~2.5–10 s of wall time and 5000 fsyncs** — for what is fundamentally ~50 ms of work under one transaction. Because the mutex is taken and released 5000 times, the ingest interleaves with and starves every other request on the single connection for that whole window, and each iteration is a lock-convoy opportunity. Correctness rider: the delete and the upserts are in *separate* transactions, so a mid-loop failure leaves the contributor's set truncated and partially rewritten — a crash makes it permanent.
- **Fix sketch**: Add a store method that does the whole replace atomically — `replace_collective_entries(&self, contributor_id: &str, entries: &[CollectiveEntry]) -> Result<()>` — implemented in `sqlite/collective.rs` as: open `conn.unchecked_transaction()`, run the existing `delete`, `conn.prepare_cached(…)` the upsert once and `execute` it per entry, then `tx.commit()`. The handler's `spawn_db` closure collapses to a single call. One commit, one prepare, one mutex acquisition, and delete+upsert become atomic.
- **Trade-offs**: A new trait method plus a default `unsupported` impl for non-SQLite backends, mirroring the existing three. The single large transaction holds the write lock for its (now much shorter) duration rather than yielding between rows — a net win here, since the per-row version held it for far longer in aggregate. The existing `upsert`/`delete` fns stay for the tests that exercise them directly.

## 3. Header `contributors` count re-scans the whole entry corpus after the merge

- **Severity**: Medium
- **Category**: hot-path / no-pushdown
- **File**: `crates/api/src/collective.rs:293-302`
- **Scenario**: Every leaderboard request, on top of finding #1. After merging, the handler builds a `BTreeSet` of the surviving `(provider, model, task_type)` keys and then makes **another full linear pass over `entries`** — all 2.5M of them — testing each against the set to count distinct contributor ids.
- **Root cause**: The count is defined over stored entries rather than over the merged rows, so the raw `entries` Vec must be kept alive past the merge and walked again: O(E · log R) tuple comparisons with three string compares per probe. `merge_leaderboard` has already visited every one of those entries and each `LeaderboardRow` already carries `n_contributors` — the handler is recomputing, from the full corpus, information the merge pass held. This is also the only thing forcing the ~400 MB `entries` allocation to survive to the end of the handler instead of being dropped after the merge.
- **Impact**: Roughly doubles the post-fetch CPU of the handler and extends the peak-memory window across the whole response build. Standalone it is bounded and modest; it matters because it sits on the same unbounded corpus as #1 and it blocks the obvious `drop(entries)` that would halve peak RSS. Fixing #1 shrinks this proportionally — but the second pass and the retained Vec remain wrong in shape.
- **Fix sketch**: Cheapest correct option — have `merge_leaderboard` return the contributing ids per row (or expose a `contributor_ids: BTreeSet<String>` on `LeaderboardRow`) and union them over the *filtered* rows: no second corpus pass, no retained `entries`, and the semantics ("contributors backing the visible rows") are preserved exactly. If the core type must not change, do the pass **before** the merge in one sweep that also builds the merge input, then `drop(entries)`. Once the store gains a filter (#1), the count can alternatively be a `SELECT COUNT(DISTINCT contributor_id) … WHERE <same predicate>` — one extra cheap query instead of an in-memory scan of millions of rows.
- **Trade-offs**: The `LeaderboardRow` option touches a core type and the render/wire contract (add the field `#[serde(skip)]`, or keep it out of the response entirely). Note the field belongs to the merge crate, which a parallel audit owns — the pre-merge-sweep variant is fully API-side and needs no cross-crate coordination. Behavior is unchanged either way; `tests_collective.rs:302-345` already pins the exact semantics and should stay green.
