# Performance Optimizer — Ingest Hardening & Idempotency

> Total: 3
> Critical: 1 | High: 1 | Medium: 1 | Low: 0

## 1. A batch is 500 autocommit transactions and 500 limit-rule reloads, holding the one SQLite connection for the whole time
- **Severity**: Critical
- **Category**: per-row-transaction / n-plus-one
- **File**: `crates/api/src/events_batch.rs:101-108` (the store call; the loop it drives is `crates/store/src/sqlite/mod.rs:133-139` → `crates/store/src/sqlite/events.rs:92-106`)
- **Scenario**: An SDK flushes a buffered batch — the documented happy path, default cap 500 items (`events_validate::max_batch`). `post_batch` hands the whole slice to `insert_events_checked` in one `spawn_db`. The SQLite override takes `usage_cache.lock()` **and** the single connection lock, then loops `insert_checked` per item.
- **Root cause**: Two multiplications inside that loop, neither visible at the call site:
  1. **No enclosing transaction.** Each `events::insert` runs in rusqlite autocommit, so every item is its own `BEGIN`/`COMMIT` → one fsync per event. 500 events = 500 fsyncs, not 1.
  2. **`limits::list(conn, &ev.project_id, true)` re-runs per item** (`events.rs:97`), re-querying and re-deserializing the *same* project's rule set 500 times for a batch that is single-project by construction — `resolve_ingest_project` forces every item to one `pid`.
  Because `SqliteStore` serializes all calls through one locked connection, this is not merely slow for the batch — it is a **global stall**: every single-event `POST /v1/events` from every other tenant blocks behind it.
- **Impact**: On durable storage (~1-5 ms/fsync) a 500-item batch holds the global ingest lock for ~0.5-2.5 s, during which cluster-wide ingest throughput is zero. The batch endpoint — sold as the efficient path — is thus *worse* than 500 single posts, which at least interleave. The rule reloads add 500 redundant queries + row-mapping allocations per request. Wrapping the loop in one transaction is the standard ~10-100× win on SQLite bulk insert.
- **Fix sketch**: In `sqlite/mod.rs::insert_events_checked`, open one `conn.transaction()` around the loop and commit once (admission semantics are unchanged — the connection already sees uncommitted siblings' inserts within the txn, which is exactly what the cap-honesty test in `sqlite/tests.rs:89-104` asserts). Then hoist the rule fetch: read `limits::list` once per distinct `project_id` before the loop and pass `&rules` into a `insert_checked_with_rules(conn, cache, ev, &rules)`, keeping `insert_checked` as a thin wrapper for the single-event path. Metric to watch: p99 `POST /v1/events/batch` latency and lock-hold time, plus fsync count per request.
- **Trade-offs**: A transaction changes the per-item-error story: a `Conflict` on item _k_ must not poison the txn. rusqlite tolerates a failed statement inside a txn, so continuing the loop and committing the survivors preserves today's "siblings already committed stay committed" contract (`events_batch.rs:129-131`) — but the existing test at `sqlite/tests.rs:107-124` must stay green to prove it. Durability window widens slightly: a crash mid-batch now loses the whole batch rather than a prefix — acceptable, and arguably better (all-or-nothing beats a torn batch a client cannot detect).

## 2. Every valid event is deep-cloned twice per batch request
- **Severity**: High
- **Category**: allocation / hot-path
- **File**: `crates/api/src/events_batch.rs:94,105`
- **Scenario**: The same 500-item flush. `LlmEvent` is not cheap to clone — it carries `input: Option<Value>` and `output: Option<Value>`, the captured prompt and completion. Those are the whole point of an observability payload, and the batch body cap is 8 MiB (`DEFAULT_MAX_BATCH_BODY_BYTES`), so a full batch is megabytes of `serde_json::Value` trees: nested `Map`/`String` nodes, each its own heap allocation.
- **Root cause**: Line 94 `valid.push(ev.clone())` deep-copies each prepared event out of `evs`, then line 105 `let to_insert = valid.clone()` deep-copies the *entire vector again* purely to satisfy the `'static` bound on the `spawn_db` closure — while `valid` is kept alive only to read `ev.id` and `ev.cost_usd` back at lines 111-125.
- **Impact**: Peak resident bytes for an 8 MiB batch is ~3× the payload (original `evs` + `valid` + `to_insert`), plus two full traversals allocating one node per JSON value — the redraw happens on the product's hottest endpoint, so it is pure per-request garbage at exactly the moment memory pressure is highest. Concurrent batches multiply it. The second clone is removable outright; the first is avoidable too.
- **Fix sketch**: Kill the second clone by moving ownership into the closure and handing it back: `spawn_db(move || { let out = store.insert_events_checked(&to_insert); Ok((to_insert, out)) })`, then destructure and index `to_insert[k]` in the result loop instead of `valid[k]`. For the first clone, drain rather than copy — iterate `evs.into_iter().enumerate()` (taking `Json(evs)` by value and preparing into an owned local) so each valid event is *moved* into `valid`, never duplicated. Metric: allocation count / RSS delta per batch request.
- **Trade-offs**: None material — it is a borrow-plumbing change with identical semantics. `prepare_event` needs `&mut LlmEvent`, which an owned loop variable satisfies just as well as `iter_mut`.

## 3. The rejection ledger scans and rewrites its whole map, under a global mutex, on every rejected event
- **Severity**: Medium
- **Category**: hot-path / lock contention
- **File**: `crates/api/src/rejections.rs:83-84` (and the same pattern in `snapshot`, `:100-101`)
- **Scenario**: A limit breach storm — precisely the high-volume moment the ledger exists to observe. Every rejected event reaches `RejectionLedger::record` via `events::on_admission` → `record_rejection`, once per enforcing breach. A misbehaving client hammering a capped project generates a rejection per request; a 500-item batch over-cap generates up to 500 in one request.
- **Root cause**: `record` calls `Self::prune(&mut map, now, self.ttl)` on *every* insert, and prune is `map.retain(...)` — an O(total keys) walk over every bucket of every project in the process, evaluating a `chrono` duration per entry, all while holding the single `Mutex<HashMap>` that every ingest worker must pass through. The prune has no rate limit and no reason to run per-event: the TTL is 24 h (`TTL_HOURS`), so a second prune microseconds after the last one can never find anything new.
- **Impact**: Bounded but real, and it scales the wrong way — cost per rejection grows with `projects × metrics × windows × scopes`, so a busy multi-tenant instance pays a full-map walk per rejected call, serialized. Under a rejection storm the ledger becomes a contention point on the ingest path at the exact moment ingest is hottest. It cannot corrupt anything (the lock is correct, and `tests_ingest.rs:272-330` pins that the ledger stays out of usage math) — it just wastes CPU and serializes workers for no correctness gain.
- **Fix sketch**: Amortize the prune. Keep a `last_prune: DateTime<Utc>` on the ledger (inside `Inner`, or an `AtomicI64` timestamp) and skip the `retain` in `record` unless `now - last_prune > prune_interval` (a minute is far finer than a 24 h TTL). `snapshot` can keep pruning eagerly — it is a low-frequency operator read, not the hot path, and that preserves the "stale buckets never surface" guarantee the tests assert (`rejections.rs:166-177`). Metric: mutex wait time on `RejectionLedger::inner` under a synthetic breach storm.
- **Trade-offs**: A stale bucket can now linger up to the prune interval past its TTL before eviction — invisible, because `snapshot` still prunes before every read, so no caller can observe an expired entry. Memory holds slightly longer between prunes; the key space is bounded by `(project, metric, window, scope)` and the module already documents itself as best-effort process-local state, so this is well within its stated contract.
