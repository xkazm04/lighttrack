# Performance Optimizer — Relay Task Leasing (Cloud Side)

> Total: 3
> Critical: 0 | High: 2 | Medium: 1 | Low: 0

## 1. Long-poll lease is a 1-second busy-poll issuing write transactions, not an event wait
- **Severity**: High
- **Category**: polling / hot-path
- **File**: `crates/api/src/relay.rs:178-192`
- **Scenario**: The device agent is the only relay client and it polls forever. With long-poll enabled (`wait_secs`, shipped in Phase 3 and agent-configurable; server clamps to 25) a single idle lease request enters the `loop` and — because the only way it learns about a new task is to ask the database again — runs ~26 iterations of `sweep_relay_dead()` + `lease_relay_tasks()` separated by `tokio::time::sleep(1s)`. That is ~52 statements per request, every one of them an `UPDATE` (a write transaction), on a queue that is empty. Agent defaults are `poll_secs = 15`, so the steady-state cycle is 25s of held poll + 15s sleep ≈ 52 write transactions per 40s ≈ **~112k write transactions/day to deliver the handful of batch tasks the relay is designed for**. Plain polling (`wait_secs = 0`) costs 2 statements per 15s; the long-poll implementation therefore multiplies DB write load by ~26× per request to buy pickup latency.
- **Root cause**: The wait is implemented by re-querying rather than by blocking on a wakeup signal. Nothing tells a waiting poller that `enqueue_task` just inserted a due task, so the loop rediscovers state by polling. Both statements in the loop body are `UPDATE … RETURNING`, so each iteration takes a write transaction (and the SQLite write lock) even when it matches zero rows — an idle poll is never a read.
- **Impact**: Two compounding costs. (a) `SqliteStore` is a **single `Mutex<Connection>`** (`crates/store/src/sqlite/mod.rs:48-85`) — every iteration grabs the process-wide DB mutex twice a second, and it is the *same* mutex the event-ingest path needs, so an idle relay poller injects steady lock contention into the actual revenue hot path. (b) On the Neon-backed cloud, a write transaction every second means the compute **never reaches the idle threshold and never autosuspends** — the relay bills for 24/7 active compute to move a few tasks a day. Both are pure waste: the work discovers nothing.
- **Fix sketch**: Put a `tokio::sync::Notify` in `AppState`; `enqueue_task` calls `notify_waiters()` after a successful `create_relay_task`. The lease loop replaces `sleep(1s)` with `tokio::time::timeout(deadline - Instant::now(), notify.notified())` and only re-queries when woken or on a **coarse fallback tick (~5s)**. The fallback is required, not optional: a task becoming due via `next_attempt_at` backoff expiry is a timer event with no enqueue to signal it. This drops an idle 25s poll from ~26 queries to ~5, and to ~1 + immediate wake in the common case.
- **Trade-offs**: `Notify` is process-local — behind a load balancer, an enqueue on instance A will not wake a poller on instance B; those pollers fall back to the tick (correct, just at tick latency). If multi-instance pickup latency ever matters, the Postgres path can use `LISTEN`/`NOTIFY` instead. The fallback tick keeps worst-case latency bounded regardless, so correctness never depends on the signal arriving.

## 2. `sweep_relay_dead()` runs on every loop iteration, though it is not required for lease correctness
- **Severity**: High
- **Category**: hot-path / redundant-write
- **File**: `crates/api/src/relay.rs:179-184` (impls: `crates/store/src/sqlite/relay.rs:80-93`, `crates/store-pg/src/relay.rs:93-106`)
- **Scenario**: The sweep sits *inside* the long-poll loop, so it executes ~26 times per long-poll request and unconditionally on every plain lease request — accounting for half the write transactions in finding #1. What it can actually find changes only when a lease deadline passes, and leases are 60s–6h (`lease_secs` clamp; default 1800s). Running it at 1Hz to catch an event that occurs on a 30-minute-to-6-hour granularity is ~1800 wasted write transactions per useful one.
- **Root cause**: Dead-lettering was coupled to the lease request path for convenience, then inherited the long-poll loop's cadence when `wait_secs` was added. The coupling is unnecessary: `lease` already excludes expired-and-exhausted tasks itself via `attempts < max_attempts` in its `WHERE` (`sqlite/relay.rs:110-113`, `store-pg/relay.rs:120-123`), so **leasing is correct whether or not the sweep ran**. The sweep exists purely to flip those rows to `dead` and fire the alert — which the code comment concedes ("a task it misses stays `leased` and harmless until the next sweep").
- **Impact**: ~26 blind `UPDATE`s per long-poll request that almost always match zero rows, each taking the SQLite global connection mutex / a PG pool connection and a write transaction. Removing it from the loop halves the idle relay's DB write load on its own, and is shippable independently of #1.
- **Fix sketch**: Move it to a periodic janitor: `tokio::spawn` an interval task (~60s) at startup that calls `sweep_relay_dead()` and `alerts.notify_relay_dead(&dead)`, and delete the call from `lease_tasks`. If a background task is unwanted, the cheaper version is an `AtomicI64` `last_swept_at` in `AppState` gating the existing call to at most once per minute.
- **Trade-offs**: Dead-letter alerts are delayed by up to the janitor period (≤60s) — immaterial against a 5h `retry_interval_secs` and a ~20h dead-letter envelope. Multi-instance safety is preserved for free: the sweep is a single `UPDATE … RETURNING` filtered on `status='leased'`, so exactly one instance's statement transitions each row and observes it in `RETURNING` — alerts stay de-duplicated exactly as they are today.

## 3. `list_tasks` full-scans and sorts a never-pruned table, dragging the JSON blobs with it
- **Severity**: Medium
- **Category**: table-scan / missing-index
- **File**: `crates/api/src/relay.rs:125-139` (impls: `crates/store/src/sqlite/relay.rs:58-75`, `crates/store-pg/src/relay.rs:73-91`)
- **Scenario**: An admin (or an app inspecting its queue) calls `GET /v1/relay/tasks` with no `status`, `limit` up to 1000. `relay_tasks` has exactly two indexes — `idx_relay_due(status, next_attempt_at)` and the partial `idx_relay_idem(project_id, idempotency_key) WHERE idempotency_key IS NOT NULL` — so nothing serves this query. Terminal tasks are never pruned, so the table only grows.
- **Root cause**: Three compounding factors. (a) No index on `created_at`, so `ORDER BY created_at DESC LIMIT n` must materialise every matching row and sort. (b) `(?1 IS NULL OR project_id = ?1)` is non-sargable — the OR-with-parameter trick that makes one SQL string serve both filters also prevents the planner from using an index on `project_id` (and no such index exists anyway; the partial unique index only answers equality on both columns for rows that have an idempotency key). (c) `COLS` selects `payload` **and** `result`, so the scan hauls every task's full JSON blobs — and relay results are Claude Code run outputs, not small — only for the sort to discard all but `limit` of them.
- **Impact**: Cost grows linearly with lifetime task count rather than with the page returned, and there is no retention policy to bound it. On SQLite this is the sharp edge: the scan+sort holds the **single global connection mutex** for its whole duration, stalling event ingest and the device's lease behind an inspection query. Bounded today only because the table is young — this degrades silently as the relay accumulates history.
- **Fix sketch**: Add `idx_relay_project_created(project_id, created_at DESC)` and `idx_relay_created(created_at DESC)` to both `schema/sqlite/001_init.sql` and `schema/postgres/001_init.sql`. Build the `WHERE` clause dynamically — push only the filters actually supplied — instead of the `IS NULL OR` trick, so the planner can use them. Longer term, give `list` a summary projection that omits `payload`/`result` (the detail is one `GET /v1/relay/tasks/:id` away, which is a PK lookup and already fast), and add a retention job that prunes terminal `succeeded`/`dead` tasks past some age.
- **Trade-offs**: Two more indexes add write amplification on insert and on every lease `UPDATE` — negligible on a table this size and clearly worth it against a full scan. Dynamic SQL means more code paths to test than one static string. Dropping the blobs from the list projection is a response-shape change, so it needs an API-contract decision and SDK check — the indexes and the sargable `WHERE` do not, and are the bulk of the win.
